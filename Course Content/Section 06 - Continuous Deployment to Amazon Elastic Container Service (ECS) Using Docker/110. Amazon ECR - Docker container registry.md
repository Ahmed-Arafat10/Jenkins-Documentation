## Lecture 110 — Amazon ECR (Docker Container Registry)

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

## Why a Docker Registry Is Required

After building a Docker image in the Jenkins pipeline, the image exists **only temporarily** on the machine where the pipeline ran.

Once the job finishes:

* the image may be removed
* the next job might run on a different node
* the image will not be accessible to other services

In addition, a Docker image is **not a normal file** that can simply be uploaded to storage services such as:

* Amazon S3
* a build artifact repository
* a local filesystem

Docker images must be stored inside a **container registry**.

---

## What Is a Container Registry

A container registry is a service designed specifically to:

* store Docker images
* manage versions of container images
* allow images to be pulled by container platforms

A well-known public container registry is:

Docker Hub

Docker Hub hosts many publicly available container images such as:

* NGINX
* MySQL
* Node.js
* Redis

However, when working with **private applications**, publishing images publicly is not desirable.

For private applications, a **private container registry** is required.

---

## Amazon Elastic Container Registry (ECR)

AWS provides its own container registry service called:

Amazon Elastic Container Registry

Amazon ECR is a fully managed container registry designed to integrate directly with AWS services such as:

* ECS
* EKS
* AWS IAM authentication

ECR supports:

* private repositories
* image versioning
* secure authentication
* tight integration with AWS infrastructure

---

## Creating an ECR Repository

Inside the AWS Console:

1. Search for **Elastic Container Registry**
2. Open the ECR service dashboard
3. Click **Create Repository**

Several configuration options appear.

---

### Repository Visibility

Repositories can be:

* **Private** (default)
* **Public**

For internal applications, the repository should remain **private** so the images cannot be downloaded by external users.

---

### Repository Name

The repository name should match the application name used in the pipeline.

Example repository name:

```id="z0m8nf"
learn-jenkins-app
```

Using consistent naming between:

* the Jenkins pipeline
* the Docker image
* the ECR repository

makes the deployment pipeline easier to manage.

---

### Creating the Repository

After providing the repository name, the repository can be created.

ECR then generates a **repository URI**, for example:

```id="sn3d2s"
123456789012.dkr.ecr.us-east-1.amazonaws.com/learn-jenkins-app
```

This URI represents the full address of the container registry repository.

---

## Adding the Registry URL to the Jenkins Pipeline

The pipeline needs to know where to push the image.

Therefore an environment variable is added in the Jenkinsfile.

Example:

```groovy id="8rcdf8"
environment {
    AWS_DOCKER_REGISTRY = "123456789012.dkr.ecr.us-east-1.amazonaws.com"
}
```

Notice that only the **registry domain** is stored, not the repository name.

The repository name will be appended later when tagging the image.

---

## Properly Tagging the Docker Image for ECR

Docker images pushed to ECR must follow a specific format.

Structure:

```id="n2ngtq"
<registry>/<repository>:<tag>
```

Example:

```id="sz9a41"
123456789012.dkr.ecr.us-east-1.amazonaws.com/learn-jenkins-app:1.0.109
```

Components:

| Part       | Meaning                     |
| ---------- | --------------------------- |
| registry   | ECR registry domain         |
| repository | application name            |
| tag        | version or build identifier |

The Jenkins pipeline must tag the image using this full path.

---

## Updated Docker Build Command

Example:

```bash id="pyb9kc"
docker build -t $AWS_DOCKER_REGISTRY/$APP_NAME:$REACT_APP_VERSION.$BUILD_ID .
```

This ensures the built image is associated with the ECR repository.

---

## Pushing the Docker Image to ECR

After building the image, it must be uploaded to the registry.

This is done using the Docker push command:

```bash id="e2yk3g"
docker push $AWS_DOCKER_REGISTRY/$APP_NAME:$REACT_APP_VERSION.$BUILD_ID
```

This command uploads the image to the ECR repository.

---

## Authenticating to Amazon ECR

Before pushing an image, Docker must authenticate to the registry.

Unlike other registries, AWS does not provide a static username and password.

Instead, authentication is performed using an AWS CLI command.

The command used is:

```bash id="sm6u7x"
aws ecr get-login-password
```

This command generates a temporary authentication token.

The token is then passed to Docker using a pipe.

---

## Docker Login Command for ECR

The authentication command used in the pipeline:

```bash id="s0m0t0"
aws ecr get-login-password \
| docker login \
--username AWS \
--password-stdin $AWS_DOCKER_REGISTRY
```

Explanation:

| Component                    | Purpose                            |                                              |
| ---------------------------- | ---------------------------------- | -------------------------------------------- |
| `aws ecr get-login-password` | retrieves temporary login token    |                                              |
| `                            | `                                  | pipe operator sending output to next command |
| `docker login`               | authenticates Docker client        |                                              |
| `--username AWS`             | required username for ECR          |                                              |
| `--password-stdin`           | reads password from standard input |                                              |

The pipe transfers the token from the AWS CLI command directly into Docker login.

---

## First Error: Missing AWS Credentials

During the first pipeline run, the login command fails with:

```id="b9m8su"
Unable to locate credentials
```

This occurs because the AWS CLI requires credentials to access ECR.

The solution is to wrap the commands with Jenkins credentials.

Example:

```groovy id="jtdhzz"
withCredentials([[$class: 'AmazonWebServicesCredentialsBinding',
credentialsId: 'aws-credentials']]) {
    sh '''
    aws ecr get-login-password | docker login \
    --username AWS \
    --password-stdin $AWS_DOCKER_REGISTRY
    '''
}
```

---

## Second Error: Access Denied

After adding credentials, another error appears:

```id="v5k03h"
AccessDeniedException
```

This means the Jenkins IAM user **does not have permission to interact with ECR**.

---

## Fixing IAM Permissions

The Jenkins IAM user must be granted ECR permissions.

Inside AWS IAM:

1. Open **IAM Users**
2. Select the Jenkins user
3. Attach a new policy

Recommended policy:

```id="a4z6k5"
AmazonEC2ContainerRegistryFullAccess
```

This policy allows:

* authentication to ECR
* pushing images
* pulling images
* managing repositories

(Added explanation – not mentioned directly in the lecture)
In production environments, it is better to create a **custom IAM policy with minimal permissions** instead of using full access policies.

---

## Successful Docker Push

After fixing permissions, the pipeline succeeds.

Key log messages:

```
Login Succeeded
docker push <repository>
```

During the push operation, Docker uploads the image in **layers**.

Each layer corresponds to part of the container image.

---

## Verifying the Image in ECR

To confirm the push was successful:

1. Open the ECR service in AWS
2. Select the repository
3. View the **Images** tab

You will see the pushed image with its tag.

Example:

```id="3h9crk"
learn-jenkins-app:1.0.109
```

This confirms that the Jenkins pipeline successfully:

* built the Docker image
* authenticated with ECR
* pushed the image to the repository
