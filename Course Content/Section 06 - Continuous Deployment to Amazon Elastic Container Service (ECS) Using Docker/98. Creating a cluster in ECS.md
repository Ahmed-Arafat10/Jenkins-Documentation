## Lecture 98 — Creating a Cluster in ECS

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

## Goal of This Lecture

The purpose of this lecture is to **create the first ECS cluster** that will later be used to run containerized applications.

To keep things simple and focus on understanding ECS basics, the lecture uses an existing Docker container instead of building a custom one.

The container used is based on the **NGINX web server**.

---

## Using NGINX as the Example Application

The application used in this demonstration is **NGINX**, which is a widely used web server.

NGINX

NGINX is commonly used for:

* Serving static websites
* Acting as a reverse proxy
* Load balancing
* API gateway functionality

Although the default NGINX welcome page may look simple, it is one of the **most widely used web servers on the internet**.

(Added explanation – not mentioned directly in the lecture)
NGINX is especially popular in modern architectures because it is lightweight, high-performance, and capable of handling thousands of concurrent connections efficiently.

---

## Using a Pre-Built Docker Image

Instead of creating a custom Docker image, the lecture uses an existing container image available on Docker Hub.

Docker Hub

Docker Hub is a public container registry that stores Docker images created by developers and organizations.

The NGINX image is already available there, meaning:

* No Dockerfile needs to be created
* No custom build process is required
* The container can be deployed immediately

This simplifies the initial ECS setup so that the focus remains on learning ECS concepts.

(Added explanation – not mentioned directly in the lecture)
In later stages of CI/CD pipelines, teams typically build their **own Docker images** and store them in private registries such as **Amazon ECR** instead of using public images from Docker Hub.

---

## Creating an ECS Cluster

To run containers in ECS, the first step is creating a **cluster**.

Clusters provide the compute environment where containers will run.

Steps to create the cluster:

1. Open the ECS dashboard in the AWS console.
2. Click **Create Cluster**.

This opens the cluster configuration page.

---

## Defining the Cluster Name

The first required field when creating a cluster is the **cluster name**.

Example used in the lecture:

```
learn-jenkins-app-cluster-prod
```

This naming convention suggests that the cluster is intended for a **production environment**.

Using descriptive names helps distinguish clusters used for different environments such as:

* Development
* Testing
* Staging
* Production

---

## Selecting the Infrastructure Type

The next important configuration is the **infrastructure type**.

The selected option should be:

**AWS Fargate**

AWS Fargate

Fargate allows containers to run without managing servers.

Even though ECS also supports EC2-based clusters, Fargate is preferred in this course because it:

* Requires less configuration
* Eliminates server management
* Simplifies container deployment

The Fargate option is typically **selected by default**, but it is important to verify this before proceeding.

---

## Creating the Cluster

After setting the required configurations:

* Cluster name
* Infrastructure type (Fargate)

The next step is to click **Create**.

AWS will then begin provisioning the cluster.

The creation process usually completes quickly.

Once finished, the cluster appears in the **cluster list** in the ECS dashboard.

---

## Cluster State After Creation

Immediately after creation, the cluster exists but **nothing is running inside it yet**.

This means:

* No tasks are running
* No containers are deployed
* No compute resources are actively used

As a result, the cluster **does not generate costs at this stage**.

Costs are only incurred when tasks or services are running and consuming compute resources.

---

## Understanding the Empty Cluster

A newly created ECS cluster is essentially **an empty container environment**.

It has the capability to run containers, but it will remain idle until tasks are launched.

In upcoming steps, the cluster will be used to run a **task** that launches a container based on the NGINX image.

---

## Environment Separation (Production vs Testing)

The lecture also highlights an important best practice regarding environments.

In real-world systems, it is generally **not recommended** to use the same ECS cluster for both:

* Testing
* Production

Instead, organizations typically create separate clusters for each environment.

Example structure:

```
production-cluster
staging-cluster
testing-cluster
development-cluster
```

This separation helps with:

* Infrastructure isolation
* Security
* Testing reliability
* Deployment safety

In this course example, the created cluster is considered a **production cluster**.

(Added explanation – not mentioned directly in the lecture)
Separating environments also prevents accidental deployments or experiments from affecting live production applications.
