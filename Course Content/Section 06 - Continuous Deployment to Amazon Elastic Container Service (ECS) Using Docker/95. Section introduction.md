## Transition From Previous Section

Up to this point in the course, the application was deployed to **AWS Simple Storage Service (S3)**.

Amazon S3 was used to host a **static website**, and Jenkins handled the deployment pipeline.

This deployment approach is considered one of the **simplest integrations between Jenkins and AWS**, because:

* S3 can directly host static files such as:

    * HTML
    * CSS
    * JavaScript
* No backend runtime or server configuration is required.
* Deployment is simply uploading files to an S3 bucket.

However, most **modern applications are more complex** and cannot be deployed as static files only.

Typical modern applications include:

* Backend services
* APIs
* Databases
* Runtime environments
* Application servers

Because of this complexity, many applications today are deployed using **containers**, most commonly with Docker.

## Introduction to Containerized Deployment

In this section, the deployment process will shift from **static file deployment** to **container-based deployment**.

Instead of uploading files directly to S3, the application will be:

1. Packaged into a Docker container
2. Stored in a container registry
3. Deployed and run inside AWS container infrastructure

This approach reflects how **real-world production systems** are deployed today.

---

## What It Means to Dockerize an Application

To **dockerize an application** means:

> Packaging the application together with all of its dependencies inside a Docker container.

A **container** is a lightweight runtime environment that contains everything required for the application to run.

This typically includes:

* Application source code
* Runtime environment (for example Node.js, PHP, Java, etc.)
* System libraries
* OS-level dependencies
* Configuration files
* Web server if needed

Containers provide **environment consistency**, meaning the application behaves the same regardless of where it runs.

For example, the same container can run in:

* A developer's laptop
* A CI/CD pipeline
* A staging environment
* Production infrastructure

This eliminates the classic problem:

> "It works on my machine."

(Added explanation – not mentioned directly in the lecture)
Containers are isolated processes running on a shared host operating system kernel. Unlike virtual machines, they do not require a full operating system, making them **much lighter and faster to start**.

## Container Packaging Using Docker Images

When dockerizing an application, the first step is creating a **Docker image**.

A **Docker image** is a blueprint or template used to create containers.

The image contains everything required for the container to run.

In this section, the Docker image will include:

* A **web server**
* The **website files**

Once the Docker image is built, containers can be created from that image.

(Added explanation – not mentioned directly in the lecture)
Docker images are typically defined using a **Dockerfile**, which describes step-by-step how the container image should be built.

---

## Storing Docker Images in AWS

After the Docker image is created, it must be stored somewhere so it can be used for deployment.

For this purpose, AWS provides a container image registry called:

Amazon Elastic Container Registry

Amazon ECR allows developers to:

* Store Docker images
* Version images
* Secure images with AWS authentication
* Integrate directly with AWS deployment services

The workflow will be:

1. Build the Docker image
2. Push the image to **Amazon ECR**

---

## Running Containers Using Amazon ECS

Once the image is stored in ECR, the next step is running it.

This will be done using:

Amazon Elastic Container Service

Amazon ECS is a container orchestration service that allows you to:

* Run Docker containers
* Scale containers
* Manage container deployments
* Control container networking and load balancing

ECS retrieves container images directly from **Amazon ECR** and runs them inside a managed cluster.

(Added explanation – not mentioned directly in the lecture)
ECS can run containers using two compute options:

* **EC2 launch type** — containers run on EC2 instances you manage.
* **Fargate launch type** — serverless containers where AWS manages the infrastructure.

The course will later choose one of these approaches for deployment.

## High-Level Deployment Architecture in This Section

The deployment pipeline that will be built in this section follows this flow:

1. Jenkins builds the application
2. Jenkins builds a **Docker image**
3. Jenkins pushes the image to **Amazon ECR**
4. **Amazon ECS** pulls the image from ECR
5. ECS runs the container as a service

Simplified pipeline flow:

```
Source Code
     │
     ▼
Jenkins Pipeline
     │
     ▼
Build Docker Image
     │
     ▼
Push Image to Amazon ECR
     │
     ▼
Deploy Container via Amazon ECS
```
