## Lecture 96 — Introduction to Amazon Elastic Container Service (ECS)

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

## Cloud Infrastructure and Virtual Machines in AWS

When using a cloud provider such as AWS, developers can rent **virtual machines** that provide computing resources such as:

* CPU
* Memory (RAM)
* Disk storage
* Networking

These virtual machines behave like traditional physical servers but are hosted in the cloud.

In AWS, this capability is provided by:

Amazon Elastic Compute Cloud

EC2 allows users to:

* Launch virtual machines
* Choose the operating system
* Install any required software
* Configure networking and storage
* Run applications directly on the instance

This provides **full control over the infrastructure**, but it also introduces operational responsibilities.

---

## Operational Responsibilities When Using EC2

When managing an EC2 instance directly, developers or operations teams are responsible for maintaining the server.

Typical responsibilities include:

* Securing the operating system
* Installing and updating software packages
* Applying security patches
* Managing system libraries
* Monitoring system health
* Scaling infrastructure when demand increases

While this provides flexibility, it can also create **significant operational overhead**, especially when the goal is simply to deploy and run an application.

For many applications, developers prefer to focus only on the application itself rather than managing the infrastructure.

---

## Introduction to Amazon Elastic Container Service

To reduce infrastructure management complexity, AWS provides a container orchestration service called:

Amazon Elastic Container Service

Amazon ECS allows developers to **deploy and manage containerized applications** in the AWS cloud without directly managing the virtual servers running those containers.

ECS integrates with container technologies such as:

* Docker

Instead of managing operating systems and servers, ECS allows teams to focus primarily on:

* The application
* The container configuration
* The deployment process

---

## Fully Managed Container Deployment

ECS can operate as a **fully managed container service**.

This means AWS can handle much of the infrastructure management required to run containerized applications, including:

* Environment setup
* Container scheduling
* Resource allocation
* Scaling infrastructure
* Maintaining the underlying systems

This significantly reduces operational complexity compared to managing EC2 instances manually.

(Added explanation – not mentioned directly in the lecture)
When ECS is configured using the **Fargate launch type**, AWS manages the entire infrastructure layer, meaning developers do not need to provision or manage EC2 instances at all.

---

## Flexibility of Containerized Applications

Because ECS runs containerized workloads, it supports a wide variety of application technologies.

Examples include applications written in:

* Python
* Java
* Node.js
* PHP
* Go
* Ruby

The container abstraction ensures ECS does not need to know the internal technology stack of the application.

ECS simply runs the container image provided.

This provides **high flexibility**, allowing different types of applications to run within the same container infrastructure.

## Important Cost Warning

Although ECS is powerful and flexible, it is important to understand that **AWS services incur costs when resources are running**.

Even if the application is not actively used, running infrastructure still consumes cloud resources.

Examples include:

* Running containers
* Allocated compute resources
* Network usage
* Storage

Running services for a short time (hours or days) typically costs very little.

However, if services remain active for long periods (such as a full month), the accumulated cost may become noticeable.

Therefore it is recommended to:

* Stop or delete services when they are no longer needed
* Monitor resource usage
* Set reminders to avoid leaving resources running unintentionally

## Overview of ECS Core Concepts

Before deploying an application in ECS, it is necessary to understand several core concepts.

The three main components involved in deploying an application are:

1. **Cluster**
2. **Task Definition**
3. **Task**

These components work together to run containerized applications.

---

## ECS Cluster

An **ECS cluster** represents a group of compute resources that run containerized applications.

Clusters can be thought of as **a pool of servers working together** to execute containers.

Clusters provide:

* Compute resources (CPU and memory)
* Storage resources
* Networking capabilities
* A runtime environment for containers

Containers deployed in ECS always run **inside a cluster**.

(Added explanation – not mentioned directly in the lecture)
Clusters may consist of:

* EC2 instances (when using EC2 launch type)
* AWS-managed compute resources (when using Fargate)

---

## Why Use a Cluster Instead of a Single Server

Running an application on a single server creates several risks and limitations.

### Reliability Issues

If the single server fails due to:

* Hardware problems
* Software crashes
* Network failures
* Resource exhaustion

The application will immediately become unavailable.

This reduces system reliability.

### Scalability Limitations

A single server can handle only a limited amount of traffic.

If user traffic increases significantly, the server may:

* Become slow
* Run out of resources
* Crash under heavy load

---

## Advantages of Using a Cluster

Using a cluster improves both **reliability and scalability**.

### Improved Reliability

Clusters allow workloads to run across multiple instances.

If one instance fails:

* Other instances can continue running the application
* Service availability is maintained

### Improved Scalability

Clusters allow new compute resources to be added when traffic increases.

This makes it easier to scale applications dynamically.

---

## Task Definition

Once a cluster exists, ECS needs instructions describing **what application should run**.

This configuration is called a **Task Definition**.

A task definition acts as a **blueprint** for running containers in ECS.

It contains all the configuration needed to run the application.

Typical task definition configuration includes:

* The Docker image to run
* CPU allocation
* Memory allocation
* Environment variables
* Network configuration
* Port mappings
* Logging configuration

Essentially, the task definition tells ECS **how to run the containerized application**.

(Added explanation – not mentioned directly in the lecture)
Task definitions are versioned in ECS. Each time a configuration change is made, a new revision of the task definition is created.

---

## Running a Task

Once both of the following exist:

* An ECS cluster
* A task definition

The application can be launched by **running the task definition inside the cluster**.

Running a task means:

* ECS pulls the Docker image from the container registry
* ECS creates a container based on the task definition
* The container starts running inside the cluster

This process effectively **deploys the application**.

---

## ECS Deployment Workflow

The deployment process in ECS generally follows this sequence:

1. Create an ECS **cluster**
2. Define a **task definition** describing the container
3. Run the **task** inside the cluster

Conceptual workflow:

```
Docker Image
      │
      ▼
Task Definition
      │
      ▼
Run Task
      │
      ▼
ECS Cluster
      │
      ▼
Application Running
```

When the task runs successfully, the application becomes active and begins serving requests.

---

## Conceptual Summary of ECS Components

To deploy an application in ECS:

* The **cluster** provides the computing environment
* The **task definition** describes how the container should run
* The **task** executes the container inside the cluster

Together, these components enable containerized applications to run reliably and scale efficiently in AWS.
