## Lecture 108 — Building a Custom AWS CLI Docker Image

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

# Problem: Slow Pipeline Due to Installing Tools Every Run

During the previous setup, the Jenkins pipeline installed required tools **every time it executed**.

Example tools installed during the pipeline:

* Docker client
* `jq` JSON parser

This added extra time to every pipeline execution.

Observed execution time:

| Stage                           | Time        |
|---------------------------------|-------------|
| Docker image build              | ~2 seconds  |
| Installing dependencies + build | ~53 seconds |

Although 53 seconds is not extremely long, it is inefficient because the same dependencies are installed repeatedly.

---

# Strategy: Build a Custom Docker Image

Instead of installing tools during every pipeline execution, the better approach is to create a **custom Docker image
that already contains everything needed**.

This approach:

* avoids repeated installation
* speeds up pipeline execution
* makes pipelines more consistent

The custom image will include:

* AWS CLI
* Docker client
* `jq`

This image will replace the standard AWS CLI image used previously.


---

# AWS CLI Custom Dockerfile

Example configuration:

```dockerfile id="eqgpxr"
FROM amazon/aws-cli

RUN yum install docker \
    && yum install jq -y
```

### Explanation

| Instruction              | Purpose                            |
|--------------------------|------------------------------------|
| `FROM amazon/aws-cli`    | Use official AWS CLI image as base |
| `RUN yum install docker` | Install Docker client              |
| `RUN yum install jq -y`  | Install jq JSON parser             |

This image now contains **all tools needed for the pipeline**.

---

# Removing Tool Installation from Jenkins Pipeline

Since the custom image already includes:

* Docker
* `jq`

These installation commands can be removed from the Jenkinsfile.

This simplifies the pipeline significantly.

---

# Building Multiple Custom Images

A separate pipeline (`Jenkinsfile-nightly`) is used to build CI-related Docker images.

The stage responsible for this is renamed:

```id="o2fd8r"
Build Docker Images
```

This stage builds two images:

1. AWS CLI image
2. Playwright image

---

# Jenkinsfile-Nightly Example

Example pipeline commands:

```groovy id="vduqpn"
stage('Build Docker Images') {
    steps {
        sh '''
        docker build -t my-aws-cli -f ci/Dockerfile-aws-cli .
        docker build -t my-playwright -f ci/Dockerfile-playwright .
        '''
    }
}
```

### Explanation

| Part           | Meaning            |
|----------------|--------------------|
| `docker build` | Build Docker image |
| `-t`           | Tag image          |
| `-f`           | Specify Dockerfile |
| `.`            | Build context      |

# Testing Root Permissions

The pipeline previously ran containers with:

```id="i0r7k0"
-u root
```

This option runs the container as the **root user**.

To verify whether this is still necessary, the pipeline temporarily removes it.

---

# Permission Error Encountered

When removing root, the pipeline fails with:

```id="8x2p2k"
permission denied
```

The error is related to the Docker socket:

```id="6u7hvv"
/var/run/docker.sock
```

Accessing this file requires elevated permissions.

Therefore, the Docker build stage must run as **root**.

---

# Final Configuration

Root access is restored only for the stage that requires Docker.

Example:

```groovy id="9x0x3n"
args '-u root -v /var/run/docker.sock:/var/run/docker.sock'
```

However, the **Deploy to AWS stage** does not require root access.

So root is removed from that stage.

---

# Final Pipeline Behavior

Pipeline execution now works correctly with optimized stages.

Example execution flow:

```id="zjvklq"
Build Project
      ↓
Build Docker Image (root required)
      ↓
Deploy to AWS
      ↓
Wait for ECS Deployment
```

# Key Lessons from This Lecture

### Avoid Installing Dependencies During Pipelines

Instead of repeatedly installing tools:

```id="4ll9g3"
yum install jq
yum install docker
```

create a **custom Docker image** with those tools pre-installed.

---

### Use Dedicated CI Images

Different pipeline tasks may require different images.

Example CI images:

| Image           | Purpose              |
|-----------------|----------------------|
| `my-aws-cli`    | AWS deployment tasks |
| `my-playwright` | end-to-end testing   |

---

### Use Root Only When Necessary

Root privileges should only be used when required.

In this pipeline:

* **Docker build stage → requires root**
* **AWS deployment stage → does not require root**
