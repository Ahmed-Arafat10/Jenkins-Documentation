## Lecture 111 — Referencing an ECR Docker Image in the Task Definition

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

## Using the Docker Image Stored in ECR

In the previous step, the Jenkins pipeline successfully:

* built a Docker image
* tagged it with a version
* pushed it to a private container registry

The registry used is:

Amazon Elastic Container Registry

Inside the ECR repository, each image has a **full image URI**.

Example structure:

```id="1y2l4g"
123456789012.dkr.ecr.us-east-1.amazonaws.com/learn-jenkins-app:1.0.113
```

Components of this URI:

| Part                                           | Meaning         |
| ---------------------------------------------- | --------------- |
| `123456789012.dkr.ecr.us-east-1.amazonaws.com` | registry domain |
| `learn-jenkins-app`                            | repository name |
| `1.0.113`                                      | image tag       |

This URI is what ECS must use to pull the container image.

---

## Updating the ECS Task Definition

To deploy the new image, the **task definition must reference the ECR image instead of the NGINX image** used earlier.

Navigate to:

Amazon Elastic Container Service

Steps performed in the AWS console:

1. Open **Task Definitions**
2. Select the existing task definition
3. Click **Create New Revision**
4. Edit the container configuration

---

## Updating the Image URI

Inside the container definition configuration, locate the field:

```id="t7m6y0"
imageUri
```

Previously this contained the public NGINX image:

```id="v8wq0e"
nginx:1.26-alpine
```

This value must be replaced with the **full ECR image URI**.

Example:

```id="v8i9ru"
123456789012.dkr.ecr.us-east-1.amazonaws.com/learn-jenkins-app:1.0.113
```

This tells ECS exactly where to pull the container image from.

---

## Task Execution Role Requirement

When using images stored in ECR, ECS requires additional permissions so it can access the private registry.

When creating the revision, an error appears indicating a missing **Task Execution Role**.

To resolve this, AWS provides an option:

```id="ps4p0a"
Create New Role
```

This automatically creates an IAM role with the permissions required to pull images from ECR.

(Added explanation – not mentioned directly in the lecture)
The role typically attaches the policy **AmazonECSTaskExecutionRolePolicy**, which allows ECS tasks to pull images from ECR and write logs to CloudWatch.

---

## Result: New Task Definition Revision

After creating the role and saving the configuration, ECS creates a **new revision of the task definition**.

Example:

```id="hwhm30"
Revision 13
```

Each revision represents a new configuration version.

---

## Deploying the Updated Task Definition

Next, the ECS service must be updated.

Steps:

1. Open the **cluster**
2. Select the **service**
3. Click **Update Service**
4. Choose the **latest task definition revision**

Example selection:

```id="dd89di"
learn-jenkins-app-task-definition : 13
```

Once the update starts, ECS begins deploying the container using the new ECR image.

---

## Updating the Local Task Definition File

The pipeline also contains a local task definition JSON file used for automation.

This file must be updated to reflect the changes made manually.

Two important updates are required.

---

### 1. Task Execution Role ARN

Inside the AWS console, the JSON view of the task definition shows a property:

```id="j79gpn"
taskExecutionRoleArn
```

Example:

```id="ytg7d6"
"executionRoleArn": "arn:aws:iam::123456789012:role/ecsTaskExecutionRole"
```

This value must be copied into the local task definition JSON file.

Important note:

This ARN is **unique for each AWS account**, so it cannot be reused from other environments.

---

### 2. Updating the Image Property

Inside the local JSON file, locate the container definition:

```id="1zpcim"
"image": "nginx:1.26-alpine"
```

Replace it with the ECR image URI:

```id="7zv6gq"
"image": "123456789012.dkr.ecr.us-east-1.amazonaws.com/learn-jenkins-app:1.0.113"
```

Now the task definition references the correct container image.

---

## Verifying the Deployment

Once the service update completes, the cluster shows:

```id="juheqk"
1 / 1 tasks running
```

This confirms that the new container is running successfully.

To verify the deployed application:

1. Open the running **task**
2. Navigate to **Networking**
3. Copy the **Public IP address**
4. Open the IP address in the browser

The deployed web application loads successfully.

---

## Confirming the Correct Version

The application displays the version number embedded in the build.

Example:

```id="v9k8oq"
Version 1.0.113
```

This version should match:

* the Docker image tag
* the Jenkins build number
* the version stored in ECR

Example verification:

| Component             | Version |
| --------------------- | ------- |
| Jenkins Build         | 113     |
| Docker Image Tag      | 1.0.113 |
| ECS Running Container | 1.0.113 |

This confirms the CI/CD pipeline deployed the correct build.

---

## End-to-End Result

The full system now performs the following automated operations:

1. Jenkins builds the application
2. Jenkins builds the Docker image
3. Jenkins tags the image using the build version
4. Jenkins pushes the image to Amazon ECR
5. ECS pulls the image from ECR
6. ECS runs the container in the cluster

The application is now successfully deployed in the AWS cloud using a **containerized CI/CD pipeline**.
