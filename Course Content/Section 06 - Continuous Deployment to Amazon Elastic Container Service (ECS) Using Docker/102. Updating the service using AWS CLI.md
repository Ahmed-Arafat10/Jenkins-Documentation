## Lecture 102 — Updating the Service Using AWS CLI

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

## Recap: What Has Already Been Achieved

In the previous lecture:

* A **new revision of the ECS task definition** was created using the AWS CLI.
* The container image version was updated from **NGINX 1.26 → 1.27**.

However, updating the task definition alone does **not automatically update the running application**.

Why?

Because the ECS service is still configured to run the **previous task definition revision**.

Therefore, the next step is to **update the ECS service so it uses the new revision**.

---

## Updating the ECS Service

To update the running service through automation, another AWS CLI command is used.

The command comes from the ECS CLI operations of:

Amazon Elastic Container Service

The command used is:

```bash
aws ecs update-service
```

This command updates a service configuration, including which task definition revision it should run.

---

## AWS CLI Example from Documentation

AWS documentation provides an example similar to the following:

```bash
aws ecs update-service \
  --service my-http-service \
  --task-definition my-task-definition
```

This command instructs ECS to update a service and use a specified task definition.

However, additional parameters should be included for clarity and reliability.

---

## Why the Cluster Should Be Specified

Although the cluster parameter is optional, specifying it explicitly is recommended.

If the cluster is not provided, AWS assumes a **default cluster**, which may lead to unexpected behavior.

The parameter used is:

```bash
--cluster
```

Example format:

```bash
--cluster cluster-name
```

---

## Collecting the Required Values

To update the ECS service, three pieces of information are required:

1. **Cluster name**
2. **Service name**
3. **Task definition (including revision)**

These values can be retrieved from the AWS console.

---

### Cluster Name

Navigate to ECS → Clusters.

Example cluster name used in the lecture:

```bash
learn-jenkins-app-cluster-prod
```

---

### Service Name

Inside the cluster, open the **Services tab**.

Example service name:

```bash
learn-jenkins-app-service-prod
```

---

### Task Definition Name

Navigate to **Task Definitions**.

Example task definition family:

```bash
learn-jenkins-app-task-definition-prod
```

---

## Specifying the Task Definition Revision

One important detail is specifying **which revision of the task definition should be used**.

Since multiple revisions exist, the command should explicitly include the revision number.

Example:

```bash
learn-jenkins-app-task-definition-prod:2
```

The syntax:

```
task-definition-name:revision-number
```

This ensures ECS uses the exact version intended.

---

## Implementing the Command in Jenkins

The command is added to the Jenkins pipeline after the task definition registration step.

Example Jenkins pipeline snippet:

```groovy
stage('Deploy to AWS') {
    steps {
        sh '''
        aws ecs register-task-definition \
        --cli-input-json file://aws/task-definition-prod.json

        aws ecs update-service \
        --cluster learn-jenkins-app-cluster-prod \
        --service learn-jenkins-app-service-prod \
        --task-definition learn-jenkins-app-task-definition-prod:2
        '''
    }
}
```

This pipeline stage performs two operations:

1. Registers a new task definition revision.
2. Updates the service to use the new revision.

## Executing the Jenkins Pipeline

After committing the changes and triggering the pipeline:

* Jenkins executes the AWS CLI commands.
* The task definition revision is registered.
* The ECS service update is triggered.

The command returns immediately without errors, indicating the update process has started.

However, the deployment itself continues **asynchronously inside ECS**.

---

## Observing the Deployment in ECS

Inside the ECS cluster, the service now shows:

```
2 / 1 tasks running
```

This may appear confusing at first.

Why are **two tasks running** if the desired count is **1**?

---

## ECS Rolling Deployment Behavior

ECS performs **rolling updates** when updating services.

This means:

1. The new task starts first.
2. ECS verifies the container is healthy.
3. The old task is stopped afterward.

This prevents downtime.

Deployment sequence:

```
Old Container (nginx 1.26)
        ↓
Start New Container (nginx 1.27)
        ↓
Verify Health
        ↓
Stop Old Container
```

This ensures the application remains available during the update.

---

## Verifying the Updated Container

Inside the **Tasks tab**, both containers can be inspected.

Older task:

```
nginx:1.26
```

New task:

```
nginx:1.27
```

This confirms that the service update successfully deployed the new container version.

After the new container becomes stable, ECS automatically stops the old task.

---

## Public IP Address Change

When accessing the service again, the container may have a **different public IP address**.

This happens because:

* Each ECS task receives its own network interface
* When a new task starts, it may receive a different IP

This is normal behavior.

(Added explanation – not mentioned directly in the lecture)
In production environments, applications are usually placed behind an **Application Load Balancer (ALB)** so that users always access the same endpoint regardless of container changes.

---

## Verifying the Application

Opening the new task's public IP address shows the **NGINX welcome page**, confirming that:

* The service update succeeded
* The new container version is running
* Deployment automation worked correctly

Even though the page appears identical, the backend container version has changed.

---

## Final Automated Deployment Flow

The Jenkins pipeline now performs the full update process automatically:

```
Jenkins Pipeline
        │
        ▼
Update Task Definition (AWS CLI)
        │
        ▼
Create New Revision
        │
        ▼
Update ECS Service
        │
        ▼
Rolling Deployment
        │
        ▼
New Container Running
```

Only **two CLI commands** were required to automate the ECS deployment process.
