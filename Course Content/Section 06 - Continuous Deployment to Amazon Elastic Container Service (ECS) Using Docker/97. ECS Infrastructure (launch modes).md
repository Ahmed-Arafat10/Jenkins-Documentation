## Lecture 97 — ECS Infrastructure (Launch Modes)

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

## Accessing ECS from the AWS Console

To begin working with ECS, the service can be accessed through the AWS Management Console.

Steps:

1. Open the **AWS Management Console**.
2. Use the search bar at the top of the console.
3. Type **ECS**.
4. Select:

Amazon Elastic Container Service

This opens the ECS dashboard where container infrastructure can be created and managed.

When accessing ECS for the first time, the dashboard will show **zero clusters**, because no container infrastructure has been created yet.

The first step toward deploying containers is to **create an ECS cluster**.

---

## ECS Cluster Creation and Infrastructure Selection

When creating a cluster in ECS, one of the most important configuration sections is **Infrastructure**.

This configuration determines **how the containers will actually run in AWS**.

ECS supports two primary **launch modes** (ways to run containers):

1. **EC2 Launch Type**
2. **Fargate Launch Type**

Each launch mode defines **who manages the underlying compute infrastructure**.

---

## ECS Launch Mode: EC2

The first option for running containers in ECS is using **EC2 instances**.

This means that containers run on virtual machines provided by:

Amazon Elastic Compute Cloud

In this setup:

* ECS orchestrates the containers
* The containers run on EC2 instances
* The user manages the virtual machines

### Responsibilities When Using EC2 Launch Type

When using EC2 instances in ECS, the user is responsible for managing the underlying servers.

This includes tasks such as:

* Applying operating system updates
* Installing and updating software
* Managing security patches
* Monitoring system health
* Troubleshooting server issues
* Scaling EC2 instances when needed

While this approach provides **full control over the infrastructure**, it also increases operational complexity.

(Added explanation – not mentioned directly in the lecture)
With EC2 launch type, you must also manage:

* Auto Scaling Groups
* Instance capacity
* ECS agent running on the EC2 instances

This setup is often used in advanced production environments where organizations require **maximum control over infrastructure**.

---

## ECS Launch Mode: Fargate

The second launch mode is **AWS Fargate**, which provides a serverless way to run containers.

AWS Fargate

Fargate is a **serverless compute engine for containers**.

In this model:

* AWS manages the infrastructure
* Users do not provision or manage EC2 instances
* Containers run on AWS-managed compute resources

This means:

* No server management
* No OS patching
* No infrastructure configuration
* No capacity planning

Developers simply specify the **resources required by the container**, and AWS runs the container automatically.

### Key Idea of Serverless

In this context, **serverless does not mean there are no servers**.

Instead, it means:

* Servers exist
* AWS manages them
* Users do not interact with them directly

Developers focus entirely on **building and running applications**.

---

## Why Fargate Is Beginner-Friendly

Fargate is considered easier to use because it removes many infrastructure responsibilities.

Benefits include:

* Simplified setup
* No server management
* Automatic resource provisioning
* Faster deployment process

For beginners learning container deployment, this approach avoids spending time troubleshooting server configuration issues.

Instead, the focus remains on:

* Containers
* Deployment pipelines
* Application behavior

This is why the course chooses **Fargate for this section**.

---

## Important Cost Consideration

A critical point to understand is that **Fargate is not included in the AWS Free Tier**.

Because of this:

* Running containers on Fargate generates charges
* Costs depend on resource usage

Charges are calculated based on:

* CPU allocation
* Memory allocation
* Storage usage
* Runtime duration

AWS provides a pricing calculator to estimate these costs.

---

## Example Cost Estimation Using the AWS Calculator

When estimating Fargate costs, several configuration options are specified.

### Region

The AWS region where the infrastructure runs.

Example:

* **US East (N. Virginia)**

(Added explanation – not mentioned directly in the lecture)
Pricing may vary slightly depending on the AWS region selected.

---

### Compute Configuration

The container requires compute resources.

Example configuration from the lecture:

* **CPU**: 0.25 vCPU
* **Memory**: 0.5 GB (approximately 512 MB)

These values represent the **minimum resources needed for the simple web server application** used in this course.

---

### Storage

Containers may also require disk storage.

Example configuration:

* **Storage**: 20 GB

This storage is used for the container’s filesystem and application files.

---

### Usage Duration

Costs also depend on how long the container runs.

Example:

* **Duration**: 1 hour

When calculating costs for this configuration, the estimated cost is approximately:

**$0.38 USD per month**

If the service runs only for a few hours during the exercises, the total cost will remain very small.

However, if the service runs continuously for an entire month, the accumulated cost could become significantly higher.

---

## Why the Course Uses Fargate

Although both EC2 and Fargate are valid ECS launch modes, the course chooses **Fargate** for the following reasons:

* Simpler setup
* No infrastructure troubleshooting
* Faster learning process
* Focus on CI/CD rather than server management

Using EC2 would require additional configuration steps such as:

* Managing EC2 instances
* Installing container agents
* Configuring scaling policies
* Debugging infrastructure issues

This would distract from the main learning objective.
