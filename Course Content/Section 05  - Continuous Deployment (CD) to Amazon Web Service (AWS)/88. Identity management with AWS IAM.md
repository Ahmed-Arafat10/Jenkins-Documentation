# Lecture 88 — Identity Management with AWS IAM

## Overview

In this lecture, the focus is on solving the **authentication problem** encountered when using the AWS CLI in Jenkins:

```
Unable to locate credentials
```

The solution is to configure **secure, programmatic access** to AWS using **IAM (Identity and Access Management)**.

Instead of using the root account (email + password), we:

* Create a **dedicated IAM user**
* Assign **limited permissions**
* Generate **access keys**
* Use those credentials in Jenkins

This approach follows a critical security principle:

> **Principle of Least Privilege** — grant only the minimum permissions required.

---

# Key Concepts

## Why Root Credentials Cannot Be Used

The AWS Management Console login uses:

* Email
* Password

These credentials:

* Are meant for **human login via UI**
* Cannot be used with AWS CLI
* Have **full access to all AWS services**

Using root credentials in automation would be:

* Unsafe
* Over-permissioned
* A major security risk

---

## IAM (Identity and Access Management)

**AWS IAM** is a service used to manage:

* Users
* Permissions
* Access to AWS resources

IAM allows you to:

* Create **users for specific purposes**
* Assign **fine-grained permissions**
* Control access to AWS services securely

---

## Principle of Least Privilege

This principle states:

> A user should have **only the permissions necessary** to perform its task — nothing more.

Example:

* If Jenkins only needs to upload files to S3:

    * It should **only have S3 permissions**
    * Not access to EC2, databases, or IAM

---

## IAM Users

An **IAM user** represents a programmatic identity in AWS.

In this lecture:

* A new user is created specifically for Jenkins:

```
Jenkins
```

This user:

* Does NOT have console access
* Is used only for **CLI/API interactions**

---

## Policies

A **policy** defines what a user is allowed to do.

Policies are:

* Attached to users
* Written as permission rules (JSON behind the scenes)

In this lecture, the policy used is:

```
AmazonS3FullAccess
```

This allows the user to:

* Upload files
* Delete files
* List buckets
* Fully interact with S3

⚠️ Note:

This policy is **not strictly least privilege**, because:

* It allows access to all S3 resources
* Not limited to a single bucket

*(For production, permissions should be restricted further — e.g., specific bucket + specific actions.)*

---

## Access Keys (Programmatic Credentials)

To use AWS CLI, IAM users require **access keys**.

An access key consists of:

1. **Access Key ID** (like a username)
2. **Secret Access Key** (like a password)

Example:

```
Access Key ID:     AKIA...
Secret Access Key: xxxxxxxxxxxxxxxxx
```

These credentials:

* Authenticate requests made via AWS CLI
* Map directly to the IAM user
* Inherit that user's permissions

---

# Detailed Explanation

## Why Credentials Are Required

When running:

```bash
aws s3 ls
```

AWS needs to know:

* Who is making the request
* What permissions they have

Without credentials:

* AWS denies access
* Returns:

```
Unable to locate credentials
```

---

## Why IAM Is the Correct Solution

Instead of using root credentials, IAM allows:

* Creating **dedicated users for automation**
* Limiting access to **specific services**
* Managing credentials securely

This ensures:

* Security
* Auditability
* Controlled access

---

# Step-by-Step Process — Creating IAM User for Jenkins

## Step 1 — Open IAM Service

1. Go to AWS Management Console
2. Search for:

```
IAM
```

3. Open the **IAM service**

---

## Step 2 — Navigate to Users

From the left menu:

```
Users → Create User
```

---

## Step 3 — Configure User

* Username:

```
Jenkins
```

* Console access:

❌ Disabled (not needed)

Click:

```
Next
```

---

## Step 4 — Assign Permissions

* Select:

```
Attach policies directly
```

* Search for:

```
S3
```

* Choose:

```
AmazonS3FullAccess
```

This grants:

* Full access to S3
* No access to other AWS services

Click:

```
Next → Create User
```

---

## Step 5 — Open the Created User

After creation:

1. Click on the **Jenkins user**
2. Go to:

```
Security credentials
```

---

## Step 6 — Create Access Key

Scroll to:

```
Access keys → Create access key
```

Select use case:

```
Command Line Interface (CLI)
```

(Optional description):

```
Access key for Jenkins
```

Click:

```
Create
```

---

## Step 7 — Save Credentials

You will receive:

* Access Key ID
* Secret Access Key

⚠️ Important:

* The **secret key is shown only once**
* You must store it securely

# Practical Notes / Best Practices

## Never Store Credentials in Code

Do NOT do this:

```bash
# ❌ BAD PRACTICE
aws configure
# or hardcoding keys in scripts
```

Instead:

* Use **Jenkins credentials manager**
* Store secrets securely

---

## Rotate and Remove Credentials

* Delete access keys when no longer needed
* Disable unused credentials
* Regularly rotate keys

---

## Use Descriptions for Keys

Adding descriptions like:

```
Access key for Jenkins
```

helps:

* Track usage
* Audit credentials later

---

## Limit Permissions Further (Advanced)

Although `AmazonS3FullAccess` is used here:

Better approach in production:

* Restrict to:

    * Specific bucket
    * Specific actions (e.g., only `PutObject`, `GetObject`)

---

# Common Mistakes or Warnings

## Using Root Account for Automation

❌ Never use:

* Root email/password
* Root access keys

This is dangerous and unnecessary.

---

## Over-Permissioned Users

Using broad policies like:

```
AmazonS3FullAccess
```

can be excessive.

Better:

* Narrow permissions as much as possible

---

## Losing Secret Access Key

The secret key:

* Is shown only once
* Cannot be retrieved again

If lost:

* You must create a new access key

---

# Summary

This lecture introduces **AWS IAM** as the solution for authenticating AWS CLI in Jenkins.

Key takeaways:

* AWS CLI requires **credentials to access resources**
* Root credentials cannot and should not be used
* IAM allows creating **secure, limited-access users**
* Access is controlled via **policies**
* Programmatic access uses:

    * Access Key ID
    * Secret Access Key
* The **principle of least privilege** should always be applied

In the next lectures, these credentials will be integrated into **Jenkins pipelines securely**, enabling full automation of AWS operations like uploading files to S3.
