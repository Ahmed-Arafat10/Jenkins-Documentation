# Lecture 91 — Hosting a Website on S3

## Overview

In this lecture, the S3 bucket is transformed into a **static website host**, allowing files stored in the bucket to be accessed publicly over HTTP.

The goal is to:

* Serve the uploaded `index.html` file as a website
* Make the bucket publicly accessible
* Configure permissions so users can retrieve files via a browser

This process involves **three essential steps**:

1. Enable **Static Website Hosting**
2. Disable **Block Public Access**
3. Add a **Bucket Policy** to allow public access

---

# Key Concepts

## Static Website Hosting (S3)

Amazon S3 can act as a **static web server**, meaning:

* It serves files directly (HTML, CSS, JS)
* No backend processing
* No database
* No server-side logic

This is ideal for:

* Static websites
* Frontend apps
* Documentation sites

---

## Index Document

When hosting a website, S3 needs a default file:

```text
index.html
```

This file is served when users access the root URL.

---

## Public Access in S3

By default:

* All S3 buckets are **private**

To host a website:

* Files must be **publicly accessible**
* Permissions must explicitly allow access

---

## Bucket Policy

A **bucket policy** is a JSON configuration that defines:

* Who can access resources
* What actions are allowed
* Which resources are affected

---

# Detailed Explanation

# Step 1 — Enable Static Website Hosting

### Navigate:

```text
S3 → Bucket → Properties → Static Website Hosting
```

### Configuration:

* Enable:

```text
Static Website Hosting
```

* Hosting type:

```text
Static website hosting
```

* Index document:

```text
index.html
```

### Result:

* S3 generates a **website endpoint URL**
* Example:

```text
http://bucket-name.s3-website-region.amazonaws.com
```

---

## First Attempt Result

Opening the URL results in:

```text
403 Forbidden
```

✔️ This is expected (not fully configured yet)

---

# Step 2 — Disable Block Public Access

### Navigate:

```text
Permissions → Block Public Access → Edit
```

### Action:

* Disable:

```text
Block all public access
```

* Confirm changes

⚠️ AWS warns:

> This may expose your data publicly

✔️ This is intentional for website hosting

---

## After Step 2

Still:

```text
403 Forbidden
```

✔️ Expected — permissions are still incomplete

---

# Step 3 — Add Bucket Policy

### Navigate:

```text
Permissions → Bucket Policy → Edit
```

> hint: use `Add new statement` & `add a resource`

---

## Goal of the Policy

Allow:

* Anyone (`*`)
* To perform:

```text
GetObject
```

* On:

```text
All objects in the bucket
```

---

## Final Bucket Policy

```json id="nq4vhc"
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::learn-jenkins-20240518-1820/*"
    }
  ]
}
```

- Or (from course, both are correct)
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "Statement1",
      "Principal": "*",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject"
      ],
      "Resource": [
        "arn:aws:s3:::learn-jenkins-2026-arafat/*"
      ]
    }
  ]
}
```

---

## Explanation of the Policy

### Effect

```json id="3c30ki"
"Effect": "Allow"
```

* Grants permission

---

### Principal

```json id="0yk7hf"
"Principal": "*"
```

* Applies to **everyone (public access)**

---

### Action

```json id="5s81ne"
"Action": "s3:GetObject"
```

* Allows reading/downloading files

---

### Resource

```json id="o8xj33"
"arn:aws:s3:::bucket-name/*"
```

* Applies to:

    * All objects in the bucket

---

# Final Result

After applying the policy:

* Refresh the website URL
* Output:

```text id="mm57z4"
Hello S3
```

✔️ Website is now live
✔️ S3 bucket is serving content publicly

# Practical Notes / Best Practices

## Use `index.html` Convention

Always name the entry file:

```text
index.html
```

So S3 knows what to serve by default.

---

## Understand Public Access Risks

Making a bucket public:

* Exposes all files
* Must be done carefully

---

## Use Bucket Policies Carefully

Policies define:

* Security
* Access control

Incorrect policies can:

* Expose sensitive data
* Break access

---

## Use S3 for Static Sites Only

S3 is suitable when:

* No backend logic is needed
* No server-side rendering

---

# Common Mistakes or Warnings

## Missing `/*` in Resource

Incorrect:

```json
"arn:aws:s3:::bucket-name"
```

Correct:

```json
"arn:aws:s3:::bucket-name/*"
```

Without `/*`:

* Objects are NOT accessible

---

## Forgetting Public Access Settings

Even with policy:

* If public access is blocked → access fails

---

## Wrong Index File Name

If not set:

```text
index.html
```

Website may not load correctly.

---

## 403 Forbidden Error

Common causes:

* Public access blocked
* Missing bucket policy
* Incorrect resource ARN
