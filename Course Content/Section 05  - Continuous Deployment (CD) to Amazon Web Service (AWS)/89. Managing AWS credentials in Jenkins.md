# Lecture 89 — Managing AWS Credentials in Jenkins

## Overview

In this lecture, we connect everything together:

* We already created **IAM credentials (Access Key + Secret Key)**
* Now we need to **store and use them securely in Jenkins**

The goal is to:

* Store AWS credentials in **Jenkins Credentials Manager**
* Inject them into the pipeline as **environment variables**
* Allow AWS CLI to authenticate successfully
* Fix the previous error:

```bash
Unable to locate credentials
```

By the end of this lecture, Jenkins will be able to:

* Authenticate with AWS
* Execute commands like:

```bash
aws s3 ls
```

---

# Key Concepts

## Jenkins Credentials Manager

Jenkins provides a **secure storage system** for sensitive data such as:

* API keys
* Tokens
* Passwords
* AWS credentials

Instead of hardcoding secrets in code, Jenkins:

* Stores them securely
* Injects them only during pipeline execution
* Masks them in logs

---

## AWS CLI Authentication via Environment Variables

AWS CLI supports authentication using environment variables:

```bash
AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
```

These must be named **exactly like this**.

AWS CLI automatically reads them and uses them for authentication.

---

## Mapping Jenkins Credentials to Environment Variables

Jenkins allows binding credentials to environment variables using:

```groovy
withCredentials
```

This step:

* Injects credentials into the pipeline
* Limits their scope to a specific block
* Masks sensitive values in logs

---

# Detailed Explanation

## Step 1 — Store AWS Credentials in Jenkins

Navigate in Jenkins:

```text
Dashboard → Manage Jenkins → Credentials → System → Global Credentials
```

Click:

```text
Add Credentials
```

### Credential Configuration

* **Kind**: Username with password
* **Username**: Access Key ID
* **Password**: Secret Access Key
* **ID**:

```text
my-aws
```

Important notes:

* Even though AWS doesn’t call them "username/password":

    * Access Key ID → acts like username
    * Secret Access Key → acts like password
* Jenkins allows treating both as secrets (secure handling)

---

## Step 2 — Required AWS Environment Variables

AWS CLI expects these exact variables:

```bash
AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
```

These must match **exactly** (case-sensitive).

# Jenkinsfile Snapshot (Final Working Version)

```groovy
stage('AWS') {
    agent {
        docker {
            image 'amazon/aws-cli'
            args "--entrypoint=''"
        }
    }

    steps {
        withCredentials([
            usernamePassword(
                credentialsId: 'my-aws',
                usernameVariable: 'AWS_ACCESS_KEY_ID',
                passwordVariable: 'AWS_SECRET_ACCESS_KEY'
            )
        ]) {
            sh '''
            aws --version
            aws s3 ls
            '''
        }
    }
}
```

---

# How This Works

### Inside `withCredentials` block:

* Jenkins injects:

```bash
AWS_ACCESS_KEY_ID=xxxx
AWS_SECRET_ACCESS_KEY=xxxx
```

* AWS CLI automatically uses them
* Authentication succeeds

---

### Outside the block:

* Credentials are **NOT available**
* Commands will fail again

# Logs Behavior (Security)

Inside Jenkins logs:

* Credentials are **masked automatically**

Example:

```text
**** (instead of actual secret)
```

Even if you try:

```bash
echo $AWS_SECRET_ACCESS_KEY
```

You will see:

```text
****
```

This ensures:

* Secrets are never exposed
* Logs remain safe

# Practical Notes / Best Practices

## Always Use Jenkins Credentials

Never hardcode credentials like:

```bash
# ❌ BAD PRACTICE
export AWS_SECRET_ACCESS_KEY=xyz
```

Use:

```groovy
withCredentials
```

---

## Limit Credential Scope

Keep credentials only where needed:

```groovy
withCredentials {
    // only here
}
```

This reduces security risks.

---

## Use Clear Credential IDs

Example:

```text
my-aws
```

Better naming helps:

* Maintainability
* Debugging
* Team collaboration

---

## Understand Environment Variable Mapping

AWS CLI automatically reads:

```bash
AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
```

No extra configuration required.

---

# Common Mistakes or Warnings

## Running Commands Outside `withCredentials`

❌ This will fail:

```groovy
sh 'aws s3 ls'
```

✔️ Must be inside:

```groovy
withCredentials { ... }
```

---

## Wrong Variable Names

Incorrect:

```bash
AWS_KEY
AWS_SECRET
```

Correct:

```bash
AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
```

---

## Storing Secrets in Code

Never store credentials:

* In Jenkinsfile
* In Git repository
* In scripts

Always use Jenkins credentials manager.

---

## Forgetting Entry Point Override

Still required:

```groovy
args "--entrypoint=''"
```

Otherwise:

* Container may hang or fail
