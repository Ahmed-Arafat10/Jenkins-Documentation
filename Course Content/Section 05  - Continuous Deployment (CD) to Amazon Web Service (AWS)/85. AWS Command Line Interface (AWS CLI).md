# Lecture 85 — AWS Command Line Interface (AWS CLI)

## Overview

In order to interact with AWS services programmatically, such as uploading files to an **Amazon S3 bucket**, we cannot rely on the **AWS Management Console UI**. Instead, we use a **Command Line Interface (CLI)** tool.

This lecture introduces the **AWS CLI**, a tool that allows developers and automation systems (like Jenkins pipelines) to communicate with AWS services using commands.

Key goals of this lecture:

* Understand why **CLI tools are necessary for automation**
* Discover the **AWS CLI**
* Run the AWS CLI inside a **Docker container**
* Integrate the AWS CLI into a **Jenkins pipeline**
* Fix a common **Docker entrypoint issue**
* Learn best practices for **Docker image versioning in pipelines**

---

# Key Concepts

## AWS Command Line Interface (AWS CLI)

The **AWS CLI** is a tool that allows developers to interact with AWS services directly from the command line.

Using the AWS CLI, you can:

* Upload files to **S3**
* Create infrastructure
* Manage databases
* Configure cloud resources
* Deploy applications

Example command:

```bash
aws s3 ls
```

This command lists all available S3 buckets.

The CLI exposes a primary command:

```
aws
```

The general command structure is:

```
aws [options] <service> <command> [parameters]
```

Example:

```
aws s3 sync build/ s3://my-bucket
```

*(Uploads files from a local directory to an S3 bucket.)*

---

## Why Jenkins Needs a CLI

Jenkins pipelines **cannot interact with graphical interfaces**.

Jenkins cannot:

* Open a browser
* Log into AWS console
* Click buttons
* Upload files manually

Automation systems interact with external systems **only through commands or APIs**.

Therefore, Jenkins requires tools like:

* **AWS CLI**
* **Docker CLI**
* **Git CLI**

These tools allow Jenkins to control external systems programmatically.

# Detailed Explanation

## Installing AWS CLI

If you search online for:

```
AWS CLI
```

You will find official installers for:

* Windows
* macOS
* Linux

However, installing AWS CLI directly on the Jenkins host is **not the preferred approach in containerized pipelines**.

Instead, the course uses a **Docker image containing AWS CLI**.

Benefits:

* No manual installation
* Reproducible environment
* Consistent tool versions
* Easy integration with Jenkins

---

## AWS CLI Docker Image

The official AWS CLI Docker image is available on **Docker Hub**.

Repository:

```
amazon/aws-cli
```

Important characteristics:

* Published by **Amazon Web Services**
* Verified publisher
* Includes AWS CLI preinstalled

Approximate image size:

```
~130 MB
```

When pulling the image, Docker automatically uses the **latest tag** if no tag is specified.

Example:

```
amazon/aws-cli
```

is equivalent to:

```
amazon/aws-cli:latest
```

# Step-by-Step Process

## Step 1 — Create a New Jenkins Pipeline Stage

A new Jenkins pipeline stage is created for experimenting with AWS CLI.

Stage name:

```
AWS
```

Pipeline structure:

```groovy
stage('AWS') {
    agent {
        docker {
            image 'amazon/aws-cli'
        }
    }

    steps {
        sh 'aws --version'
    }
}
```

This stage:

* Starts a Docker container
* Uses the **amazon/aws-cli image**
* Executes the command:

```
aws --version
```

The purpose of this command is to **verify that AWS CLI is installed and working**.

---

# Common Mistake — Docker Entrypoint Conflict

When running the pipeline, the stage **hangs indefinitely** and does not complete.

The logs show an error similar to:

```
The container started but didn't run the expected command.
Please double check your entrypoint.
```

This happens because the **AWS CLI Docker image defines its own entrypoint**.

An **entrypoint** is the default command executed when a container starts.

In the `amazon/aws-cli` image:

* The container automatically launches the AWS CLI
* Jenkins expects to control the command execution

This creates a **conflict between Jenkins and the container entrypoint**.

---

# Fix — Override the Docker Entrypoint

To solve this issue, the Docker container entrypoint must be **disabled**.

This is done using the `--entrypoint` argument.

Updated Jenkins pipeline configuration:

```groovy
stage('AWS') {
    agent {
        docker {
            image 'amazon/aws-cli'
            args "--entrypoint=''"
        }
    }

    steps {
        sh '''
        aws --version
        '''
    }
}
```

Explanation:

* `args` allows passing arguments to the Docker run command
* `--entrypoint=''` removes the container’s default entrypoint
* Jenkins can now execute commands normally

---

# Successful Execution

After fixing the entrypoint issue, the pipeline runs successfully.

Example output:

```
aws-cli/2.15.53 Python/3.x Linux/amd64
```

This confirms:

* AWS CLI is installed
* The Docker container works correctly
* Jenkins can execute AWS commands

# Practical Notes / Best Practices

## Always Verify Tool Versions

It is recommended to run commands like:

```
aws --version
```

in pipelines to verify the tool version.

This helps debugging when:

* Docker images get updated
* CLI behavior changes
* Pipelines suddenly break

---

## Avoid Using `latest` in Production Pipelines

Although using the **latest tag** is convenient, it can introduce instability.

Example problem:

* Today → AWS CLI version 2.x
* Future → AWS CLI version 3.x
* Commands might change or break

Best practice:

Use a **specific image tag**.

Example:

```groovy
image 'amazon/aws-cli:2.15.53'
```

This ensures consistent pipeline behavior.

---

## Example Jenkinsfile with Fixed Version

```groovy
stage('AWS') {
    agent {

            image 'amazon/aws-cli:2.15.53'
            args "--entrypoint=''"
        }
    }

    steps {
        sh '''
        aws --version
        '''
    }
}
```

# Common Mistakes or Warnings

## Pipeline Hanging Indefinitely

If the pipeline stage runs forever without completing, the cause is often:

* A Docker **entrypoint conflict**

Solution:

```
args "--entrypoint=''"
```

---

## Relying on `latest` Docker Images

Using `latest` may cause pipelines to break unexpectedly.

Always consider:

* Pinning versions
* Logging tool versions in pipelines
