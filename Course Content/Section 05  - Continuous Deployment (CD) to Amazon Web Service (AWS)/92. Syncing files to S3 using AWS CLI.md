# Lecture 92 — Syncing Files to S3 Using AWS CLI

## Overview

In this lecture, the pipeline evolves from uploading a **single file** to deploying the **entire website build directory** to Amazon S3.

Key improvements:

* Replace `aws s3 cp` with **`aws s3 sync`**
* Deploy the **full build output**
* Ensure S3 always reflects the **latest version of the application**
* Integrate deployment into the pipeline flow

This marks the transition from **basic upload → real deployment strategy**.

---

# Key Concepts

## `aws s3 sync` Command

The `sync` command is designed to:

* Synchronize a local directory with an S3 bucket
* Upload only new or modified files
* Optionally delete outdated files

Basic syntax:

```bash
aws s3 sync <source> <destination>
```

Example:

```bash
aws s3 sync build/ s3://my-bucket
```

---

## Why Not Use `cp`?

Previously:

```bash
aws s3 cp index.html s3://bucket/index.html
```

Limitations:

* Only uploads one file at a time
* Requires manual handling of multiple files
* Does not handle deletions

---

## What `sync` Does

`sync` ensures:

> The contents of the local directory and the S3 bucket are aligned.

It will:

* Upload new files
* Update changed files
* (Optionally) remove files no longer present locally

# Detailed Explanation

## Problem with Current Setup

The pipeline currently:

* Creates a single file
* Uploads it using `cp`

But in reality:

* The **build stage produces a `build/` directory**
* This directory contains:

    * HTML
    * CSS
    * JavaScript
    * Assets

Uploading file-by-file is inefficient.

---

## Step 1 — Use `sync` Instead of `cp`

From AWS CLI documentation:

```bash
aws s3 sync . s3://my-bucket
```

Adapted for our pipeline:

```bash
aws s3 sync build/ s3://$AWS_S3_BUCKET
```

---

## Step 2 — Update Jenkins Pipeline

### Remove:

* `echo "Hello S3"`
* `aws s3 cp ...`

### Add:

```bash
aws s3 sync build/ s3://$AWS_S3_BUCKET
```

---

# Jenkinsfile Snapshot (Sync Version)

```groovy
stage('Deploy to AWS') {
    environment {
        AWS_S3_BUCKET = 'learn-jenkins-20240518-1820'
    }

    agent {
        docker {
            image 'amazon/aws-cli'
            args "--entrypoint=''"
            reuseNode true
        }
    }

    steps {
        withCredentials([
            usernamePassword(
                credentialsId: 'my-aws',
                usernameVariable: 'AWS_ACCESS_KEY_ID',
                passwordVariable: 'AWS_SECRET_ACCESS_KEY'
            )
        ]) {
            sh '''
            aws s3 sync build/ s3://$AWS_S3_BUCKET
            '''
        }
    }
}
```

# Critical Issue — Missing Build Directory

## Error Encountered

```bash
The user-provided path build does not exist
```

---

## Root Cause

* Jenkins runs each Docker agent in a **separate workspace**
* The `build/` directory was created in a **previous stage**
* The AWS stage does not see it

---

## Fix — Reuse Workspace

Add:

```groovy
reuseNode true
```

This ensures:

* Same workspace is reused
* Build artifacts are accessible

---

# What Happens Internally

`aws s3 sync`:

* Compares local vs remote files
* Uploads differences
* Skips unchanged files
* Can delete outdated files (if configured)

---

# Step-by-Step Deployment Flow

1. **Build Stage**

    * Generates `build/` directory

2. **Deploy Stage**

    * Uses AWS CLI container
    * Authenticates via credentials
    * Runs:

```bash
aws s3 sync build/ s3://bucket
```

3. **S3 Bucket**

    * Updated with latest files

4. **Website**

    * Serves new version

# Practical Notes / Best Practices

## Always Use `sync` for Deployments

Advantages:

* Efficient
* Automatic
* Scalable

---

## Reuse Workspace in Docker Agents

Without:

```groovy
reuseNode true
```

You risk:

* Missing files
* Broken deployments

---

## Placement in Pipeline

Current setup:

* Deploys immediately after build

⚠️ Not ideal

Better:

* Build → Test → Deploy

---

## Use Logs for Debugging

Logs show:

* Uploaded files
* Skipped files
* Errors

Always inspect logs when debugging deployments.

---

# Common Mistakes or Warnings

## Missing `reuseNode true`

Causes:

```bash
build directory not found
```

---

## Wrong Source Path

Incorrect:

```bash
aws s3 sync wrong-folder/ ...
```

---

## Not Understanding Sync Behavior

* Sync does NOT always delete old files
* Requires additional flags (e.g., `--delete`) for full cleanup

---

## Deploying Without Testing

Current pipeline:

* Skips tests

⚠️ In real CI/CD:

* Never deploy without validation
