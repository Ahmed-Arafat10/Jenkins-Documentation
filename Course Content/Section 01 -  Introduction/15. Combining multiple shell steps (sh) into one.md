### 15. Combining Multiple Shell Steps (`sh`) into One

In Jenkins pipelines, it’s common to execute multiple shell (`sh`) commands such as `mkdir`, `touch`, `echo`, etc. A typical beginner approach is to wrap each command in its own `sh` step. While this works, it is not the most efficient or clean approach.

---

#### ✅ Problem with Multiple `sh` Steps

* Each `sh` invocation creates a **separate shell execution context**.
* Jenkins controller sends each instruction one by one to the agent:

    * Controller: "Run `mkdir`" → Agent runs → sends back result.
    * Controller: "Run `touch`" → Agent runs → sends back result.
* This back-and-forth creates **overhead**.
* In logs, each `sh` call shows as a separate *Shell Script* block, which can clutter logs and make them harder to follow.
* For very small jobs this is negligible, but in larger jobs it can become inefficient.

---

#### ✅ A Better Way: Multi-Line `sh` Block

Instead of writing multiple `sh` steps, you can combine all commands inside a **single `sh` step** using a multi-line script.

**Example:**

```groovy
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                cleanWs()
                echo "Running build steps..."

                sh '''
                    mkdir project
                    touch project/mainboard
                    touch project/keyboard
                    echo "Build complete!"
                '''
            }
        }
    }
}
```

---

#### ✅ Benefits

1. **Efficiency**

    * Only one shell context is created, reducing communication overhead between Jenkins controller and agent.
    * Execution is faster (e.g., build time drops from \~2–3 seconds to nearly instant for small pipelines).

2. **Cleaner Logs**

    * Instead of multiple `Shell Script` blocks, all commands appear inside **one single block**.
    * Easier to trace and debug.

3. **Maintainability**

    * Commands are grouped logically.
    * Easier to add, remove, or modify commands without rewriting multiple `sh` steps.
    * Indentation improves readability.

---

#### ✅ Verification

* After restructuring, the build still produces the correct artifacts (`mainboard`, `keyboard`).
* Logs show commands under one consolidated **Shell Script** section.
* Output artifacts and pipeline results remain unchanged.

---

#### 📌 Key Takeaway

Use a **single multi-line `sh` block** for related commands instead of multiple individual `sh` calls.
This makes your Jenkinsfile **faster, cleaner, and easier to maintain**.
