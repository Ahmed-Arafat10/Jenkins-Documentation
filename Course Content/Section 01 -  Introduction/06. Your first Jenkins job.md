# Lecture 6: Your First Jenkins Job

## Overview

This lecture introduces how to create your first **Jenkins job**. A job in Jenkins is a set of commands or tasks that you want Jenkins to automate. We will cover two types of jobs:

1. **Freestyle Project** – Configured through the user interface.
2. **Pipeline Job** – Configured via pipeline scripts (more powerful and flexible).

---

## Part 1: Creating a Freestyle Job

### Steps

1. From the Jenkins dashboard, click **New Item** (or **Create a Job** if it’s the first time).

2. Enter a name for the job, e.g.:

   ```
   hello-freestyle
   ```

3. Select **Freestyle Project** and click **OK**.

4. Scroll down to **Build Steps → Add build step → Execute shell**.

5. Enter a simple command, for example:

   ```bash
   echo "Hello from Jenkins"
   ```

    * `echo` is a Linux command that prints a string (text).

6. Click **Save**.

7. From the job page, click **Build Now** to run the job.

### Results

* The job runs instantly since the command is simple.
* The **Build History** section shows past runs.
* Click the build number (or green check mark ✅) to open **Console Output**, which displays:

    * Jenkins execution logs.
    * The command you ran.
    * The output of the command (in this case, `"Hello from Jenkins"`).

---

## Part 2: Creating a Pipeline Job

### Steps

1. Return to the dashboard → **New Item**.
2. Enter a name for the job, e.g.:

   ```
   hello-pipeline
   ```
3. Select **Pipeline** and click **OK**.
4. Scroll down to the **Pipeline** section.
5. In the pipeline script editor, you can use a basic pipeline:

```groovy
pipeline {
    agent any
    stages {
        stage('Hello') {
            steps {
                echo 'Hello from Jenkins Pipeline'
            }
        }
    }
}
```

6. Save the job.
7. Click **Build Now** to run it.

### Difference vs. Freestyle

* **Freestyle Job**: Configured entirely through the UI.
* **Pipeline Job**: Defined via **pipeline scripts** (more powerful, supports multiple stages, advanced logic).

---

## Part 3: Running Shell Commands in Pipeline

Instead of using Jenkins’ `echo` step, you can run actual shell commands with `sh`:

```groovy
pipeline {
    agent any
    stages {
        stage('Hello') {
            steps {
                sh 'echo "Hello from Jenkins via Shell"'
            }
        }
    }
}
```

### Example: `whoami` Command

```groovy
pipeline {
    agent any
    stages {
        stage('Check User') {
            steps {
                sh 'whoami'
            }
        }
    }
}
```

* Output will display `jenkins`, since the Jenkins container runs under the `jenkins` user.

---

## Configuration Tips

* **Save vs. Apply**:

    * **Save** → Saves configuration and exits.
    * **Apply** → Saves but keeps you on the same screen (useful if you want to keep editing).
* Each new execution increases the **build number** in build history.

---

## Key Takeaways

* Jenkins jobs = tasks that Jenkins executes (automation units).
* Two main types:

    * **Freestyle** – simple, UI-driven.
    * **Pipeline** – script-driven, flexible, allows multiple stages.
* Console Output is where you verify job execution.
* Pipelines typically consist of **multiple stages** that represent parts of a CI/CD process.
* Use `sh` step for real shell commands in pipeline scripts.

## AI: Notes

* 📌 Pipelines are the preferred modern approach in Jenkins because:

    * They support **version control** (pipeline scripts can be stored in Git).
    * They are **declarative** and can model complex workflows.
    * They enable **multi-stage builds** (e.g., build, test, deploy).
* 📌 You can install the **Blue Ocean plugin** for a more visual representation of pipeline stages.
* 📌 `whoami`, `echo`, `ls`, and `pwd` are common Linux commands you’ll often test within Jenkins jobs.
