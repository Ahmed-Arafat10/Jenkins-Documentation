# 18. Testing Build Artifacts

## Overview

So far, we’ve verified that artifacts exist (using `test`), but **existence alone is not enough**. A valid build artifact must also contain the **correct contents**.
This lecture introduces **`grep`** as a tool to check file contents in Jenkins pipelines.

---

## Key Concepts

### 1. Why File Content Testing Matters

* A file may exist but not contain the required data.
* Example: `computer.txt` must include:

    * `mainboard`
    * `display`
    * `keyboard`

### 2. Why `cat` is Not Enough

* Command `cat` shows file contents.
* Useful for **troubleshooting**.
* **Problem**: If the file is empty or missing expected text, `cat` won’t fail → it just prints nothing.
* Jenkins would still treat this as success because no non-zero exit code is returned.

---

## Using `grep` for Testing

### `grep` Basics

* `grep <pattern> <file>` → searches for `<pattern>` in `<file>`.
* If found: prints the matching line(s) and exits with **code 0** (success).
* If not found: prints nothing and exits with **code 1** (failure).
* This makes it perfect for testing in CI pipelines.

### Example in Pipeline Script

```groovy
stage('Test') {
    steps {
        sh '''
            test -f build/computer.txt
            grep "mainboard" build/computer.txt
            grep "display" build/computer.txt
            grep "keyboard" build/computer.txt
        '''
    }
}
```

* Each `grep` command ensures the file contains the required component.
* If **any check fails**, Jenkins will stop and mark pipeline as failed.

---

## Demonstration from Lecture

1. **Failing on Purpose**

    * Commented out the line adding `mainboard` to the file.
    * `grep "mainboard" build/computer.txt` returned **exit code 1** → Jenkins failed pipeline.

2. **Debugging**

    * To verify what really happened:

        * Inspect Jenkins logs.
        * Navigate to **Workspace** → check actual `computer.txt`.
    * Important: artifacts shown under “Last Successful Artifacts” may belong to a **previous successful build**, not the failed one. Always check **current build workspace**.

3. **Expanding Tests**

    * Added `grep` checks for all required components (mainboard, display, keyboard).
    * Now the pipeline validates both **existence and correctness** of the artifact.

---

## Benefits of Automated Testing in Pipelines

* **Confidence in changes**: You can safely optimize build steps (e.g., removed unnecessary `touch` command).
* **Catch errors early**: Missing or incorrect contents will immediately fail the build.
* **Repeatability**: Every future pipeline run automatically verifies artifact correctness.

---

## Key Takeaways

* `test -f file` checks **existence**.
* `grep pattern file` checks **contents**.
* **Exit codes** drive Jenkins decisions:

    * Found → `0` (success).
    * Not found → `1` (failure).
* Always inspect the **workspace**, not just previous artifacts, when debugging failed builds.
* Automated tests = safer optimizations + faster debugging.

## (AI: Notes)

* You can chain multiple `grep` commands together using `&&` to make the pipeline fail as soon as one check fails.
  Example:

  ```sh
  grep "mainboard" build/computer.txt && \
  grep "display" build/computer.txt && \
  grep "keyboard" build/computer.txt
  ```
* For more complex validation, tools like **`awk`**, **`jq`** (for JSON), or even custom test scripts can be integrated.
