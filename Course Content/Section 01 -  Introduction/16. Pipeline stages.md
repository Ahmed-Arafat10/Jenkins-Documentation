# 📘 Lecture 16 – Pipeline Stages in Jenkins

### 🔹 Problem with current workflow

* Every time we modify the pipeline (build process), we must:

    * Save changes.
    * Run the pipeline.
    * Verify logs to ensure no errors occurred.
    * Manually inspect the artifact (e.g., files) to confirm correctness.
* ❌ This introduces **manual verification overhead**, which is error-prone and inefficient.

---

### 🔹 Introducing a **Test Stage**

* Instead of relying on manual inspection, we add a **test stage** in the pipeline.
* Example syntax in `Jenkinsfile`:

  ```groovy
  stages {
      stage('Build') {
          steps {
              // build steps
          }
      }

      stage('Test') {
          steps {
              echo 'Testing the new laptop'
          }
      }
  }
  ```
* Jenkins UI reflects stages individually → makes tracking easier.

---

### 🔹 Console Output in Jenkins

* **Single log view**: Jenkins aggregates all logs from all stages.
* **Per-stage logs**: Can click on stage names to isolate logs (useful for debugging).

---

### 🔹 Adding a Real Test

* Use **Linux `test` command**:

  ```bash
  test -f /path/to/build/computer.txt
  ```

    * `-f` checks if the file exists.
    * Returns `0` (success) if file exists.
    * Returns `1` (failure) if not found.

* Example in `Jenkinsfile`:

  ```groovy
  stage('Test') {
      steps {
          sh 'test -f build/computer.txt'
      }
  }
  ```

---

### 🔹 Demonstration

1. ✅ If the file exists → test passes, stage succeeds.
2. ❌ If the file is deleted (`rm build/computer.txt`) → test fails, Jenkins reports:

   ```
   Error: script returned exit code 1
   ```

---

### 🔹 Key Concepts

* Jenkins determines stage success/failure based on the **exit code** of shell commands:

    * `0` → success
    * Non-zero → failure
* This allows us to automate validation instead of doing manual artifact inspection.

---

### ⚡ Takeaway

* **Pipeline stages** allow separation of concerns:

    * `Build` → creates artifacts.
    * `Test` → verifies correctness automatically.
* By using shell commands like `test`, we integrate automated checks directly into the CI/CD process.
* Manual verification becomes unnecessary → reducing human error and increasing reliability.
