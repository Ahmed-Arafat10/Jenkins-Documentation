# 17. Why Do Pipelines Fail? Exit Codes Explained

## Overview

This lecture explains **how Jenkins determines whether a pipeline has failed**, focusing on the concept of **exit codes** that commands return after execution. Jenkins uses these exit codes to decide whether to continue executing subsequent stages or mark the pipeline as failed.

---

## Key Concepts

### 1. How Jenkins Detects Failures

* If a command is written incorrectly, Jenkins may fail because the command outputs an error message.
* But sometimes, commands fail **without producing explicit error messages**. Instead, they return a **non-zero exit code**, which Jenkins interprets as failure.
* Jenkins monitors these codes and acts accordingly.

### 2. Exit Codes Explained

* Exit codes are a **convention used by programs** to indicate success or failure.
* **Exit Code 0** → Success (command executed as expected).
* **Exit Code 1–255** → Failure (type of failure depends on the program).

**Examples from lecture:**

* Command `test` → Returned **exit code 1** because the expected file (`build/computer.txt`) did not exist.
* Command `exit 2` → Manually simulates a failure with **exit code 2**.

### 3. Jenkins Behavior

* Jenkins assumes **success (0)** if nothing else is indicated.
* If a command returns a **non-zero exit code**:

    * Jenkins **stops executing further stages**.
    * The pipeline is marked as **failed**.
    * All subsequent stages are **skipped** (with log message: *Stage skipped due to earlier failure*).

---

## Debugging Pipeline Failures

1. **Look at the end of the logs** → Find the last executed command.
2. **Check its exit code** → That’s where the failure originated.
3. **Understand what the command was trying to do** (e.g., test for file existence).
4. **Correct the pipeline logic** → Example: ensure the file is not deleted before being checked.

---

## Practical Demonstration

* **Case 1:** Missing file

    * Command: `test -f build/computer.txt`
    * Result: `exit code 1` → Jenkins fails pipeline.

* **Case 2:** Simulated failure

    * Command: `exit 2` inside pipeline.
    * Result: Jenkins marks build as failed and **skips subsequent stages**.

---

## Key Takeaways

* **Exit codes are the hidden language between commands and Jenkins.**
* Always remember:

    * `0` = success
    * `1–255` = failure
* Jenkins **stops pipeline execution on the first non-zero exit code**.
* To debug:

    * **Check logs carefully**.
    * Identify **last executed command**.
    * Investigate **why it returned a non-zero exit code**.

---

## (AI: Notes)

* Some programs use specific exit codes to represent different types of failures (e.g., `1` for general error, `2` for misuse of shell command, `127` for command not found). It’s useful to know these conventions when debugging.
* Jenkins can be configured to **ignore certain exit codes** or **continue on failure** using constructs like `catchError` or `returnStatus` in pipelines.
