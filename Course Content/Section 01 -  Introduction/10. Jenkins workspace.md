# 10. Jenkins Workspace

## Concept Overview

* A **Jenkins workspace** is a directory on the Jenkins controller (or agent) where Jenkins runs, stores build files, and keeps results.
* By default, jobs reuse the same workspace across multiple builds.
* This reuse can cause issues if leftover files from previous builds remain and affect subsequent runs.

Example issue in the lecture:

* Running the pipeline multiple times resulted in duplicate components (`mainboard`, `display`, `keyboard`) in the output.
* This happened because the `computer.txt` file already existed in the workspace and was appended to, not recreated fresh.

---

## Why It Happens

* The `touch` command creates a file only if it doesn’t exist.
* If the file already exists, it simply updates the timestamp while keeping the contents.
* Since Jenkins reuses the workspace, the file persisted between builds.

---

## Workspace Management Approaches

1. **Manual Cleanup in Pipeline**

    * Add shell steps at the beginning of a job to delete specific files or directories.
    * Example:

      ```groovy
      stage('Clean Start') {
        steps {
          sh 'rm -rf build/*'
        }
      }
      ```
    * Limitation: You must know which files to remove, which is not always feasible for large or complex projects.

2. **Jenkins Native Cleanup**

    * Jenkins provides a **post-build action** to automatically clean the workspace.
    * Ensures no stale or unnecessary files remain for the next build.
    * Prevents unpredictable behavior caused by leftover files.

---

## Post-Build Actions in Jenkins Pipelines

* **Definition:** Actions executed after all stages are complete, regardless of build success or failure.
* Syntax structure:

  ```groovy
  pipeline {
    agent any

    stages {
      stage('Build') {
        steps {
          // build commands here
        }
      }
    }

    post {
      always {
        cleanWs()
      }
    }
  }
  ```
* **Key points:**

    * `post` must be at the same indentation level as `stages`.
    * `always` ensures the action runs whether the build succeeds or fails.
    * `cleanWs()` is the official Jenkins Pipeline command to delete everything in the workspace.

---

## Example Run

1. Pipeline executes build steps → `computer.txt` is created with components.
2. Build finishes.
3. **Post-build action triggers:**

    * Logs show `Workspace cleanup`.
    * Workspace directory is deleted.
4. On the next run:

    * Jenkins starts with a fresh workspace.
    * Output only contains the intended build result (`mainboard`, `display`, `keyboard`) without duplication.

---

## Benefits of Workspace Cleanup

* Prevents unexpected side effects from leftover files.
* Ensures reproducibility of builds.
* Keeps Jenkins controller/agent storage usage clean and efficient.

---

## AI: Notes

* **Best practice:** For production pipelines, use `cleanWs()` at either the beginning or end of pipelines to avoid stale files.
* In large-scale projects, cleaning the workspace every build might increase build time slightly (due to re-downloading dependencies), so some teams use it selectively (e.g., only on failure or nightly builds).
* Jenkins also provides **"Discard Old Builds"** options in job configuration to clean old workspaces and artifacts, which helps with disk management.
