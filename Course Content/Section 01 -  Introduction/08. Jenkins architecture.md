# Jenkins Architecture

When running a Jenkins job, two main components are involved:

### 1. **Jenkins Controller (Server)**

* Also known as the **Jenkins Master** in older terminology.
* Responsible for:

    * Managing the execution of pipelines.
    * Scheduling jobs.
    * Storing build results, configurations, and job definitions.
* The controller **knows what needs to be done**, but it does **not execute the actual tasks**.
* Think of it like **MI6 in James Bond movies**—it coordinates and manages the mission but does not perform the fieldwork itself.

---

### 2. **Jenkins Agent (Node)**

* A program responsible for **executing the actual job steps**.
* Agents receive instructions from the controller, execute commands, and send back the results.
* A Jenkins setup must have **at least one agent**, but usually multiple agents are used to distribute workloads efficiently.

---

### 3. **Controller-Agent Interaction**

* The controller delegates execution tasks to agents.
* Agents:

    * Fetch the job instructions.
    * Run the required stages (build, test, deploy, etc.).
    * Report back the execution results to the controller.

---

### 4. **Simple vs. Production Setup**

* **Simple Setup (Local Jenkins Installation):**

    * The same instance plays both roles: controller **and** agent.
    * Useful for testing, learning, or small-scale use.
* **Production Setup (Best Practice):**

    * The controller is kept **lightweight**—it should **not execute jobs directly**.
    * Separate agents (on different machines/containers/cloud instances) handle execution to avoid overloading the controller.

---

✅ **Summary:**

* **Controller = Brain** → schedules, manages, stores.
* **Agent = Hands** → executes, runs, reports back.
* For production, always separate **controller** and **agents** to scale efficiently and follow best practices.
