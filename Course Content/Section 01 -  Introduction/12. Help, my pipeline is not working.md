# 12. Help, My Pipeline Is Not Working

## Overview

In Jenkins, you’ll spend **more time debugging pipelines** than writing them. This lecture reinforces practical techniques to identify and fix issues quickly: noticing editor warnings, understanding syntax errors, and reading logs methodically. The golden rule: **use the logs and proceed in small steps**.

---

## 1) Catching Problems in the Editor

* The Jenkins pipeline editor flags obvious syntax issues.
* **Missing braces example:** if you remove a `}` you may see a small **“x”** near the end of the editor (easy to miss).
* Even if you don’t notice the warning and click **Save** then **Build**, Jenkins will fail **immediately**.

### What failure looks like

* Console shows an early error like **“startup failed”** before any of your steps run.
* The error often references a **line number** (e.g., “line 27 expected …”). This can be **confusing or slightly off**.

### How to pinpoint syntax issues

* Don’t rely only on the line number. Use structure:

    * Verify block boundaries by **indentation** and by **collapsing/expanding** sections:

        * `pipeline { … }`
        * `stages { … }`
        * `stage('Name') { … }`
        * `steps { … }`
* Common mistakes:

    * **Mismatched/missing braces**.
    * **Misspelled keywords** (e.g., `stage` vs `stages`).
* **False editor warnings** can happen:

    * If you believe the script is valid, **Save**, leave, and **reopen** the configuration; warnings may clear.

---

## 2) Reading Logs Like a Pro

* During a build, you can click the **progress bar** on the job to jump straight to **Console Output**.
* Always read **top → bottom** to understand the sequence.
* For each step:

    * Identify the **command** that ran.
    * Check the **output** or **error** line(s) beneath it.
* Typical early parse error: **no steps ran** and you see **“startup failed”**—that’s a Jenkinsfile problem, not your shell command.
* For runtime errors, find the **exact step** (e.g., `sh '…'`) that failed, note the message, and adjust that command.

---

## 3) Compare Good vs. Bad Runs

* Open the **last successful** build’s logs side-by-side with the **failing** one.
* Ask:

    * What **changed** since the last success?
    * Where do the logs **diverge**?
* Make **small, incremental edits**, then rebuild. This isolates the cause and speeds up fixes.

---

## 4) Practical Debugging Workflow (Checklist)

1. **Syntax sanity**

    * Balance braces, verify `pipeline/stages/stage/steps` structure.
    * Look for the tiny **“x”** markers and fix them.
2. **Save → Build**

    * If it fails instantly with “startup failed”, fix the Jenkinsfile structure.
3. **Use Console Output**

    * Click progress bar → read logs top-to-bottom.
    * Find the first real error and the step that produced it.
4. **Minimal change, re-run**

    * Change one thing, build again.
    * Compare with the last successful run.
5. **Repeat** until green ✅.

---

## Key Takeaways

* Expect to **debug a lot**—that’s normal with Jenkins.
* Editor warnings help, but **logs are the source of truth**.
* Line numbers in errors help but can be **misleading**; verify structure via indentation and folding.
* Proceed in **small steps**; compare against the last green build.

---

## AI: Notes

* Use the **Declarative Pipeline Linter** (often available via “Pipeline Syntax” or at `/pipeline-model-converter/validate`) to validate your Jenkinsfile text before saving.
* Add more visibility in shell steps:

  ```groovy
  sh 'set -euxo pipefail; your commands here'
  ```

  This makes failures explicit (`-e`) and echoes commands (`-x`).
* Capture exit codes without failing the build to inspect behavior:

  ```groovy
  def code = sh(script: 'your command', returnStatus: true)
  echo "Exit code: ${code}"
  ```
* If available, **Blue Ocean** or **Stage View** provides clearer stage-by-stage logs.
* When a warning looks wrong, **save & reload** the editor; minor UI lint glitches can occur.
