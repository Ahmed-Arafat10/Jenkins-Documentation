# 19. Defining Environment Variables

## Overview

In Jenkins pipelines, repeating values (like filenames, paths, or configurations) directly in multiple places makes maintenance harder. Instead, we can define **environment variables** and reuse them throughout the pipeline. This avoids duplication, improves readability, and simplifies future changes.

In this lecture, we fixed a mistake where `computer.txt` was used everywhere, even though the build was actually for `laptop.txt`. Instead of replacing each occurrence manually, we created and used an environment variable.

---

## Defining Environment Variables

* Environment variables in Jenkins pipelines are defined in a special block called `environment`, placed right after the `agent` and before `stages`.

### Syntax:

```groovy
pipeline {
    agent any

    environment {
        BUILD_FILE_NAME = "laptop.txt"
    }

    stages {
        ...
    }
}
```

### Conventions:

* Use **uppercase letters** for variable names.
* Use underscores (`_`) to separate words.
* Assign the value with `=` and wrap it in **quotes**.

This makes `BUILD_FILE_NAME` available globally across all stages for the entire job execution.

---

## Using Environment Variables

### Inside Jenkins Scripted Steps

* To access a variable:

  ```groovy
  echo "${BUILD_FILE_NAME}"
  ```
* Always use a **dollar sign (`$`)** before the variable name.
* Variables work best when wrapped in **double quotes**, since single quotes will prevent variable substitution.

#### Examples:

```groovy
// Correct - variable is rendered
echo "${BUILD_FILE_NAME}"   // Output: laptop.txt

// Incorrect - treated as literal string
echo '${BUILD_FILE_NAME}'   // Output: ${BUILD_FILE_NAME}
```

### Without Quotes

* Using variables without quotes inside Jenkins DSL (like `echo $BUILD_FILE_NAME`) may lead to errors such as:

  ```
  No such property: BUILD_FILE_NAME
  ```
* Always wrap with quotes when using `echo` in the pipeline script itself.

---

## Using Variables Inside Shell Steps

Within `sh` (shell execution steps), environment variables behave differently:

```groovy
sh '''
    echo $BUILD_FILE_NAME
'''
```

* Here, `$BUILD_FILE_NAME` is interpreted by the shell correctly.
* Unlike `echo` inside the Groovy pipeline script, quotes are not strictly necessary in `sh`.

---

## Replacing Hardcoded Values

After confirming the variable works, replace every instance of the hardcoded value (`computer.txt`) with `$BUILD_FILE_NAME`.

Example:

```groovy
stages {
    stage('Build') {
        steps {
            sh 'cat $BUILD_FILE_NAME'
        }
    }
    stage('Test') {
        steps {
            sh 'grep "expected" $BUILD_FILE_NAME'
        }
    }
}
```

---

## Debugging & Testing

* After updating variables, rerun the pipeline to confirm correct behavior.
* If tests still pass, manually verify the output file or logs to ensure values are updated correctly.
* Refresh Jenkins’ job page to ensure the updated value (e.g., `laptop.txt`) appears instead of the old one.

---

## Key Takeaways

1. **Don’t repeat yourself (DRY principle):** use environment variables instead of hardcoding values.
2. **Always use uppercase names** for pipeline variables.
3. **Quotes matter:**

    * Double quotes → variable substitution works.
    * Single quotes → literal text, no substitution.
4. **Shell vs Jenkins script difference:**

    * In `echo` inside Groovy → must use quotes.
    * In `sh` block → `$VAR` works normally.
5. **Verify correctness:** both through automated tests and occasional manual checks.

## (AI: notes)

* Jenkins distinguishes between **Groovy string interpolation** (inside pipeline `echo`) and **shell variable substitution** (inside `sh`). This is a frequent beginner mistake.
* It’s a best practice to keep all filenames, credentials, paths, and external resource references as environment variables. This makes pipelines cleaner and easier to migrate.
* For secrets (like passwords, API keys), Jenkins provides **Credentials Binding** rather than plain environment variables (covered in later topics).
