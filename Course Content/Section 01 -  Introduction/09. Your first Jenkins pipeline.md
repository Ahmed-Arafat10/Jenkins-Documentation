# Your First Jenkins Pipeline

## 1. What is a Pipeline?

A **pipeline** in Jenkins is similar to an **assembly line in manufacturing**.
Just like building a laptop involves multiple steps (e.g., adding the body, mainboard, keyboard, quality checks, packaging, shipping), software development also requires **sequential stages** (build, test, deploy, etc.).

Pipelines allow us to **automate** these steps in a structured, repeatable way.

---

## 2. Creating the First Pipeline Job

1. Create a new job in Jenkins:

    * Name: `laptop-assembly`
    * Type: **Pipeline**

2. Select **Pipeline script** and start with the "Hello World" template.

3. Replace `"Hello"` with `"Build"` → this becomes the **build stage**.

---

## 3. Building the Example Pipeline (Laptop Assembly Line)

### Stage: Build

Inside the **steps**, we run commands:

#### Example Script:

```groovy
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                echo 'Building a new laptop'

                // Create build directory (with -p to avoid errors if it exists)
                sh 'mkdir -p build'

                // Create a file named computer.txt inside build directory
                sh 'touch build/computer.txt'

                // Add "mainboard" text to the file
                sh 'echo mainboard >> build/computer.txt'

                // Display contents of file
                sh 'cat build/computer.txt'
            }
        }
    }
}
```

---

## 4. Explanation of Key Commands

* `echo`: Prints messages or adds text to a file.
* `mkdir build`: Creates a directory named *build*.

    * `mkdir -p build` → prevents error if directory already exists.
* `touch build/computer.txt`: Creates a file or updates its timestamp.
* `echo mainboard >> build/computer.txt`: Appends text (`mainboard`) to the file.
* `cat build/computer.txt`: Displays file contents.

---

## 5. Debugging and Logs

* Jenkins provides logs under **Build History → Console Output**.
* Always read logs **top-to-bottom** to find the error’s origin.
* Example error:

  ```
  Cannot create a directory build: File exists
  ```

    * Fixed by adding `-p` to `mkdir`.

---

## 6. Extending the Pipeline

You can add more **stages** to simulate adding laptop parts:

```groovy
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                sh 'mkdir -p build'
                sh 'touch build/computer.txt'
                sh 'echo mainboard >> build/computer.txt'
                sh 'cat build/computer.txt'
            }
        }
        stage('Add Display') {
            steps {
                sh 'echo display >> build/computer.txt'
                sh 'cat build/computer.txt'
            }
        }
        stage('Add Keyboard') {
            steps {
                sh 'echo keyboard >> build/computer.txt'
                sh 'cat build/computer.txt'
            }
        }
    }
}
```

---

## 7. Observed Behavior

* Running multiple builds appends content repeatedly:

    * Example after 3 runs:

      ```
      mainboard
      mainboard
      mainboard
      display
      keyboard
      ```
* This happens because Jenkins **does not clean the workspace by default** between builds.

---

## 8. Key Takeaways

* Pipelines = repeatable, automated workflows.
* Each **stage** represents a logical step (build, add components, test, etc.).
* Use Linux shell commands (`echo`, `mkdir`, `touch`, `cat`) to manipulate files.
* Debug issues by **checking Jenkins logs**.
* Use `mkdir -p` to avoid directory re-creation errors.

---

### (AI: Notes)

* In **real-world pipelines**, instead of simulating with files, you would:

    * **Build**: Compile code, install dependencies.
    * **Test**: Run automated tests.
    * **Package**: Create artifacts (e.g., `.jar`, `.war`, `.zip`).
    * **Deploy**: Push artifacts to staging/production environments.
* To avoid duplicate file content, Jenkins workspaces are often **cleaned before builds** using:

    * The **"Clean Workspace" plugin**, or
    * Adding `cleanWs()` in the pipeline.
