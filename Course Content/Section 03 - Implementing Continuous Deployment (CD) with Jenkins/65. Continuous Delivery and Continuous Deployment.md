## Lecture 65 – Continuous Delivery and Continuous Deployment

### Why a manual approval step can be useful

Even in highly automated pipelines, a manual approval step may be required before production deployment.

Common reasons include:

#### 1. Quality Assurance (QA)

Automated tests:

* validate functionality
* check integration
* verify APIs and workflows

However, they may not detect:

* visual inconsistencies
* content issues
* UX problems
* business rule nuances

A manual approval stage allows:

* a final human review
* verification of non-functional requirements
* confirmation that everything meets business expectations

---

#### 2. Risk Management

Production deployments can:

* impact revenue
* affect user experience
* damage reputation if something fails

The approval step acts as:

* a risk checkpoint
* a safeguard before going live

If something looks suspicious in staging:

* production deployment can be prevented

---

#### 3. Regulatory and Compliance Requirements

Some industries require:

* documented approvals
* audit trails
* change logs
* legal validation

The manual approval step:

* records who approved
* logs timestamps
* creates traceability

This is especially common in:

* finance
* healthcare
* government systems

---

#### 4. Stakeholder Sign-off

In larger organizations:

* business stakeholders may need to approve changes
* significant feature updates require authorization
* non-technical approval may be mandatory

The approval stage:

* provides structured sign-off
* integrates business governance into CI/CD

---

# Continuous Delivery

### Definition

Continuous Delivery means:

* the entire build → test → deploy pipeline is automated
* every successful build is production-ready
* deployment to production requires a **manual trigger or approval**

There is always:

* a releasable artifact
* fully tested
* ready to go live

---

### Example: Continuous Delivery pipeline

```groovy
pipeline {
  agent any

  stages {

    stage('Build') {
      steps {
        echo 'Building application'
      }
    }

    stage('Test') {
      steps {
        echo 'Running tests'
      }
    }

    stage('Deploy Staging') {
      steps {
        echo 'Deploying to staging'
      }
    }

    stage('Approval') {
      steps {
        timeout(time: 30, unit: 'MINUTES') {
          input message: 'Approve deployment to production?',
                ok: 'Deploy to production'
        }
      }
    }

    stage('Deploy Production') {
      steps {
        echo 'Deploying to production'
      }
    }
  }
}
```

Characteristics:

* fully automated pipeline
* production-ready build available
* deployment requires explicit approval

---

### Key Principle of Continuous Delivery

At any time:

* there must be a build that can be deployed
* deployment is safe and repeatable
* the release is controlled

---

# Continuous Deployment

### Definition

Continuous Deployment builds on Continuous Delivery, but:

* removes the manual approval step
* automatically deploys to production if all checks pass

If:

* build succeeds
* tests pass
* staging validation passes

Then:

* production deployment happens automatically

No human intervention.

---

### Example: Continuous Deployment pipeline

```groovy
pipeline {
  agent any

  stages {

    stage('Build') {
      steps {
        echo 'Building application'
      }
    }

    stage('Test') {
      steps {
        echo 'Running tests'
      }
    }

    stage('Deploy Staging') {
      steps {
        echo 'Deploying to staging'
      }
    }

    stage('Staging E2E') {
      steps {
        echo 'Running staging tests'
      }
    }

    stage('Deploy Production') {
      steps {
        echo 'Deploying to production automatically'
      }
    }

    stage('Production E2E') {
      steps {
        echo 'Validating production deployment'
      }
    }
  }
}
```

There is:

* no approval stage
* no pause
* no manual gate

---

# Real-World Continuous Deployment

Large organizations:

* deploy many times per day
* sometimes multiple times per hour
* automate rollouts entirely

Users typically:

* do not notice deployments
* experience seamless updates

This requires:

* extremely reliable tests
* strong observability
* automated rollback mechanisms

---

# Transitioning from Continuous Delivery to Continuous Deployment

In this pipeline, converting to Continuous Deployment means:

Remove:

```groovy
stage('Approval') {
  steps {
    timeout(time: 15, unit: 'MINUTES') {
      input message: 'Approve deployment?',
            ok: 'Deploy'
    }
  }
}
```

Pipeline becomes fully automated.

---

# Practical Considerations

In reality, removing approval is not trivial.

Organizations adopting DevOps often:

* hesitate to relinquish control
* require cultural change
* need strong confidence in test coverage

Continuous Deployment requires:

* high-quality automated tests
* stable infrastructure
* monitoring and alerting
* rollback strategies

---

### (Added by me – not mentioned in the lecture)

For true Continuous Deployment in production systems, teams typically implement:

* automated rollback if health checks fail
* feature flags for gradual rollout
* blue-green or canary deployments
* monitoring-based deployment verification

Without these safeguards, removing manual approval can introduce unacceptable risk.
