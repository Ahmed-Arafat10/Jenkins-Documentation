## Lecture 55 – Post-deployment tests

### Why post-deployment tests are needed

At this point in the pipeline, the application is:

* built
* tested
* deployed

However, there is still a **manual step**:

* opening the production website
* visually checking if the application works

This lecture replaces that **manual verification** with **automated post-deployment tests**, making the pipeline truly end-to-end automated.

The goal is:

* deploy to production
* automatically verify the deployed application
* fail the pipeline if production is broken

---

## Reusing existing end-to-end tests

The pipeline already contains an **end-to-end (E2E) test stage** that:

* uses Playwright
* runs against a **locally started website**
* starts the site using `serve`

Instead of creating new tests, we **reuse the same Playwright tests**, but:

* run them **against the production deployment**
* remove anything related to running the site locally

---

## Creating a production E2E stage

### Step 1: Duplicate the existing E2E stage

* Copy the entire existing E2E stage
* Paste it **after the deploy stage**
* Fix indentation using `Shift + Tab` twice
* Rename the stage to:

```
prod e2e
```

This makes it clear these tests run against **production**.

---

## Differentiating HTML test reports

Both local E2E and production E2E tests generate **HTML reports**.

If both use the same report name:

* Jenkins will show two reports
* It will be impossible to know which is which

### Solution: Rename reports

* Production report name:

  ```
  Playwright E2E
  ```
* Local report name:

  ```
  Playwright Local
  ```

Now Jenkins clearly shows:

* which report is for local tests
* which report is for production tests

---

## Cleaning up the production E2E stage

The production environment is already running (Netlify), so:

### What is no longer needed

* `serve`
* local server startup
* local installation steps

### Final steps inside `prod e2e`

Only keep:

```
npx playwright test
```

---

## How Playwright decides which URL to test

The target URL is defined in:

```
playwright.config.js
```

Inside this file:

* tests use a `baseURL`
* the value is selected like this:

```
CI_ENVIRONMENT_URL (environment variable)
OR
http://localhost:3000 (fallback)
```

Meaning:

* If `CI_ENVIRONMENT_URL` exists → Playwright uses it
* If not → Playwright tests localhost

This is why the **same tests can run locally or against production**.

---

## Providing the production URL to Playwright

The production site is deployed to Netlify and has a URL like:

```
<random-name>.netlify.app
```

To make Playwright use this URL:

* set an environment variable:

  ```
  CI_ENVIRONMENT_URL
  ```

---

## Important environment variable scope problem

At the top of the pipeline, there are already environment variables:

* Netlify Site ID
* Netlify Auth Token

Variables defined at the **top-level environment block**:

* are available to **all stages**

If `CI_ENVIRONMENT_URL` is defined there:

* local E2E tests will also run against production
* this is **not desired**

---

## Correct solution: stage-level environment variables

### Approach

* Remove `CI_ENVIRONMENT_URL` from the global environment
* Define it **only inside the `prod e2e` stage**

### Result

* Local E2E tests → use `localhost`
* Production E2E tests → use Netlify URL

### Example structure (conceptual)

```
stage('prod e2e') {
  environment {
    CI_ENVIRONMENT_URL = '<netlify-url>'
  }
  steps {
    npx playwright test
  }
}
```

Now:

* the variable exists only during production testing
* test behavior is fully controlled

---

## URL consistency check

Inside the Playwright project:

* `localhost` does **not** end with `/`

To stay consistent:

* remove the trailing `/` from the production URL
* prevents subtle mismatches or redirects

---

## Running the updated pipeline

After committing the changes:

* the pipeline runs again
* a new stage appears:

  ```
  prod e2e
  ```

In Jenkins UI:

* two HTML reports are visible:

    * Playwright Local
    * Playwright E2E

This confirms:

* tests ran twice
* against two different environments

---

## Verifying tests actually run against production

A green pipeline alone is **not enough**.

> If tests never fail, even when they should, the pipeline is useless.

### Principle

Always **test the tests**.

---

## Simulating a broken production deployment

### Steps performed in Netlify

1. Open **Deploys**
2. Scroll to **Upload deploy**
3. Upload a different application build (provided ZIP)
4. Confirm upload

Now:

* production site shows a **different application**
* this simulates an incorrect deployment

---

## Re-running only the production E2E stage

Instead of running the entire pipeline:

* click **Restart from stage**
* select:

  ```
  prod e2e
  ```

Result:

* all previous stages are skipped
* only production E2E tests run

This is a fast way to:

* re-validate production
* without modifying pipeline code

---

## Observing test failure

The production E2E stage:

* fails as expected

Checking the HTML report shows:

* tests expecting title **“Learn Jenkins”**
* production site does not match
* assertions fail

This confirms:

* tests truly run against production
* failures are detected correctly

---

## Final outcome

Post-deployment tests:

* fully replace manual production checks
* automatically validate the deployed application
* fail the pipeline if production is wrong

The pipeline now:

* builds
* tests
* deploys
* **verifies production automatically**

This completes a **true end-to-end CI/CD pipeline** using Jenkins and Playwright.

---

### (Added by me – not mentioned in the lecture)

In real production pipelines, post-deployment tests are often:

* run against a **temporary production URL**
 or executed as **smoke tests** (small, critical checks only)
  to reduce risk and execution time while still catching major failures.
