# 54. Build Triggers – SCM Polling (Git Polling)

## Why We Need a Better Trigger

So far, pipelines have been triggered in two ways:

* **Manually** by clicking **Build Now**
* **Periodically** using a cron schedule

Neither is ideal for real CI.

What we actually want is:

> **The pipeline should start automatically when code changes.**

When we push code to **GitHub**, GitHub knows about the change — **but Jenkins does not**.

---

## Why Not GitHub Webhooks (Direct Notifications)

The *best* solution would be for GitHub to notify Jenkins directly using **webhooks**.

However, this requires:

* Jenkins to be **publicly accessible**
* Extra **security configuration**
* Additional setup complexity

Because of this (and security concerns), webhooks are **out of scope** for this course.

> If you want to implement webhooks later, check the **resources section**.

---

## What Is SCM Polling?

Since Jenkins cannot automatically detect repository changes, it can **periodically check (poll)** the repository.

This mechanism is called:

> **Poll SCM (Source Code Management polling)**

### How It Works

* Jenkins checks the Git repository on a schedule
* If **no changes** → nothing happens
* If **new commits are found** → pipeline is triggered

This gives us **near-automatic builds** without exposing Jenkins to the internet.

---

## Enabling SCM Polling

1. Open the Jenkins job
2. Click **Configure**
3. Scroll to **Build Triggers**
4. Enable **Poll SCM**

You will see a **schedule field**, just like with **Build periodically**.

---

## SCM Polling Uses Cron Syntax

SCM polling uses **the exact same cron syntax** as scheduled builds:

```
MINUTE  HOUR  DAY_OF_MONTH  MONTH  DAY_OF_WEEK
```

But instead of *running the build*, Jenkins is only **checking for changes**.

---

## Example: Poll Every 15 Minutes

Jenkins provides example cron expressions.

To poll every 15 minutes:

```
H/15 * * * *
```

This means:

* `H/15` → every 15 minutes (spread using hash)
* The rest → every hour, every day, every month

Jenkins will show:

* Last poll time
* Next scheduled poll

---

## Polling Every Minute (Important Gotcha)

### ❌ Incorrect (polls once per hour)

```
H/1 * * * *
```

This looks like “every minute” — but it is **not**.

This actually means:

> Every hour at a hashed minute

You’ll see Jenkins polling at:

* 8:56
* 9:56
* 10:56

---

### ✅ Correct (poll every minute)

```
* * * * *
```

This means:

* Every minute
* Every hour
* Every day
* Every month

Jenkins will now check the repository **once per minute**.

---

## Saving and Observing SCM Polling

After clicking **Save**:

* No build starts immediately
* Jenkins waits until the next polling interval

A new menu item appears:

👉 **Git Polling Log**

---

## Git Polling Log

This log shows:

* When Jenkins checked the repository
* Whether changes were detected

Initially, it will say:

```
Polling has not run yet
```

After one minute, it will show:

```
No changes
```

---

## Triggering a Build via SCM Polling

Any repository change will trigger the pipeline:

* Editing the **Jenkinsfile**
* Changing source code
* Adding a new file
* Modifying configuration files

### Example

1. Make a small change (e.g., add an `echo "small change"`)
2. Commit and push to GitHub
3. Wait up to **one minute**
4. Jenkins automatically starts a build

---

## How to Confirm the Trigger Source

Open the build logs and look at the top.

### SCM-triggered build

```
Started by SCM change
```

### Manual build

```
Started by user <username>
```

This confirms that **SCM polling triggered the pipeline**, not a human.

---

## Important Notes About SCM Polling

* Jenkins checks the repository **even if nothing changed**
* Polling too frequently can:

    * Increase load on Jenkins
    * Increase GitHub API usage
* 1-minute polling is fine for learning and small projects
* Larger systems often prefer **webhooks**

---

## Summary

* **SCM Polling** checks Git for changes on a schedule
* Uses **cron syntax**
* Triggers builds **only when changes exist**
* Safer and simpler than webhooks
* Ideal for:

    * Learning CI/CD
    * Private Jenkins installations
    * No public exposure

This is the **closest thing to automatic CI** without webhook configuration and a major step forward from manual builds.
