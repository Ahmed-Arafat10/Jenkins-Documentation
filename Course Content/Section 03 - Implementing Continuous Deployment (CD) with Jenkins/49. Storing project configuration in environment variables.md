# 49. Storing Project Configuration in Environment Variables

## Purpose of This Lecture

Before Jenkins can deploy the application to Netlify, it must know **which Netlify site** it should deploy to.

This lecture explains:

* How Netlify identifies a site
* Why **environment variables** are used for configuration
* How to define and use environment variables in a `Jenkinsfile`
* How Netlify CLI automatically detects configuration from environment variables

---

## Why Configuration Is Needed Before Deployment

When deploying manually via Netlify’s UI:

* You visually select a site
* Upload files
* Netlify knows where to deploy

In **automation**, Jenkins cannot “select” a site visually.

So we must explicitly tell Netlify CLI:

> “Deploy this build to **this specific site**”

This is done using an **environment variable**.

---

## Netlify Site Identification

### Site Name vs Site ID

Netlify provides:

* **Site name** (human-friendly, changeable)
* **Site ID** (machine-friendly, immutable)

Key point:

* ❌ Site name is **not reliable**
* ✅ Site ID is **unique and stable**

Therefore, **Site ID must be used for automation**.

---

## Where to Find the Site ID in Netlify

Steps in Netlify UI:

1. Select the correct site
2. Go to **Site overview**
3. Open **Site configuration**
4. Locate **Site ID**
5. Copy the Site ID value

> Each Netlify account and site has a **different Site ID**

---

## Why Use Environment Variables

Environment variables are ideal because:

* They centralize configuration
* They avoid hardcoding values in commands
* They are easy to change
* Tools like Netlify CLI automatically detect them

This follows best practices for CI/CD pipelines.

---

## Defining an Environment Variable in Jenkinsfile

Inside the `Jenkinsfile`, an `environment` block is added:

```groovy
environment {
    NETLIFY_CI_SITE_ID = 'your-site-id-here'
}
```

### Important Rules

* Variable name **must be exactly**:
  `NETLIFY_CI_SITE_ID`
* Must be **uppercase**
* No spelling mistakes
* Value must be the **exact Site ID from Netlify**

Why the name matters:

* Netlify CLI automatically checks for this variable
* If present, it knows which site to deploy to
* No extra configuration is required later

---

## Scope of the Environment Variable

The `environment` block:

* Applies to the **entire pipeline**
* Is accessible from **all stages**
* Behaves like a system environment variable during execution

---

## Verifying the Environment Variable

Before actually deploying, a simple validation step is added.

In the `deploy` stage:

```bash
echo "Deploying to production site ID: $NETLIFY_CI_SITE_ID"
```

This helps verify:

* The variable is defined correctly
* Jenkins can access it
* The value is being interpolated correctly

---

## Pipeline Execution Result

After committing and running the pipeline:

* Pipeline executes successfully
* Deploy stage logs show:

```
Deploying to production site ID: <your-site-id>
```

This confirms:

* Environment variable is set
* Jenkinsfile syntax is correct
* Netlify CLI will be able to use this value later

---

## Why This Step Is Critical

Without this configuration:

* Netlify CLI would not know where to deploy
* Deployment would fail or target the wrong site

This lecture establishes:

* **Deterministic deployment**
* **Environment-based configuration**
* **Separation of configuration from logic**

---

## Best Practice Highlight

**AI: notes**
In real-world projects:

* Site IDs, tokens, and secrets are often stored in **Jenkins credentials**, not directly in the Jenkinsfile
* This lecture intentionally stores the value in the Jenkinsfile to clearly demonstrate how environment variables work
* Secure handling of secrets will typically be introduced later in CI/CD setups

---

## Key Takeaways

* Automation requires explicit configuration
* Netlify Site ID uniquely identifies a deployment target
* Environment variables are the correct way to store configuration
* `NETLIFY_CI_SITE_ID` is automatically detected by Netlify CLI
* Always validate configuration before relying on it in deployment

This step completes the **configuration phase** and prepares the pipeline for **actual automated deployment** in the next lectures.
