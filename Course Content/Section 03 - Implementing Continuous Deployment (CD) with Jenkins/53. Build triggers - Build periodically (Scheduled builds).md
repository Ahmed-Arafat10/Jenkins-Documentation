# 53. Build Triggers – Build Periodically (Scheduled Builds)

## Purpose of Build Triggers

So far, the pipeline has been triggered **manually** using the **Build Now** button.
While this works, relying only on manual execution is inefficient and outdated for modern CI/CD workflows.

Jenkins provides several **automatic build trigger mechanisms**, and in this lecture we focus on **Build periodically**, which allows pipelines to run on a fixed schedule.

---

## Where to Configure Scheduled Builds

To configure scheduled builds:

1. Open the Jenkins job
2. Click **Configure**
3. Scroll down to **Build Triggers**
4. Enable **Build periodically**

Once enabled, Jenkins asks for a **schedule definition**.

---

## What “Build Periodically” Is Used For

Scheduled builds are useful in several scenarios:

* **Long-running jobs**
  Pipelines that take hours can be scheduled overnight so results are ready by morning.

* **Maintenance pipelines**
  Even if code hasn’t changed, dependencies, base images, or environments may have changed.

* **Stability checks**
  Ensures the pipeline itself is still valid and functional.

This makes scheduled builds valuable even when development activity is low.

---

## Jenkins Uses Cron Syntax

The schedule is defined using **cron syntax**, not plain English.

The syntax consists of **five fields**, separated by spaces:

```
MINUTE  HOUR  DAY_OF_MONTH  MONTH  DAY_OF_WEEK
```

### Field Ranges

* **Minute**: `0–59`
* **Hour**: `0–23`
* **Day of month**: `1–31`
* **Month**: `1–12`
* **Day of week**: `0–7`
  (`0` and `7` = Sunday, `1` = Monday)

---

## Example: Run Every Weekday at 3:00 AM

### Step-by-step breakdown

**Requirement:**
Run the pipeline every weekday (Monday–Friday) at 3:00 AM.

### Initial cron expression

```
0 3 * * 1-5
```

Explanation:

* `0` → minute (exactly on the hour)
* `3` → hour (3 AM)
* `*` → every day of the month
* `*` → every month
* `1-5` → Monday to Friday

This works, but Jenkins will display a warning.

---

## Jenkins Warning: Load Spreading

Jenkins warns when many jobs are scheduled to start **at the exact same time**, as this can overload the server.

To avoid this, Jenkins recommends using the **`H` (hash)** symbol.

---

## Using `H` to Spread Load

Replace the minute field with `H`:

```
H 3 * * 1-5
```

### What `H` Does

* Jenkins computes a **stable, deterministic minute value**
* The value depends on the **job name**
* Each job runs at a **slightly different minute**
* Prevents resource spikes at exact times like `03:00`

Example result:

* Job A → runs at `03:56`
* Job B → runs at `03:12`

The job still runs **every weekday around 3 AM**, but without overload risk.

---

## How Jenkins Displays the Schedule

After entering the cron expression, Jenkins shows:

* **Last run time**
* **Next scheduled run**

This confirms the schedule is valid and helps verify correctness.

---

## Saving the Configuration

Once the schedule is correct:

* Click **Save**
* Jenkins will automatically trigger the pipeline at the scheduled times
* No manual action is required

---

## Why This Is Not Ideal for CI

Despite being useful, **Build periodically is not ideal for continuous integration**:

* Developers may push **many changes in a single day**
* Feedback is delayed until the next scheduled run
* Running too frequently wastes resources when no changes exist

Jenkins itself warns that this trigger is **not recommended for continuously built software projects**.

---

## Key Takeaways

* **Build periodically** enables scheduled pipeline execution
* Jenkins uses **cron syntax**, not plain language
* The `H` symbol helps **spread load across jobs**
* Scheduled builds are good for:

    * Maintenance
    * Long-running jobs
    * Stability checks
* They are **not ideal** for fast feedback CI workflows

This trigger is important to understand, but it is **not the final solution** for automated CI/CD pipelines.
