## Lecture 61 – Parsing response data

### Goal of this lecture

There is still manual work in the pipeline:

* after deploying to **staging**
* someone manually opens the staging URL
* visually checks the website
* then approves deployment to production

The goal is to **fully automate this** by:

* running end-to-end tests against **staging**
* without manual inspection

The blocker is that the **staging URL is generated dynamically** by Netlify.

---

### The core problem: dynamic staging URLs

When deploying to staging, Netlify outputs logs similar to:

* a lot of unstructured text
* somewhere inside, a line like:

  ```
  Website Draft URL: https://random-string.netlify.app
  ```

Problems:

* the URL changes on every deployment
* it cannot be hardcoded
* Jenkins cannot reliably extract data from plain text logs

To automate tests, the pipeline must:

1. **extract the staging URL**
2. **pass it to the E2E test stage**

---

### Why JSON is required

Plain text logs:

* are easy for humans
* are difficult for machines to parse

What is needed instead:

* **structured output**
* predictable key/value format

Netlify supports this via **JSON output**.

---

### Enabling JSON output from Netlify CLI

The `netlify deploy` command supports a JSON flag.

Example:

```bash
netlify deploy --dir=build --json
```

This returns structured output like:

```json
{
  "site_name": "example-site",
  "deploy_id": "abc123",
  "deploy_url": "https://random-string.netlify.app"
}
```

This format is:

* human-readable
* machine-parsable

---

### Using Jenkins Replay to experiment safely

Instead of:

* editing the Jenkinsfile
* committing changes
* waiting for the full pipeline

Jenkins **Replay** is used to:

* modify pipeline steps temporarily
* skip unnecessary stages
* experiment quickly

Only the **deploy staging** logic is kept during experimentation.

---

### Redirecting command output to a file

To parse JSON, it must first be **saved to a file**.

Linux recap:

* command output → standard output
* redirection operator `>` saves output to a file

Example:

```bash
echo "hello" > hello.txt
```

Important:

* `>` overwrites the file
* `>>` appends to the file

---

### Saving Netlify JSON output to a file

In the pipeline, the Netlify deploy command is updated to:

```groovy
sh 'netlify deploy --dir=build --json > deploy.output'
```

Result:

* nothing appears in Jenkins logs
* JSON output is written to `deploy.output`
* the file exists in the workspace

---

### Verifying the output file in Jenkins

From Jenkins UI:

* open the build
* go to **Workspace**
* locate `deploy.output`
* file contains the JSON response from Netlify

This confirms the data is now accessible to other pipeline steps.

---

### Need for a JSON parsing tool

Having JSON in a file is not enough.

The pipeline must:

* read the file
* extract a specific property:

  ```
  deploy_url
  ```

For this, a JSON parser is required.

---

### Introducing `jq`

`jq` is a command-line tool for parsing JSON:

* extracts properties
* works well in pipelines

Because the project already uses Node.js:

* a Node wrapper is used instead of installing system jq

Tool used:

```
node-jq
```

---

### Installing node-jq in the pipeline

Instead of running multiple install commands, tools are installed together.

Example:

```groovy
sh 'npm install netlify-cli node-jq'
```

This installs:

* Netlify CLI
* node-jq

---

### Parsing JSON to extract the staging URL

Command structure:

```bash
node_modules/.bin/node-jq -r '.deploy_url' deploy.output
```

Explanation:

* `-r` → raw output (no quotes)
* `.deploy_url` → JSON property to extract
* `deploy.output` → file containing JSON

---

### Jenkins pipeline example: parsing the staging URL

```groovy
stage('Deploy Staging') {
  steps {
    sh 'npm install netlify-cli node-jq'

    sh 'netlify deploy --dir=build --json > deploy.output'

    sh 'node_modules/.bin/node-jq -r ".deploy_url" deploy.output'
  }
}
```

Pipeline result:

* staging deployment runs
* JSON response is saved
* staging URL is printed to logs

---

### Why this step is important

Although the visible result is “just a URL”:

* this enables **dynamic data flow** in the pipeline
* Jenkins can now:

    * capture runtime-generated values
    * pass them to later stages
    * fully automate staging E2E tests

This is a key building block for:

* removing manual approval checks
* running automated tests against staging

---

### Integrating changes into the main pipeline

Final changes to keep:

1. Install `node-jq`
2. Use `--json` with Netlify deploy
3. Redirect output to a file
4. Parse `deploy_url` from that file

These changes are committed to the Jenkinsfile so the full pipeline can use them.

---

### (Added by me – not mentioned in the lecture)

In production pipelines, the extracted staging URL is usually:

* stored in an environment variable
* written to a temporary file
* or exported for later stages
  so it can be reused by automated tests and approval messages.
