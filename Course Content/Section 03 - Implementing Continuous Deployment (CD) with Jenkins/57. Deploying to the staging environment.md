## Lecture 57 – Deploying to the staging environment

### Goal of a staging deployment

The objective is to deploy the application to a **staging environment** before production, using the **same deployment logic** as production but without exposing changes to the main live site.

Instead of manually creating a second Netlify site, the staging environment will be created **automatically** using the Netlify CLI.

---

### Possible manual approach (not used)

A staging environment could be created by:

* going to Netlify
* creating a new site manually
* naming it with a prefix such as `staging`
* deploying to it the same way as production

This approach works but is intentionally avoided because it does not add new learning value.

---

### Using Netlify CLI for staging deployment

Netlify CLI can:

* deploy the same build output
* automatically create a **temporary preview environment**
* provide a unique URL for testing

This preview deployment will be treated as the **staging environment**.

---

### Modifying the Jenkins pipeline

To deploy to staging, the production deploy stage is reused with minimal changes.

#### Step 1: Duplicate the deploy stage

* Copy the existing production deploy stage
* Paste it **above** the production deploy stage

#### Step 2: Rename stages

* Rename the copied stage to:

  ```
  deploy staging
  ```
* Rename the original stage to:

  ```
  deploy prod
  ```

This clearly separates staging and production deployments.

---

### Reviewing the existing deploy logic

The deploy stage already includes:

* installing Netlify CLI
* printing Netlify CLI version
* printing a deployment message
* checking login status using:

  ```
  netlify status
  ```
* deploying using:

  ```
  netlify deploy
  ```

These steps remain unchanged.

---

### Adjusting the deployment message

In the staging stage:

* change the message from:

  ```
  Deploying to production
  ```

  to:

  ```
  Deploying to staging
  ```

This improves log clarity.

---

### Key difference between staging and production deployment

The **only functional difference** is the `--prod` flag.

#### Production deployment

```
netlify deploy --dir=build --prod
```

#### Staging deployment

```
netlify deploy --dir=build
```

Behavior:

* when `--prod` is **present** → deploys to production
* when `--prod` is **absent** → Netlify creates a **temporary preview (draft) deployment**

This preview deployment is used as the staging environment.

---

### Running the pipeline

After committing the changes:

* the pipeline runs normally
* a new stage appears:

  ```
  deploy staging
  ```

The pipeline execution takes a few minutes.

---

### Inspecting staging deployment logs

In the Jenkins logs:

* expand the final `netlify deploy` command
* Netlify reports that it is deploying to a **draft URL**
* no errors or warnings are shown

Netlify outputs a **draft website URL**.

---

### Accessing the staging environment

Clicking the draft URL:

* opens a temporary version of the application
* shows the deployed build
* uses a different URL than production

URL characteristics:

* contains a random identifier
* publicly accessible
* not easily discoverable
* known only to the deployment owner

This makes it suitable for pre-production testing.

---

### Limitations of the current setup

At this stage:

* the pipeline stops if the staging deployment command fails
* basic deployment errors are caught

However:

* if the **wrong application** is deployed successfully
* the pipeline will still continue
* the problem may only be discovered later

This limitation will be addressed in upcoming lectures.

---

### (Added by me – not mentioned in the lecture)

In real CI/CD pipelines, staging deployments are usually followed by:

* automated staging end-to-end tests
* validation checks specific to the staging URL
  before allowing promotion to production.
