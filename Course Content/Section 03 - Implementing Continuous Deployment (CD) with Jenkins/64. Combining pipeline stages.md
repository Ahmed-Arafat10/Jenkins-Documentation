## Lecture 64 – Combining pipeline stages

### Motivation for combining stages

The pipeline has grown in length and complexity.
Some stages are logically connected:

* `Deploy Staging` → `Staging E2E`
* `Deploy Production` → `Prod E2E`

In both cases:

* deployment happens
* immediately after, tests validate that deployment
* if tests fail → deployment should be considered failed

Because:

* these stages depend on each other
* they cannot run in parallel
* they form one logical unit

It makes sense to **merge them into a single stage**.

Benefits:

* clearer pipeline structure
* fewer Docker containers started/stopped
* reduced resource usage
* improved readability in Blue Ocean view

---

# Merging Production Deploy + Production E2E

### Observing the Docker images

Originally:

* `Deploy Prod` uses Node 18 image
* `Prod E2E` uses Playwright Docker image

Important observation:

* Playwright Docker image already contains Node.js

Verification example:

```groovy
sh 'node --version'
```

If Node is available in the Playwright image:

* Netlify CLI can be installed there
* no need for a separate Node image

---

### Combined Production Stage

Instead of two stages:

```groovy
stage('Deploy Production') { ... }

stage('Prod E2E') { ... }
```

We create one:

```groovy
stage('Deploy Prod') {
  agent {
    docker {
      image 'mcr.microsoft.com/playwright:v1.40.0-focal'
      args '-u root'
    }
  }

  steps {
    sh 'node --version'

    sh 'npm install netlify-cli'

    sh '''
      netlify deploy --dir=build --prod
    '''

    // Optional delay if deployment propagation takes time
    // sh 'sleep 10'

    sh 'npx playwright test'
  }

  post {
    always {
      publishHTML(target: [
        reportDir: 'playwright-report',
        reportFiles: 'index.html',
        reportName: 'Prod E2E'
      ])
    }
  }
}
```

Result:

* single Docker container
* deploy + test in same runtime
* if either step fails → entire stage fails

---

### Note about deployment timing

Even if Netlify responds successfully:

* deployment may not be instantly ready
* a short delay might be required

Optional safeguard:

```groovy
sh 'sleep 5'
```

(Not added in lecture, but important to consider)

# Merging Staging Deploy + Staging E2E

Staging is slightly more complex because:

* it extracts a dynamic URL
* previously passed between stages
* used `script {}` and `env.STAGING_URL`

Now, since everything happens in one stage:

* cross-stage variable passing is no longer required
* the variable can exist only inside this stage

---

## Using Shell Variable Assignment with `$()`

In Linux:

```bash
MY_VAR=$(date)
echo $MY_VAR
```

Same concept applies in Jenkins shell steps.

---

### Combined Staging Stage

```groovy
stage('Deploy Staging') {
  agent {
    docker {
      image 'mcr.microsoft.com/playwright:v1.40.0-focal'
      args '-u root'
    }
  }

  steps {
    sh '''
      npm install netlify-cli node-jq

      netlify deploy --dir=build --json > deploy.output

      CI_ENVIRONMENT_URL=$(node_modules/.bin/node-jq -r '.deploy_url' deploy.output)

      echo "Staging URL: $CI_ENVIRONMENT_URL"

      npx playwright test
    '''
  }

  post {
    always {
      publishHTML(target: [
        reportDir: 'playwright-report',
        reportFiles: 'index.html',
        reportName: 'Staging E2E'
      ])
    }
  }
}
```

Important:

* no spaces around `=`
* `$()` executes the command
* value becomes available inside this shell session
* Playwright reads `CI_ENVIRONMENT_URL`

---

## Why the first attempt failed

Error observed:

```
Connection refused at localhost:3000
```

Reason:

* `CI_ENVIRONMENT_URL` was not properly set
* Playwright fallback logic used:

  ```
  localhost:3000
  ```

From `playwright.config.js`:

```js
baseURL: process.env.CI_ENVIRONMENT_URL || 'http://localhost:3000'
```

If the variable is not visible to the shell process:

* Playwright defaults to localhost
* tests fail

---

## Key Rule When Combining Stages

When using:

```bash
VAR=$(command)
```

The variable:

* exists only in that shell execution context
* must be used in the same `sh` block
* cannot be relied upon in a separate `sh` step

This is why the entire logic is placed inside **one multiline `sh` block**.

# Final Combined Structure Example

```groovy
pipeline {
  agent any

  stages {

    stage('Deploy Staging') {
      agent {
        docker {
          image 'mcr.microsoft.com/playwright:v1.40.0-focal'
          args '-u root'
        }
      }

      steps {
        sh '''
          npm install netlify-cli node-jq

          netlify deploy --dir=build --json > deploy.output

          CI_ENVIRONMENT_URL=$(node_modules/.bin/node-jq -r '.deploy_url' deploy.output)

          echo "Running tests against: $CI_ENVIRONMENT_URL"

          npx playwright test
        '''
      }

      post {
        always {
          publishHTML(target: [
            reportDir: 'playwright-report',
            reportFiles: 'index.html',
            reportName: 'Staging E2E'
          ])
        }
      }
    }

    stage('Approval') {
      steps {
        timeout(time: 15, unit: 'MINUTES') {
          input message: 'Approve deployment to production?',
                ok: 'Deploy to production'
        }
      }
    }

    stage('Deploy Prod') {
      agent {
        docker {
          image 'mcr.microsoft.com/playwright:v1.40.0-focal'
          args '-u root'
        }
      }

      steps {
        sh '''
          npm install netlify-cli
          netlify deploy --dir=build --prod
          npx playwright test
        '''
      }

      post {
        always {
          publishHTML(target: [
            reportDir: 'playwright-report',
            reportFiles: 'index.html',
            reportName: 'Prod E2E'
          ])
        }
      }
    }
  }
}
```

## Results of Combining Stages

* fewer Docker containers started
* fewer stage transitions
* cleaner Blue Ocean visualization
* deployment and validation treated as one atomic operation
* if E2E fails → stage fails immediately

---

## (Added by me – not mentioned in the lecture)

In large production systems, teams often combine deployment + validation into:

* a single stage
* or a single reusable shared library function

This improves:

* maintainability
* reusability
* consistency across pipelines

However, very large pipelines sometimes keep them separate for better observability and granular metrics.
