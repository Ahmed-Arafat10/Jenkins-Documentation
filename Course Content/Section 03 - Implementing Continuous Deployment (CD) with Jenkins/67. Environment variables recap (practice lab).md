## Lecture 67 – Environment Variables Recap (Practice Lab)

This lab focuses on working with **environment variables** in a CI/CD context.

---

# 1️⃣ Checking if an Environment Variable Is Defined

Variable to check:

```
version
```

---

## Method 1 – Using `echo`

```bash
echo $version
```

Behavior:

* If defined → prints its value
* If not defined → prints nothing (empty output)

If nothing is returned, the variable is:

* undefined
* or empty

---

## Method 2 – Using `env`

```bash
env
```

This lists all available environment variables.

To filter:

```bash
env | grep version
```

In real CI/CD pipelines:

* this list is usually very long
* contains many predefined variables
* includes build metadata, paths, credentials, etc.

---

# 2️⃣ Defining an Environment Variable

Using `export`:

```bash
export version=1.2.3
```

Now:

```bash
echo $version
```

Output:

```
1.2.3
```

And:

```bash
env | grep version
```

Shows:

```
version=1.2.3
```

---

# 3️⃣ Using the Variable Inside a Message

Required output:

```
building version 1.2.3
```

Correct command:

```bash
echo "building version $version"
```

Output:

```
building version 1.2.3
```

---

## ⚠ Important: Double vs Single Quotes

Correct (variable expands):

```bash
echo "building version $version"
```

Incorrect (variable not expanded):

```bash
echo 'building version $version'
```

Single quotes treat everything literally.

---

# 4️⃣ Creating a File with the Variable Content

Goal:

Create:

```
build/info.txt
```

Containing:

```
build version 1.2.3
```

---

## Step 1 – Create Directory

```bash
mkdir build
```

---

## Step 2 – Write File Using Redirection

```bash
echo "build version $version" > build/info.txt
```

---

## Step 3 – Verify Content

```bash
cat build/info.txt
```

Output:

```
build version 1.2.3
```

---

# 💡 Can It Be Done in One Command?

Yes.

```bash
mkdir -p build && echo "build version $version" > build/info.txt
```

Explanation:

* `mkdir -p build` → creates directory safely
* `&&` → runs next command only if first succeeds
* `>` → redirects output into file

---

# Applying This in Jenkins

Equivalent Jenkins stage:

```groovy
stage('Build Info File') {
  steps {
    sh '''
      export version=1.2.3
      mkdir -p build
      echo "build version $version" > build/info.txt
      cat build/info.txt
    '''
  }
}
```

---

# Key Takeaways

* Use `$variable` to reference environment variables
* Use `export` to define them in shell
* Use double quotes for variable expansion
* Use `>` to write to files
* Use `mkdir -p` to avoid directory errors
* Environment variables are heavily used in CI/CD pipelines

---

### (Added by me – not mentioned in the lecture)

In Jenkins pipelines, variables defined with `export` inside a `sh` block:

* only exist within that shell execution
* do not automatically persist across stages

To persist across stages in Jenkins:

* use `environment {}` block
* or `script { env.VAR = value }`
