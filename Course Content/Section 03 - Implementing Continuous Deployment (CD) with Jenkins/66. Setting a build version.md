## Lecture 66 – Setting a build version

### Objective

Two improvements are introduced:

1. Remove the manual approval step → move to **Continuous Deployment**
2. Ensure that the version deployed is exactly the version built

Problem being solved:
Previously, it was technically possible to:

* skip the build stage
* deploy an old artifact still present in the workspace

This creates a risk of deploying the wrong version.

Solution:

* Inject a **build-based application version**
* Validate it through automated tests

---

# Removing the Approval Stage

The approval stage is deleted from the pipeline:

```groovy
// Removed approval stage for Continuous Deployment
```

Now:

* Every successful pipeline run deploys automatically
* No manual gate

---

# Introducing a Dynamic Application Version

## Current State in `app.js`

Originally:

```javascript
Application Version 1
```

This is hardcoded.

Goal:
Replace it with a dynamic value from the pipeline.

---

## Defining a Build-Time Environment Variable

A new environment variable must be defined in Jenkins:

Name must be exactly:

```
REACT_APP_VERSION
```

Example (hardcoded test first):

```groovy
pipeline {
  agent any

  environment {
    REACT_APP_VERSION = "1.2.3"
  }

  stages {
    stage('Build') {
      steps {
        sh 'npm install'
        sh 'npm run build'
      }
    }
  }
}
```

Important:

* In React applications, variables prefixed with `REACT_APP_`
  are injected during build time.
* `process.env.REACT_APP_VERSION` is replaced during build.

---

## Updating `app.js`

Replace hardcoded version with:

```javascript
Application Version {process.env.REACT_APP_VERSION}
```

More precisely:

```javascript
Application Version {process.env.REACT_APP_VERSION}
```

At build time:

* this is replaced with the actual value from Jenkins.

---

# Test Validation

Inside `app.spec.js`, there is a test that checks:

If the variable is not defined:

* it defaults to `1`

Now:

* it should match the dynamically injected version

# Making the Version Increment Automatically

Hardcoding is not useful.

Instead, use Jenkins predefined variables.

Important predefined variable:

```
BUILD_ID
```

`BUILD_ID` equals the Jenkins build number.

Example:
If pipeline run is build #50,
then:

```
BUILD_ID = 50
```

---

# Setting Semantic Version Using BUILD_ID

Desired version format:

```
1.0.<build_number>
```

Example: `1.0.50`

Pipeline configuration:

⚠ Important: Must use double quotes for variable expansion.

Correct:

```groovy
environment {
  REACT_APP_VERSION = "1.0.${BUILD_ID}"
}
```

Incorrect (will not expand variable):

```groovy
environment {
  REACT_APP_VERSION = '1.0.${BUILD_ID}'
}
```

Single quotes prevent variable interpolation.

---

# Full Example Pipeline Snippet

```groovy
pipeline {
  agent any

  environment {
    REACT_APP_VERSION = "1.0.${BUILD_ID}"
  }

  stages {

    stage('Build') {
      steps {
        sh 'npm install'
        sh 'npm run build'
      }
    }

    stage('Deploy Production') {
      steps {
        sh 'netlify deploy --dir=build --prod'
        sh 'npx playwright test'
      }
    }
  }
}
```

---

# Result After Deployment

If current build number is 50:

Website displays:

```
Application Version 1.0.50
```

Next run (build #51):

```
Application Version 1.0.51
```

Each pipeline execution produces a new version.

---

# Why This Matters

Now the pipeline ensures:

* Build 51 produces version 1.0.51
* Tests expect 1.0.51
* Deployment must contain 1.0.51
* Any mismatch causes failure

This guarantees:

* build and deployment consistency
* no stale artifact deployment
* version traceability across environments

---

# Testing the Tests (Restart From Stage)

To verify correctness:

1. Go to a completed build (e.g., build #50)
2. Click **Restart from stage**
3. Choose `Deploy Prod`
4. This skips the build
5. Deploys old build (version 1.0.50)
6. Pipeline run is now build #51

Now:

* Jenkins expects version 1.0.51
* Production contains 1.0.50

Result:

* Playwright test fails
* Pipeline fails

Failure confirms:

* version mismatch detected
* safety mechanism works

# Example of Version Check in Test (Conceptual)

```javascript
test('should display correct version', async ({ page }) => {
  await expect(page.locator('#version'))
    .toHaveText(`Application Version ${process.env.REACT_APP_VERSION}`);
});
```

(Exact syntax may differ, but concept is validation against expected version)

# (Added by me – not mentioned in the lecture)

In real production systems, versioning often uses:

* Git commit SHA
* Git tag
* semantic version from `package.json`
* or a combination like:

  ```
  1.0.${BUILD_ID}-${GIT_COMMIT}
  ```

This allows:

* exact traceability
* rollback to specific commits
* correlation with monitoring and logs
