# 50. Managing Secrets in Jenkins

## Purpose of This Lecture

At this point in the pipeline:

* ✅ Netlify CLI is installed
* ✅ The Netlify **Site ID** is configured
* ❌ Jenkins is **not yet authorized** to deploy

This lecture explains **how Jenkins securely authenticates** with Netlify by using **tokens**, and how those secrets are **safely stored and managed in Jenkins**.

---

## Why Authorization Is Required

Even though Jenkins knows:

* *Which site* to deploy to (Site ID)
* *How* to deploy (Netlify CLI)

It still **cannot deploy** without authentication.

Why?

* Jenkins has no username
* Jenkins has no password
* Jenkins has no proof that it is allowed to deploy to your Netlify account

If deployment worked without authentication:

* Anyone could deploy to your site
* Your Netlify account would be compromised

So authentication is mandatory.

---

## Why Tokens Are Used Instead of Username & Password

Using a **token** is preferred because:

* Tokens are **more secure** than passwords
* Tokens can have **limited scope**
* Tokens can have **expiration dates**
* Tokens can be **revoked independently**
* Tokens **cannot be used to log in** to the UI

You can think of a token as:

> A temporary, restricted, machine-friendly password

CLI tools almost always use tokens.

---

## Generating a Netlify Access Token

### Steps in Netlify

1. Go to **Profile**
2. Open **User Settings**
3. Select **Applications**
4. Scroll to **Personal Access Tokens**
5. Click **New access token**

---

### Token Configuration

When creating the token:

* **Description**:
  Use something meaningful, for example:

  ```
  Jenkins
  ```

* **Expiration**:

    * Strongly recommended
    * Example: `60 days`
    * Avoid “no expiration” unless absolutely required

* Click **Generate token**

---

### Important: Copy the Token Immediately

* Netlify will show the token **only once**
* Copy it immediately
* You will **not be able to view it again**

> Each user must generate **their own token**
> Tokens shown in lectures or screenshots will not work

---

## Why Secrets Must NOT Be Stored in Jenkinsfile

❌ **Never do this**:

```groovy
environment {
    NETLIFY_TOKEN = "my-secret-token"
}
```

Reasons:

* Jenkinsfile is stored in Git
* Secrets would be committed to version control
* Anyone with repo access could steal them
* This is a **critical security violation**

---

## Jenkins Credential Management

Jenkins has a **dedicated secure storage** for secrets.

This is where secrets belong.

---

## Storing the Netlify Token in Jenkins

### Step-by-Step in Jenkins UI

1. Go to **Dashboard**
2. Select **Manage Jenkins**
3. Click **Credentials**
4. Choose:

    * **System**
    * **Global credentials**
5. Click **Add Credentials**

---

### Credential Configuration

* **Kind**:
  `Secret text`

* **Scope**:
  `Global` (default)

* **Secret**:
  Paste the Netlify access token

* **ID**:

  ```
  netlify-token
  ```

  (You will reference this ID later in the pipeline)

* Click **Create**

---

## After Storing the Credential

* The token is now:

    * Encrypted
    * Hidden
    * Not readable in plain text
* Jenkins can use it
* Humans cannot accidentally expose it

Even if someone opens the credential later:

* They **cannot see** the secret value
* They can only replace it

---

## Why This Is Secure

* Jenkins controls access to credentials
* Secrets are injected **only at runtime**
* Logs do not expose secrets
* Git repository remains clean

This is the **correct and professional way** to manage secrets in CI/CD.

---

## What This Enables Next

With this token securely stored:

* Jenkins can authenticate with Netlify CLI
* Netlify CLI can deploy the application
* Deployment can be fully automated
* No passwords are hardcoded
* Secrets are centrally managed

---

## Best Practice Highlight

**AI: notes**
In real production environments:

* Tokens should be rotated regularly
* Short expiration times are preferred
* Access should be limited to the minimum required
* Credential IDs should follow clear naming conventions

---

## Key Takeaways

* Jenkins must authenticate before deploying
* Tokens are safer than usernames and passwords
* Secrets must **never** be stored in Jenkinsfiles
* Jenkins Credentials store secrets securely
* `Secret text` is the correct credential type for API tokens
* The credential **ID** is what pipelines reference later

This completes the **security foundation** for automated deployment.
Next steps will use this stored secret to perform a real deployment from Jenkins.
