## Lecture 68 – Basic App Content Checking with `curl` (Practice Lab)

This lab focuses on verifying website content using Linux commands — especially useful in CI/CD pipelines for quick validation checks.

---

# 1️⃣ Fetch Website Content with `curl`

Basic command:

```bash
curl https://example.com
```

What this returns:

* The **raw HTML page source**
* Not the fully rendered page
* JavaScript is not executed

Important:

* You only receive static HTML
* Dynamically rendered content (via JS) will not appear

---

## ⚠ Common Mistake: HTTP vs HTTPS

Incorrect:

```bash
curl http://example.com
```

May return:

* Redirect
* Different content
* Unexpected response

Always verify:

* The protocol matches what the site expects
* The response content is correct before writing assertions

---

# 2️⃣ Check If Website Contains Specific Text

Goal:
Verify the page contains the word:

```
Valentino
```

Solution:

```bash
curl https://example.com | grep Valentino
```

Explanation:

* `|` (pipe) passes output of curl to grep
* `grep Valentino` searches for matching text
* If found → line containing the match is printed
* If not found → nothing printed

---

## Checking Exit Code

Linux stores the exit code of the last command in:

```bash
echo $?
```

* `0` → success (match found)
* `1` → no match
* Non-zero → error

In CI/CD:

* If grep does not find text → pipeline fails automatically
* Manual `echo $?` usually not needed

---

# 3️⃣ Silence Mode (Cleaner CI Output)

In pipelines, we usually:

* care about success/failure
* not full HTML output
* not curl progress meter

---

## Silence `curl`

Add the `-s` flag:

```bash
curl -s https://example.com
```

Removes:

* Download progress
* Extra output noise

⚠ Warning:
If the URL is wrong:

* errors may be hidden
* troubleshooting becomes harder

Best practice:

* Remove `-s` temporarily when debugging

---

## Silence `grep`

Add the `-q` (quiet) flag:

```bash
curl -s https://example.com | grep -q Valentino
```

Behavior:

* No output displayed
* Exit code still indicates success or failure

---

## Fully Silent Content Check

```bash
curl -s https://example.com | grep -q Valentino
echo $?
```

If:

* `0` → text exists
* `1` → text not found

---

# CI/CD Usage Example (Jenkins)

Basic verification stage:

```groovy
stage('Verify Content') {
  steps {
    sh '''
      curl -s https://example.com | grep -q Valentino
    '''
  }
}
```

If text is missing:

* `grep` exits with code 1
* Jenkins marks stage as failed

---

# Even Better: Fail Fast and Show Error

More robust approach:

```groovy
stage('Verify Content') {
  steps {
    sh '''
      if ! curl -s https://example.com | grep -q Valentino; then
        echo "Expected content not found!"
        exit 1
      fi
    '''
  }
}
```

---

# Why This Is Useful in CI/CD

This method allows:

* Simple health checks
* Smoke tests
* Basic deployment validation
* Quick verification without heavy test frameworks

Useful for:

* Checking version strings
* Verifying landing page text
* Validating API responses
* Confirming deployment success

# Advanced Tip (Added by me – not mentioned in the lecture)

For production-grade checks, consider:

```bash
curl -sf https://example.com
```

Flags:

* `-s` → silent
* `-f` → fail on HTTP errors (e.g., 404, 500)

This ensures:

* HTTP errors fail the pipeline
* silent mode still applies
* better CI safety

Example:

```groovy
stage('Health Check') {
  steps {
    sh '''
      curl -sf https://example.com | grep -q Valentino
    '''
  }
}
```

# Debugging Strategy

If command fails unexpectedly:

1. Remove `-s`
2. Remove `-q`
3. Run commands separately:

   ```bash
   curl https://example.com
   ```

   Then:

   ```bash
   curl https://example.com | grep Valentino
   ```

Break complex commands into smaller steps to isolate issues.

---

# Key Takeaways

* `curl` fetches static page source
* `grep` searches text
* `|` pipes output
* `-s` silences curl
* `-q` silences grep
* Exit code determines CI success/failure
* Always verify raw output before automating checks
