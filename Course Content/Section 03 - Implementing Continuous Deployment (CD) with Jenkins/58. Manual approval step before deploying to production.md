## Lecture 58 – Manual approval step before deploying to production

### Purpose of a manual approval step

The pipeline currently:

* deploys to **staging**
* then deploys automatically to **production**

In some cases, a **human decision** is required before production deployment:

* a manager or reviewer verifies the staging environment
* business approval is needed before going live

The requirement is to **pause the pipeline** without:

* editing the Jenkinsfile each time
* commenting out stages
* re-running jobs manually

Jenkins provides a built-in solution for this.

---

### Using a separate pipeline for experimentation

To understand the feature without impacting the main project:

* create a new pipeline job
* name it:

  ```
  learning
  ```
* choose **Pipeline**
* start from a **Hello World** template

This allows fast experimentation without waiting for long project pipelines.

---

### Minimal example pipeline (build → deploy)

```groovy
pipeline {
  agent any

  stages {
    stage('Build') {
      steps {
        echo 'building'
      }
    }

    stage('Deploy') {
      steps {
        echo 'deploying'
      }
    }
  }
}
```

This simple structure makes pipeline behavior easy to observe.

---

### Adding an approval stage

To pause execution between build and deploy:

* insert a new stage named `Approval`
* use the Jenkins `input` step

Conceptually:

* pipeline stops
* waits for user interaction
* continues or aborts based on the decision

---

### Generating the input step using Pipeline Syntax

The `input` step syntax can be tricky.

Jenkins provides **Pipeline Syntax** (Snippet Generator):

* open **Pipeline Syntax**
* select `input`
* provide a message, for example:

  ```
  Ready to deploy?
  ```
* generate the pipeline script

The generated snippet is copied into the pipeline.

---

### Pipeline with approval stage

```groovy
pipeline {
  agent any

  stages {
    stage('Build') {
      steps {
        echo 'building'
      }
    }

    stage('Approval') {
      steps {
        input message: 'Ready to deploy?'
      }
    }

    stage('Deploy') {
      steps {
        echo 'deploying'
      }
    }
  }
}
```

---

### Runtime behavior of the approval step

When the pipeline runs:

1. `Build` stage executes
2. `Approval` stage pauses execution

Jenkins UI shows:

* the approval message
* two options:

    * **Proceed**
    * **Abort**

If **Proceed** is clicked:

* pipeline continues to the deploy stage

---

### Customizing the approval dialog

The `input` step supports advanced options, such as customizing the OK button text.

Example:

```groovy
stage('Approval') {
  steps {
    input message: 'Ready to deploy?',
          ok: 'Yes, I am sure I want to deploy'
  }
}
```

This only changes the UI text; behavior remains the same.

---

### Aborting the pipeline

If **Abort** is selected:

* the pipeline stops immediately
* the job status becomes **ABORTED**
* it is **not marked as FAILED**
* the deploy stage does not run

Jenkins logs record:

* the approval prompt
* the user who approved or aborted

---

### Problem: pipeline waits forever

By default:

* Jenkins waits indefinitely at the `input` step
* the pipeline remains suspended
* executors may stay blocked

This is not desirable in real pipelines.

---

### Adding a timeout to the approval step

To prevent infinite waiting, Jenkins provides the `timeout` step.

Using Pipeline Syntax:

* select `timeout`
* choose a duration (for example, 1 hour)
* generate the script

---

### Approval with timeout

```groovy
stage('Approval') {
  steps {
    timeout(time: 1, unit: 'HOURS') {
      input message: 'Ready to deploy?',
            ok: 'Yes, I am sure I want to deploy'
    }
  }
}
```

Behavior:

* Jenkins waits for user input
* if no action occurs within the timeout
* the pipeline is automatically interrupted

---

### Reducing timeout duration for testing

For demonstration purposes, the timeout can be shortened.

Example (1 minute):

```groovy
stage('Approval') {
  steps {
    timeout(time: 1, unit: 'MINUTES') {
      input message: 'Ready to deploy?',
            ok: 'Yes, I am sure I want to deploy'
    }
  }
}
```

---

### Timeout readability best practice

Avoid unclear timeout definitions like:

```groovy
timeout(time: 1)
```

Recommended:

```groovy
timeout(time: 1, unit: 'MINUTES')
```

This makes the configuration self-explanatory when reviewed later.

---

### Timeout behavior at runtime

If no user action is taken:

* the approval stage remains idle
* after the timeout expires
* the pipeline is aborted automatically

Logs may show:

* a timeout interruption
* an error ID
* limited explanation

The important part:

* the pipeline does not hang
* the timeout works as intended

---

### Applying this to production deployments

This approval stage can be placed:

* after staging deployment
* before production deployment

Example placement:

```groovy
stage('Deploy Staging') {
  steps {
    echo 'deploying to staging'
  }
}

stage('Approval') {
  steps {
    timeout(time: 1, unit: 'HOURS') {
      input message: 'Approve deployment to production?',
            ok: 'Deploy to production'
    }
  }
}

stage('Deploy Production') {
  steps {
    echo 'deploying to production'
  }
}
```

In real-world Jenkins setups:

* approval steps are often restricted to specific users or groups
* approval messages usually include the **staging URL**
* approval is used only before irreversible actions, such as production deployments
