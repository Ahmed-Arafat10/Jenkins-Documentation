## Lecture 56 – Deployment Strategies: Why use a staging environment?

### Production environment definition

After the build and test stages, deployment is automated to Netlify.

The current website address represents the **production environment**:

* it hosts the latest version of the application
* it is publicly accessible
* it is visible to all users

Any change deployed here is immediately seen by customers.

---

### The problem with deploying directly to production

If a mistake occurs during deployment:

* production end-to-end tests may detect it
* but detection happens **after** deployment

At that point:

* users may already have seen a broken website
* this can negatively affect trust and credibility

This creates a risk where validation happens too late.

---

### The need to test deployment before production

A key question arises:

* Is there a way to test a deployment **before** it reaches production?

The solution is not theoretical or paradoxical.

The same deployment commands used for production can be applied to a **non-production environment**.

---

### Non-production environments

A non-production environment:

* is used internally
* is meant for development, testing, and validation
* mirrors production behavior without public exposure

One of the most common non-production environments is **staging**.

---

### Staging environment explained

A staging environment is:

* non-production
* usually non-public
* extremely similar to production

The goal is for staging to be:

* as close to production as possible
* ideally identical, except for public accessibility

The closer staging is to production, the more reliable its test results are.

---

### Infrastructure as Code

Staging environments are often:

* created automatically
* configured using automation

This practice is known as **infrastructure as code**:

* environments are defined through configuration
* manual setup is avoided
* consistency is enforced

This approach ensures staging and production environments remain nearly identical.

---

### Staging as a pre-production environment

Staging is considered a **pre-production** environment:

* it exists before the live environment
* deployments are validated here first

The workflow is:

1. deploy to staging
2. verify deployment behavior
3. troubleshoot safely if issues occur
4. deploy to production only after success

---

### Benefits of using a staging environment

If deployment fails in staging:

* production remains unaffected
* users never see the issue
* there is time to debug and fix problems

If deployment succeeds in staging:

* there is a very high likelihood
* that the same deployment will succeed in production

---

### (Added by me – not mentioned in the lecture)

In mature CI/CD pipelines, staging environments are often combined with:

* automated smoke tests
* integration tests
* post-deployment checks
  to further increase confidence before promoting a release to production.
