# 51. Using Credentials in the Pipeline

## Goal of This Lecture

The goal here is to **connect Jenkins securely to Netlify** so that Jenkins can authenticate and perform actions (like deployment) using the **Netlify CLI**.

At this point, we already have:

* ✅ Netlify CLI installed
* ✅ Netlify **Site ID** configured as an environment variable
* ✅ Netlify **access token** securely stored in Jenkins Credentials

What’s missing is **connecting that credential to the pipeline** in a way Netlify CLI understands.

---

## How Netlify CLI Authenticates

Netlify CLI does **not** ask for username/password interactively in CI.

Instead, it looks for a **specific environment variable**:

```
NETLIFY_AUTH_TOKEN
```

Important points:

* The variable name **must be exactly this**
* Netlify CLI automatically checks for it
* If found, Netlify CLI uses it for authentication
* If missing → authentication fails

This is similar to how Netlify CLI also auto-detects the **Site ID** via an environment variable.

---

## Why We Cannot Hardcode the Token

❌ **Wrong approach (never do this):**

```groovy
environment {
    NETLIFY_AUTH_TOKEN = "my-secret-token"
}
```

Reasons:

* Jenkinsfile is stored in Git
* Token would be exposed to anyone with repo access
* Token could be leaked in logs
* This breaks all CI/CD security best practices

---

## Correct Way: Using Jenkins Credentials Helper

Jenkins provides a **credentials helper** that safely injects secrets at runtime.

### Syntax Overview

```groovy
environment {
    NETLIFY_AUTH_TOKEN = credentials('credential-id')
}
```

* `credentials(...)` pulls the secret securely
* Value is **never exposed**
* Value is **masked in logs**
* Value exists only during pipeline execution

---

## Using the Netlify Token Credential

Earlier, we stored the Netlify token in Jenkins with this ID:

```
netlify-token
```

To use it in the pipeline:

```groovy
environment {
    NETLIFY_SITE_ID   = "your-site-id"
    NETLIFY_AUTH_TOKEN = credentials('netlify-token')
}
```

Key notes:

* `NETLIFY_AUTH_TOKEN` → required by Netlify CLI
* `credentials('netlify-token')` → pulls secret from Jenkins
* Token value is **not visible** in Jenkinsfile

---

## Verifying Authentication with Netlify CLI

Before deploying, we want to **verify authentication works**.

Netlify CLI provides a command for this:

```
netlify status
```

### Why `netlify status`?

* Does **not** deploy anything
* Prints authentication & site info
* Confirms:

    * Token is valid
    * Site ID is correct
    * Jenkins can talk to Netlify

---

## Adding the Verification Command

Inside the **deploy stage**, replace the version check with:

```bash
netlify status
```

This command will:

* Authenticate using `NETLIFY_AUTH_TOKEN`
* Resolve site using `NETLIFY_SITE_ID`
* Print user + site information

---

## Expected Successful Output

If authentication is working, logs will show:

* Netlify CLI version
* Logged-in Netlify user
* Site name
* Site URL
* Site ID

Example (conceptually):

```
Current Netlify User: your-email@example.com
Site Name: your-site
Site URL: https://your-site.netlify.app
Site ID: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
```

Seeing this means:

✅ Jenkins credentials are correct
✅ Environment variables are set correctly
✅ Netlify CLI authentication is successful

---

## If It Does NOT Work

If `netlify status` fails:

Check the following carefully:

1. Credential ID spelling

    * Must match **exactly** (`netlify-token`)
2. Token value

    * No extra spaces
    * Correct token copied from Netlify
3. Environment variable name

    * Must be `NETLIFY_AUTH_TOKEN`
4. Token expiration

    * Token may have expired
5. Jenkinsfile committed & pushed

    * Jenkins only sees committed changes
