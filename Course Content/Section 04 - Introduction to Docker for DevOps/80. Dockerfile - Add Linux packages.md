## Lecture 80 – Dockerfile: Add Linux Packages

### Objective

Instead of:

* Running Docker containers as `root`
* Installing `node-jq` via `npm`

We will:

* Install `jq` directly as a Linux package
* Use the system package manager (`apt`)
* Improve Docker image design
* Avoid unnecessary Node-based wrappers

---

# Why Install jq via Linux Instead of npm?

Previously:

```dockerfile
RUN npm install -g netlify-cli node-jq
```

But:

* `node-jq` is only a wrapper around `jq`
* `jq` is originally a Linux tool
* We don’t need Node.js to use `jq`

If our base image:

* does not include Node
* or we want a cleaner system-level tool

Then it’s better to install `jq` using the Linux package manager.

---

# Step 1 – Identify the Linux Distribution

Our base image ends with:

```
jammy
```

"Jammy" refers to:

* Ubuntu 22.04 (Jammy Jellyfish)

Ubuntu uses:

```
apt
```

as the package manager.

So we use:

```bash
apt update
apt install jq
```

---

# Step 2 – Modify the Dockerfile

Original:

```dockerfile
FROM mcr.microsoft.com/playwright:v1.40.0-jammy

RUN npm install -g netlify-cli node-jq
```

Updated version:

```dockerfile
FROM mcr.microsoft.com/playwright:v1.40.0-jammy

RUN npm install -g netlify-cli

RUN apt update
RUN apt install -y jq
```

We removed:

```
node-jq
```

Because now we install:

```
jq
```

as a Linux package.

---

# Step 3 – Update Jenkinsfile

Previously:

```bash
node-jq -r '.deploy_url' deploy.output
```

Now replace with:

```bash
jq -r '.deploy_url' deploy.output
```

No more `node-jq`.

---

# Step 4 – Rebuild the Docker Image

Important:

The nightly Docker build job:

* builds the image
* does NOT automatically rebuild when Dockerfile changes

So we must manually trigger:

```
nightly-build-docker-image
```

Once rebuilt, the updated image contains:

* netlify-cli (global npm)
* jq (system package)

---

# Important Docker Layer Caching Optimization

There is a subtle issue here:

If you write:

```dockerfile
RUN apt update
RUN apt install -y jq
```

Docker may cache `apt update` separately.

If the cache is reused:

* package lists may become outdated
* installation may fail later

Better approach:

Combine both commands into one layer.

---

# Correct Optimized Dockerfile

```dockerfile
FROM mcr.microsoft.com/playwright:v1.40.0-jammy

RUN npm install -g netlify-cli

RUN apt update && apt install -y jq
```

Why?

Because:

* `apt update` and `apt install` run together
* They are executed in the same layer
* Avoid stale cache issues
* More reliable builds

---

# Why Use `&&`?

In Linux:

```bash
command1 && command2
```

Means:

* Run command2 only if command1 succeeds
* Both executed in same shell session
* Single Docker layer

---

# Full Clean Version (Best Practice)

```dockerfile
FROM mcr.microsoft.com/playwright:v1.40.0-jammy

RUN npm install -g netlify-cli

RUN apt update && \
    apt install -y jq && \
    apt clean && \
    rm -rf /var/lib/apt/lists/*
```

(Added by me – not mentioned in the lecture)

Why clean?

* Reduces image size
* Removes cached package lists
* More production-ready image

# Updated Jenkins Usage Example

```groovy

}
```
