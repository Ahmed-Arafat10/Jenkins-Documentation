## Lecture 74 – Building a Docker Image

### Problem with the Current Setup

In the `deploy staging` and `deploy production` stages, the pipeline repeatedly runs:

```bash
npm install netlify-cli node-jq
```

This causes:

* Extra download time on every pipeline run
* Repeated installation of the same tools
* Slower CI execution

Instead of installing tools every time, we can:

* Build a custom Docker image
* Pre-install required tools inside it
* Reuse the image in the pipeline

This improves performance and efficiency.

---

# When Do We Need a Custom Docker Image?

We build a custom Docker image when:

* We need additional tools not available in the base image
* We want faster pipelines
* We want consistent environments
* We want to avoid repeated installations

---

# Step 1 – Create a `Dockerfile`

A **Dockerfile** is a text file that contains instructions for building a Docker image.

It acts like a recipe:

* Start from a base image
* Install tools
* Configure environment

---

## Location of Dockerfile

The Dockerfile must be placed:

* In the **root of the project**
* Same level as `Jenkinsfile`

Not inside `src/`.

Correct structure:

```
project-root/
├── Dockerfile
├── Jenkinsfile
├── package.json
└── src/
```

---

# Step 2 – Define the Base Image

We start from an existing working image:

```dockerfile
FROM mcr.microsoft.com/playwright:v1.40.0-focal
```

Key idea:

* We never build from scratch
* We extend an existing image

---

# Step 3 – Install Required Tools

We need:

* Netlify CLI
* node-jq

Because this is not a project directory, we install globally:

```dockerfile
RUN npm install -g netlify-cli node-jq
```

* `RUN` executes commands during image build
* `-g` installs globally

---

## Complete Minimal Dockerfile

```dockerfile
FROM mcr.microsoft.com/playwright:v1.40.0-focal

RUN npm install -g netlify-cli node-jq
```

That’s enough for this use case.

---

# Step 4 – Build the Docker Image in Jenkins

Add a new stage to the pipeline:

```groovy
stage('Docker') {
  steps {
    sh 'docker build -t my-playwright .'
  }
}
```

Explanation:

* `docker build` → builds image
* `-t my-playwright` → tags image
* `.` → current directory (where Dockerfile exists)

Important:

* The dot means: “Look in this directory for Dockerfile”

---

# What Happens During Docker Build?

When the stage runs:

1. Docker reads `Dockerfile`

2. Executes:

   ```
   FROM mcr.microsoft.com/playwright:v1.40.0-focal
   ```

3. Executes:

   ```
   RUN npm install -g netlify-cli node-jq
   ```

4. Creates a new image:

   ```
   my-playwright
   ```

---

# Using the Custom Image in Pipeline

Instead of:

```groovy
agent {
  docker {
    image 'mcr.microsoft.com/playwright:v1.40.0-focal'
  }
}
```

We now use:

```groovy
agent {
  docker {
    image 'my-playwright'
  }
}
```

Now:

* Netlify CLI is already installed
* node-jq is already installed
* No need for `npm install` in every run

---

# Example Updated Stage

```groovy
stage('Deploy Staging') {
  agent {
    docker {
      image 'my-playwright'
      args '-u root'
    }
  }

  steps {
    sh '''
      netlify deploy --dir=build --json > deploy.output
      CI_ENVIRONMENT_URL=$(node-jq -r '.deploy_url' deploy.output)
      npx playwright test
    '''
  }
}
```

---

# Why This Is Faster

Before:

* Pull base image
* Install tools
* Run commands

After:

* Pull prebuilt image
* Tools already installed
* Run commands immediately

The image build step is done once, not every time.

---

# Key Dockerfile Instructions Used

### `FROM`

Defines the base image.

```dockerfile
FROM image-name
```

### `RUN`

Executes commands during build.

```dockerfile
RUN command
```

---

# What Docker Actually Does

Each instruction creates a layer:

* Base image layer
* Tool installation layer
* Final exported image

These layers are cached, so rebuilds are fast unless changes occur.

---

# Important Concept

Everything installed in Dockerfile:

* Happens at **image build time**
* Not at container runtime
* Becomes permanently part of the image

---

# Final Result

We now have:

* A reusable custom Docker image
* Pre-installed Netlify CLI
* Pre-installed node-jq
* Faster pipeline
* Cleaner stages

# (Added by me – not mentioned in the lecture)

In production environments, best practice is to:

* Tag images with version:

  ```
  my-playwright:1.0.0
  ```
* Push to a Docker registry:

  ```
  docker push my-registry/my-playwright:1.0.0
  ```
* Avoid relying only on `latest`
* Scan image for vulnerabilities

This makes Docker image management production-ready and traceable.
