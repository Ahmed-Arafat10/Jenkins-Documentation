## Lecture 78 – Running a Nightly Docker Image Build Job

### Objective

Optimize the main pipeline by:

* Removing the Docker image build stage from the daily pipeline
* Building the Docker image once per day (during the night)
* Reusing the prebuilt image during the day

Why?

Building Docker images inside the same CI pipeline:

* increases execution time
* consumes unnecessary resources
* slows down developer feedback loop

Better approach:

* Separate image building into a scheduled job
* Use that image in the main pipeline

---

# Step 1 – Create a Separate Jenkinsfile for Nightly Build

At the root of the repository:

Create a new file:

```
Jenkinsfile-nightly
```

This file will contain only the Docker build logic.

---

## Example `Jenkinsfile-nightly`

```groovy
pipeline {
  agent any

  stages {
    stage('Build Docker Image') {
      steps {
        sh '''
          docker build -t my-app:latest .
        '''
      }
    }
  }
}
```

Key idea:

* Only build the Docker image
* No tests
* No deploy
* No extra stages

---

# Step 2 – Remove Docker Build Stage from Main Pipeline

Previously, the main pipeline had:

```groovy
stage('Build Docker Image') {
  steps {
    sh 'docker build -t my-app:latest .'
  }
}
```

This stage is now removed.

The main pipeline becomes faster because:

* It no longer builds images
* It only uses prebuilt images

---

# Step 3 – Create a New Jenkins Job

In Jenkins:

1. Go to **Dashboard**
2. Click **New Item**
3. Use **Copy From**
4. Copy the existing pipeline job
5. Name it:

```
nightly-build-docker-image
```

Select:

* Pipeline type

---

# Step 4 – Configure the Nightly Job

Inside job configuration:

### Change Script Path

Default:

```
Jenkinsfile
```

Change to:

```
Jenkinsfile-nightly
```

This tells Jenkins:

* Use the nightly pipeline file
* Not the main one

---

# Step 5 – Schedule the Nightly Build

Enable:

```
Build periodically
```

Example schedule (once per day at 2:00 AM):

```
H 2 * * *
```

Cron syntax meaning:

```
MIN HOUR DAY MONTH WEEKDAY
```

This ensures:

* Docker image is rebuilt every night
* Latest dependencies are included
* Fresh base image updates are pulled

---

# Example Nightly Pipeline with Schedule (Defined in Jenkinsfile)

Alternative approach (pipeline as code):

```groovy
pipeline {
  agent any

  triggers {
    cron('H 2 * * *')
  }

  stages {
    stage('Build Docker Image') {
      steps {
        sh '''
          docker build -t my-app:latest .
        '''
      }
    }
  }
}
```

This embeds scheduling into version-controlled pipeline code.

---

# Important Note About Local Jenkins

If Jenkins runs on:

* Your local machine
* And your computer is turned off at night

Then:

* The scheduled job will not run

In production:

* Jenkins typically runs on a server
* Scheduled builds run regardless of user presence

---

# Final Main Pipeline (Optimized)

Now the main pipeline:

* Does not build Docker images
* Uses existing image
* Executes faster

Example:

```groovy
pipeline {
  agent {
    docker {
      image 'my-app:latest'
    }
  }

  stages {
    stage('Build') {
      steps {
        echo 'Building application'
      }
    }

    stage('Test') {
      steps {
        echo 'Running tests'
      }
    }

    stage('Deploy') {
      steps {
        echo 'Deploying'
      }
    }
  }
}
```

# Benefits of This Design

### 1️⃣ Faster Developer Pipeline

* No Docker build during normal runs
* Shorter feedback cycle

### 2️⃣ Resource Optimization

* Image built once per day
* Not rebuilt on every commit

### 3️⃣ Cleaner Separation of Concerns

* Nightly job → infrastructure preparation
* Main job → CI/CD logic

### 4️⃣ Better Scalability

In large organizations:

* Base images built nightly
* Security updates applied automatically
* Dependency refresh scheduled

# (Added by me – not mentioned in the lecture)

In mature CI/CD systems, nightly Docker jobs often:

* Tag images with date:

  ```
  my-app:2026-02-15
  ```
* Push to Docker registry:

  ```
  docker push my-registry/my-app:latest
  ```
* Run vulnerability scans (e.g., Trivy, Grype)
* Fail if security issues are detected

This turns nightly builds into:

* Security refresh jobs
* Dependency update jobs
* Image hardening workflows
