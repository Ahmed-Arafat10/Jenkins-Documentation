# 35. Publishing a JUnit Test Report

## 1. What is a JUnit Test Report?

* **Definition:**
  A JUnit report is a file generated by the **JUnit testing framework** in Java projects, typically in **XML format**.
* **Contents of the Report:**

    * Pass/Fail status of tests
    * Execution time
    * Error messages
    * Additional test statistics
* **Importance in CI/CD:**

    * Allows automatic monitoring of test executions
    * Tracks which tests fail and why
    * Provides trend analysis over time
    * More convenient than parsing console logs manually

---

## 2. Applicability Beyond Java

* Our project is **not Java-based** and does **not use JUnit** directly.
* However, most testing frameworks (including JavaScript ones) support exporting results in **JUnit XML format**.
* In this course:

    * Running `npm test` generates results in a **`test-results/`** directory.
    * Inside it, we find a file: **`junit.xml`**.
    * This file holds execution details (e.g., 1 test passed, execution time).

**AI: Notes** → This portability is why JUnit XML has become a **de facto standard** for CI/CD pipelines across many programming languages.

---

## 3. Why Not Just Read the XML?

* JUnit XML is **not human-friendly**; it’s structured for machines.
* Jenkins can parse and visualize it:

    * Consolidates test outcomes into dashboards.
    * Provides historical trends and clickable drill-downs.

---

## 4. Publishing JUnit Reports in Jenkins

### Step 1: Modify the `Jenkinsfile`

* Reports should be published **regardless of build success/failure**.
* Use a `post` block with `always`:

```groovy
pipeline {
    agent any
    stages {
        // your stages here
    }
    post {
        always {
            junit 'test-results/junit.xml'
        }
    }
}
```

**Key Points:**

* `junit` directive tells Jenkins where to find the XML file.
* Path must match exactly (case-sensitive, spelling-sensitive).
* If the file isn’t found, Jenkins will throw an error.

---

## 5. Running the Pipeline

* After committing and pushing the updated `Jenkinsfile`:

    * Run the pipeline.
    * Observe the **new post action stage** → this is where Jenkins publishes the JUnit report.

**Possible Warning:**

* You may see:

  ```
  [Checks API] No suitable checks publisher found.
  ```
* **This is not an error.** It can be safely ignored.

---

## 6. Test Result Trend

* After running the pipeline **multiple times** and refreshing Jenkins:

    * A **"Test Result Trend" graph** will appear on the job page.
    * This shows test execution history:

        * If tests fail → graph trends down.
        * If more tests pass → graph trends up.
* Graph is **clickable**:

    * Provides details per build.
    * Lists test cases with pass/fail status.
    * Helps identify which specific test failed.

---

## 7. Benefits of JUnit Report Integration

* Centralized view of test health.
* Historical tracking of test stability.
* Quick identification of failing tests.
* Enhances team visibility in CI/CD pipelines.

---

✅ **Summary:**
JUnit reports, even when not using JUnit directly, provide a **universal, machine-readable testing format**. Jenkins can consume these reports and display them as user-friendly dashboards with trends over time. Integrating `junit` in the `post { always { ... } }` block ensures visibility of test results in every pipeline execution.
