## 🔁 Jenkins Architecture with Docker (Behind the Scenes)

### 1. Controller & Agent

* **Controller (Jenkins server):**

    * Manages the pipeline logic.
    * Knows *what* to run but does **not** execute stages.
* **Agent (Node):**

    * Executes the actual build steps.
    * Must have **Docker installed** to run containerized stages.

---

### 2. Workflow of a Docker-based Stage

1. **Repository cloning**

    * Controller clones the Git repo into the **workspace**.
    * Workspace = shared directory used by Jenkins for all files.

2. **Agent prepares Docker container**

    * Agent runs:

      ```bash
      docker run --rm -v /jenkins/workspace:/workspace node:18-alpine ...
      ```
    * The **workspace is mounted** inside the container.
    * Ensures files created/modified in the container are available after the container stops.

3. **Execution inside the container**

    * Container has the exact tools you need (e.g., NodeJS runtime).
    * Steps like `npm ci` and `npm run build` run *inside* this container.

4. **Container termination**

    * When stage finishes:

        * `docker stop` is called.
        * Temporary container removed.
        * Volumes cleaned up.
    * Workspace persists on the host (outside the container).

---

### 3. Why Docker is Important in Jenkins

* ✅ Consistency — same environment everywhere (no “works on my machine”).
* ✅ Isolation — each stage has its own clean runtime.
* ✅ Reproducibility — pinned images guarantee same tools/versions.
* ⚠️ Files must be **mapped to workspace** → otherwise all build outputs vanish once container stops.

---

### 4. Logs You’ll See in Jenkins

* **Step 1:** Cloning repo → `[Git] Checking out ...`
* **Step 2:** Container start → `docker run ...`
* **Step 3:** Commands run → `npm ci`, `npm run build`, etc.
* **Step 4:** Container stop → `docker stop ...`, cleanup

---

👉 So in short: **Controller orchestrates → Agent runs Docker → Workspace is the bridge between Jenkins and the container → Container dies but workspace lives.**
