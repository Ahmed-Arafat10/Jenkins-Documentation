## 23. Introduction to Continuous Integration (CI)

### 🔹 What is CI?

* **Continuous Integration (CI)** = practice of frequently integrating code changes into a shared repository.
* Each integration is **automatically built and tested**.
* Core principle: **catch integration issues early** by merging and testing continuously.

---

### 🔹 Why CI?

* In modern projects, multiple developers contribute simultaneously.
* Without CI → merging once per week/month → high risk of **integration conflicts** and costly fixes.
* With CI → merge **several times per day (or hour)** → immediate feedback, faster fixes, reduced risks.

---

### 🔹 Workflow

1. Developer writes code.
2. Pushes to **Git** (version control).
3. CI system (e.g., **Jenkins**) triggers:

    * Build → Test → Verify.
4. If successful → code gets integrated into main project.
5. If failing → developer is notified to fix issues immediately.

---

### 🔹 Tools for CI

* **Git** → tracks changes, enables collaboration.
* **Jenkins** → automates building, testing, and integration steps.

👉 In this section, we will use **Git + Jenkins** together to implement CI.

---

### 🔹 Best Practices & Tips

* **Integrate frequently** (several times per day).
* **Pay attention to errors**:

    * Error messages are designed to be helpful.
    * Understand what you did and why the error occurred.
* **Troubleshooting resources**:

    * Troubleshooting lectures in the course.
    * Q\&A section for asking/answering common issues.

---

### 🔹 Recap

* CI = practice of continuously merging and testing code changes.
* Multiple developers can safely work on the same repository.
* Frequent integration reduces conflicts and ensures new versions of the product are always deployable.
* In upcoming lessons → CI will be **implemented hands-on** with Git & Jenkins.

### (AI: Notes)

* In real-world CI setups, additional practices often complement Jenkins:

    * **Automated unit tests** for each commit.
    * **Code quality checks** (linting, static analysis).
    * **Branching strategy** (e.g., GitFlow, trunk-based development).
    * **Notifications** (Slack, email) when builds fail.
* These make CI more reliable and keep the team aligned.
