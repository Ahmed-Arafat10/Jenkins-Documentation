## Lecture 104 — AWS CLI `wait` Command

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

## The Problem with Fast CLI Commands

When running the deployment stage in the Jenkins pipeline, most AWS CLI commands execute **very quickly**.

For example:

* Registering a task definition takes only a few seconds.
* Updating an ECS service also returns almost immediately.

However, although the CLI command finishes quickly, the **actual deployment process continues in the background**.

When updating a service in ECS:

1. A **new task starts**
2. ECS waits until it becomes healthy
3. The **old task stops**

This process can take **several seconds or even a few minutes**.

During this time, ECS temporarily runs **two tasks simultaneously**.

Example state during deployment:

```id="hvq7vl"
2 / 1 tasks running
```

After the deployment finishes:

```id="8q89hg"
1 / 1 tasks running
```

This delay creates an issue for CI/CD pipelines.

---

## Why This Is a Problem for Pipelines

If the pipeline immediately continues to the next stage after updating the service, it might:

* Run tests before the new container is ready
* Attempt to access an endpoint that is not yet available
* Produce unreliable results

One possible workaround could be adding a delay:

```bash id="j2np64"
sleep 60
```

However, this approach is unreliable because:

* Some deployments may finish faster
* Others may take longer
* The pipeline becomes inefficient or unstable

Instead, the pipeline should **wait until the deployment is actually finished**.

---

## The AWS CLI `wait` Command

The AWS CLI provides a special command designed exactly for this purpose.

The command is:

```bash id="njqwrv"
aws ecs wait services-stable
```

This command waits until an ECS service reaches a **stable state**.

A service is considered **stable** when:

* The desired number of tasks are running
* All tasks are healthy
* No deployment is in progress

This command continuously checks the service status until it becomes stable.

---

## How the Wait Command Works

The command requires two parameters:

1. **Cluster name**
2. **Service name**

Example:

```bash id="b5alpa"
aws ecs wait services-stable \
--cluster my-cluster \
--services my-service
```

Important detail:

The argument is **`--services` (plural)**, even if only one service is provided.

---

## Adding the Wait Command to the Jenkins Pipeline

The wait command should run **after updating the ECS service**.

Pipeline sequence:

1. Register task definition
2. Update service
3. Wait until service becomes stable

Example pipeline commands:

```bash id="stq14g"
aws ecs register-task-definition \
--cli-input-json file://aws/task-definition-prod.json

aws ecs update-service \
--cluster learn-jenkins-app-cluster-prod \
--service learn-jenkins-app-service-prod \
--task-definition learn-jenkins-app-task-definition-prod:$latest_td_revision

aws ecs wait services-stable \
--cluster learn-jenkins-app-cluster-prod \
--services learn-jenkins-app-service-prod
```

---

## What Happens During the Wait Stage

When the pipeline reaches this command, the terminal appears **inactive**.

However, the AWS CLI is actively checking the deployment state in the background.

The command repeatedly queries ECS until:

* the service finishes deploying
* the tasks reach a stable state

If the deployment fails or takes too long, the command eventually **times out**.

---

## Pipeline Execution Time Change

Before using the wait command, the deployment stage took roughly:

```id="0tt9h5"
~24 seconds
```

After introducing the wait command, the stage may take:

```id="nfyrz0"
~4 minutes
```

Although the pipeline takes longer, it becomes **much more reliable**.

The next stages of the pipeline can safely run tests against a fully deployed application.

---

## Improving the Jenkinsfile with Environment Variables

After implementing the deployment commands, the Jenkinsfile contains repeated configuration values.

Examples of repeated values:

* Cluster name
* Service name
* Task definition name

Hardcoding these values inside commands makes the pipeline **hard to maintain**.

Instead, environment variables should be introduced.

---

## Defining Environment Variables

Example variables defined inside the Jenkins pipeline:

```groovy id="vgtm8q"
environment {
    AWS_ECS_CLUSTER = "learn-jenkins-app-cluster-prod"
    AWS_ECS_SERVICE_PROD = "learn-jenkins-app-service-prod"
    AWS_ECS_TD_PROD = "learn-jenkins-app-task-definition-prod"
}
```

Explanation:

| Variable               | Purpose                           |
| ---------------------- | --------------------------------- |
| `AWS_ECS_CLUSTER`      | ECS cluster name                  |
| `AWS_ECS_SERVICE_PROD` | Production service name           |
| `AWS_ECS_TD_PROD`      | Production task definition family |

These are **custom environment variables**, not built-in AWS variables.

---

## Using the Variables in Commands

Once defined, the commands become cleaner.

Example:

```bash id="ogui7i"
aws ecs update-service \
--cluster $AWS_ECS_CLUSTER \
--service $AWS_ECS_SERVICE_PROD \
--task-definition $AWS_ECS_TD_PROD:$latest_td_revision
```

Wait command:

```bash id="9whmwy"
aws ecs wait services-stable \
--cluster $AWS_ECS_CLUSTER \
--services $AWS_ECS_SERVICE_PROD
```

This eliminates hardcoded values from the commands.

---

## Benefits of Using Environment Variables

Advantages include:

### Easier Maintenance

Configuration changes occur in **one place only**.

### Reusability

The same pipeline can deploy to:

* staging
* production
* testing

by simply changing environment variables.

### Cleaner Commands

Deployment commands become easier to read and maintain.

---

## Final Jenkins Deployment Flow

The improved deployment stage now performs the following sequence:

```id="h3d00h"
Install jq
       │
       ▼
Register Task Definition
       │
       ▼
Extract Revision
       │
       ▼
Update ECS Service
       │
       ▼
Wait for Service Stability
```

This ensures that:

* The correct task revision is deployed
* The ECS service finishes deploying
* The pipeline continues only when the deployment is stable

---

## Key Outcome of This Lecture

By introducing the **AWS CLI wait command**, the CI/CD pipeline now:

* Handles asynchronous deployments correctly
* Avoids unreliable delays like `sleep`
* Ensures the environment is ready before further pipeline stages execute

This makes the deployment pipeline **much more reliable and production-ready**.

### Updated Jenkins
````
pipeline {
    agent any
    environment {
        AWS_DEFAULT_REGION = 'us-east-1'
        AWS_ECS_CLUSTER = 'LearnJenkinsAppClusterProd'
        AWS_ECS_SERVICE_PROD = 'learn-jenkins-app-task-definition-prod-service-vd0czv2v'
        AWS_ECS_TD_PROD = 'learn-jenkins-app-task-definition-prod'
    }
    stages {
        stage('Deploy to AWS ECS') {
            agent {
                docker {
                    image 'amazon/aws-cli'
                    reuseNode true
                    args "-u root --entrypoint ''"
                }
            }
            steps {
               withCredentials([usernamePassword(credentialsId: 'aws-ecs', passwordVariable: 'AWS_SECRET_ACCESS_KEY', usernameVariable: 'AWS_ACCESS_KEY_ID')]) {
                    sh '''
                    yum install -y jq
                    aws --version
                    LATEST_TD_REVISION=$(aws ecs register-task-definition --cli-input-json file://aws/task-definition-prod.json | jq '.taskDefinition.revision')
                    
                    aws ecs update-service \
                     --service $AWS_ECS_SERVICE_PROD \
                     --task-definition $AWS_ECS_TD_PROD:$LATEST_TD_REVISION \
                     --cluster $AWS_ECS_CLUSTER

                    aws ecs wait services-stable \
                     --services $AWS_ECS_SERVICE_PROD \
                     --cluster $AWS_ECS_CLUSTER

                    echo "Deployment to AWS ECS completed successfully." 
                    '''
                }
            }
        }
    }
}
````
