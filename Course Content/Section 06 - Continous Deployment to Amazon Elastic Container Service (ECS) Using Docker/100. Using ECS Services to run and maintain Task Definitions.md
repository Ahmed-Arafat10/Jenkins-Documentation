## Lecture 100 — Using ECS Services to Run and Maintain Task Definitions

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

## Understanding the Deployment Flow

At this stage of the process:

* An **ECS cluster** has been created.
* A **task definition** has been created.

However, the application is **not running yet**.

If we check the ECS dashboard:

* The cluster shows **no tasks running**
* The cluster shows **no services**

This happens because a **task definition is only a configuration**.

It describes **how a container should run**, but it does not automatically start the container.

To actually run the container, another component is required.

---

## The Role of ECS Services

In ECS, the component responsible for running and maintaining tasks is called a **service**.

A service is responsible for:

* Starting tasks
* Keeping tasks running
* Restarting tasks if they fail
* Maintaining the desired number of running tasks

Services operate inside an ECS cluster and run tasks based on a **task definition**.

This means the deployment flow becomes:

```
Task Definition → Service → Running Task
```

(Added explanation – not mentioned directly in the lecture)
Without a service, tasks can still be launched manually as **one-time tasks**, but services are used for **long-running applications** such as web servers or APIs.

---

## Navigating to the Cluster

To see what is currently running:

1. Open:

Amazon Elastic Container Service

2. Refresh the dashboard.

The dashboard shows:

* **0 tasks running**

Next:

1. Click the **cluster name**.
2. Open the cluster details.

Inside the cluster interface, two important tabs appear:

* **Services**
* **Tasks**

At this point both are empty.

---

## Creating a Service

To run the container defined in the task definition, a **service must be created**.

Steps:

1. Inside the cluster page, open the **Services** tab.
2. Click **Create Service**.

---

## Service Compute Configuration

The first configuration section defines the **compute capacity provider**.

The default option is used:

**Fargate**

AWS Fargate

This means:

* Containers run using serverless compute
* No EC2 instances must be managed
* AWS handles the infrastructure

The default configuration can be kept unchanged.

---

## Selecting the Task Definition

Next, the service must specify **which task definition it will run**.

The previously created task definition family is selected.

Example:

```
learn-jenkins-app-task-definition-prod
```

The **latest revision** is used automatically.

Since this is the first task definition created, the revision number is **1**.

---

## Naming the Service

A service name must also be provided.

Example used in the lecture:

```
learn-jenkins-app-service-prod
```

Naming conventions like this help distinguish environments such as:

* development
* staging
* production

---

## Desired Number of Tasks

The service must specify how many task instances should run.

Configuration used in the lecture:

```
Desired tasks: 1
```

This means:

* One container instance will run
* ECS will ensure exactly one task stays active

If the container stops or crashes, ECS automatically starts a replacement.

---

## Creating the Service

After the configuration is completed, click **Create Service**.

The service begins deploying the task.

Deployment status initially shows:

```
Deployment in progress
```

The process may take a few minutes.

---

## Verifying the Running Task

Once deployment completes:

1. Refresh the cluster view.
2. Open the **Services** tab.

Now the service appears in the list.

Example status:

```
1 / 1 tasks running
```

This confirms the service successfully launched the container.

---

## Viewing the Running Task

To inspect the container:

1. Click the service.
2. Open the **Tasks** tab.

The interface shows the running task instance.

Clicking the task provides detailed information including:

* container status
* networking configuration
* resource usage
* logs

---

## Accessing the Container Through Its Public IP

To access the running container:

1. Open the task details.
2. Go to the **Networking** tab.

The task receives a **Public IP address**.

Opening this IP in a browser should display the application running inside the container.

However, initially the request **fails**.

The browser does not display the application.

---

## Why the Application Is Not Accessible

Although the container is running, AWS networking rules block external access.

This happens because of **security groups**.

Security groups act as **virtual firewalls** controlling network traffic.

AWS uses them to regulate:

* inbound traffic (incoming requests)
* outbound traffic (responses and external connections)

---

## Security Groups in AWS

Security groups are managed using:

Amazon EC2 Security Groups

They control which ports and protocols are allowed.

By default, the container’s security group does **not allow public HTTP access**.

Therefore, requests to port **80** are blocked.

---

## Editing the Security Group

To allow access to the web server:

1. Click the **Security Group** linked to the task.
2. Open the **Inbound Rules** section.
3. Click **Edit inbound rules**.

---

## Adding an HTTP Rule

A new rule must be added to allow HTTP traffic.

Configuration:

**Type**

```
HTTP
```

This automatically opens:

```
Port 80
```

---

### Source Configuration

The source determines **who can access the service**.

Options include:

* a specific IP address
* a network range
* the public internet

To allow public access:

```
IPv4: 0.0.0.0/0
```

This means:

* anyone on the internet can access the web server.

(Added explanation – not mentioned directly in the lecture)
For production systems, it is usually recommended to place containers behind a **load balancer** instead of exposing them directly to the internet.

---

## Saving the Rule

After adding the rule:

1. Save the inbound rules.
2. Wait a few seconds for AWS to apply the change.

The security group now allows HTTP traffic.

---

## Accessing the NGINX Server

Return to the browser tab containing the public IP.

Refresh the page.

Now the default **NGINX welcome page** appears.

This confirms that:

* The container is running
* Networking rules are correct
* The service is accessible from the internet

The application is now successfully deployed.

---

## What Has Been Achieved

At this point the containerized application has been deployed to AWS using ECS.

Deployment components involved:

* ECS Cluster
* Task Definition
* ECS Service
* Docker Container
* Security Group Configuration

The container is running the **NGINX web server** and is accessible through its public IP address.

---

## Conceptual Flow of ECS Deployment

The complete ECS deployment workflow implemented so far:

```
Docker Image (nginx)
        │
        ▼
Task Definition
        │
        ▼
ECS Service
        │
        ▼
ECS Cluster
        │
        ▼
Running Container
        │
        ▼
Accessible via Public IP
```

This demonstrates how containerized applications can be deployed and managed in the AWS cloud using ECS.
