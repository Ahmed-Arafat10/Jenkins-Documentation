## Lecture 112 — Linux `sed` Command for Text Substitution

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

## Problem in the Current Pipeline

Although the CI/CD pipeline works correctly, there is still one remaining issue.

Inside the **task definition JSON file**, the Docker image version is **hardcoded**.

Example:

```json
"image": "123456789012.dkr.ecr.us-east-1.amazonaws.com/learn-jenkins-app:1.0.113"
```

This creates a problem because every time the Jenkins pipeline runs:

* a **new build number** is generated
* a **new Docker image tag** is produced
* the task definition must be **manually updated**

This manual step breaks full automation.

The goal of this lecture is to **remove the hardcoded version and replace it dynamically during the pipeline execution
**.

---

## Introducing the `sed` Command

The solution uses the Linux command:

sed

`sed` stands for **Stream Editor**.

It is a powerful command-line utility used to:

* search text
* replace text
* transform streams of text
* modify files programmatically

A common use case is **search and replace**, similar to the "Find and Replace" functionality in a text editor.

---

## Step 1 — Introduce a Placeholder in the Task Definition

Instead of writing the image version directly in the task definition, a placeholder is inserted.

Example modification in the JSON file:

```json
"image": "123456789012.dkr.ecr.us-east-1.amazonaws.com/learn-jenkins-app:#APP_VERSION#"
```

Here:

```
#APP_VERSION#
```

is a **placeholder** that will later be replaced automatically.

The placeholder is intentionally made unique to avoid accidental replacement of other text.

---

## Step 2 — Use Jenkins Variables as the Source of Truth

The Jenkins pipeline already contains a variable holding the application version.

Example:

```groovy
REACT_APP_VERSION
```

This variable contains the version used to tag Docker images.

The goal is to replace the placeholder with:

```
$REACT_APP_VERSION
```

during the pipeline execution.

---

## Step 3 — Running `sed` Before Registering the Task Definition

Before executing the AWS CLI command that registers the task definition, the pipeline modifies the JSON file using
`sed`.

Example command:

```bash
sed -i "s/#APP_VERSION#/$REACT_APP_VERSION/g" aws/task-definition-prod.json
```

---

## Understanding the `sed` Syntax

The command structure is important to understand.

### Substitution Expression

```
s/search/replace/
```

| Element   | Meaning              |
|-----------|----------------------|
| `s`       | substitution command |
| `search`  | text to find         |
| `replace` | replacement text     |

Example:

```
s/#APP_VERSION#/$REACT_APP_VERSION/
```

This tells `sed` to replace the placeholder with the Jenkins version variable.

---

### Global Replacement Flag

```
g
```

This flag means **global replacement**.

Without it:

* only the **first occurrence** in each line would be replaced.

With it:

* **all occurrences** will be replaced.

---

### Editing the File Directly

```
-i
```

The `-i` option means:

```
edit the file in place
```

Instead of printing the result to the terminal, the file itself is modified.

---

### File Path

The last argument specifies the file to modify.

Example:

```
aws/task-definition-prod.json
```

This is the task definition file stored inside the project repository.

---

## Resulting Pipeline Behavior

When the pipeline runs:

1. Jenkins generates a new build number
2. The Docker image is tagged with that version
3. The placeholder inside the task definition is replaced
4. The updated task definition is registered
5. ECS deploys the new container version

No manual edits are required anymore.

## Final Result

At this stage, the CI/CD pipeline is **fully automated**.

The pipeline now performs the entire deployment lifecycle:

1. Build the application
2. Build the Docker image
3. Tag the image using the Jenkins build number
4. Push the image to ECR
5. Update the ECS task definition dynamically
6. Deploy the container to ECS

The system can now run repeatedly **without any manual configuration updates**.


### updated Jenkinsfile
````
pipeline {
    agent any
    environment {
        REACT_APP_VERSION = "1.0.$BUILD_NUMBER"
        APP_NAME = 'practice-jenkins-ecs-ecr-app'
        AWS_DEFAULT_REGION = 'us-east-1'
        AWS_ECS_CLUSTER = 'LearnJenkinsAppClusterProd'
        AWS_ECS_SERVICE_PROD = 'learn-jenkins-app-task-definition-prod-service-vd0czv2v'
        AWS_ECS_TD_PROD = 'learn-jenkins-app-task-definition-prod'
        AWS_DOCKER_REGISTRY = '508261693782.dkr.ecr.us-east-1.amazonaws.com'
    }
    stages {
        stage('Build App') {
            agent {
                docker {
                    image 'node:18-alpine'
                    reuseNode true
                }
            }
            steps {
                sh '''
                echo "=== Workspace before build ==="
                ls  -lah

                echo "=== Node & NPM versions ==="
                node -v
                npm -v

                echo "=== Installing dependencies ==="
                npm ci

                echo "=== Running build ==="
                npm run build

                echo "=== Workspace after build ==="
                ls -lah
                '''
            }
        }
        stage('Build Docker Image') {
            agent {
                docker {
                    image 'custom-aws-cli'
                    reuseNode true
                    args "-u root -v /var/run/docker.sock:/var/run/docker.sock --entrypoint=''"
                }
            }
            steps {
                withCredentials([usernamePassword(credentialsId: 'aws-ecr', passwordVariable: 'AWS_SECRET_ACCESS_KEY', usernameVariable: 'AWS_ACCESS_KEY_ID')]) {
                    sh '''
                    docker build -t $AWS_DOCKER_REGISTRY/$APP_NAME:$REACT_APP_VERSION .
                    aws ecr get-login-password | docker login \
                    --username AWS \
                    --password-stdin \
                    $AWS_DOCKER_REGISTRY
                    docker push $AWS_DOCKER_REGISTRY/$APP_NAME:$REACT_APP_VERSION
                    '''
                }
            }
        }
        stage('Deploy to AWS ECS') {
            agent {
                docker {
                    image 'custom-aws-cli'
                    reuseNode true
                    args "--entrypoint ''"  //root not needed here
                }
            }
            steps {
                withCredentials([usernamePassword(credentialsId: 'aws-ecs', passwordVariable: 'AWS_SECRET_ACCESS_KEY', usernameVariable: 'AWS_ACCESS_KEY_ID')]) {
                    sh '''
                    aws --version
                    sed -i "s|#APP_VERSION#|$REACT_APP_VERSION|g" aws/task-definition-prod.json
                    LATEST_TD_REVISION=$(aws ecs register-task-definition --cli-input-json file://aws/task-definition-prod.json | jq '.taskDefinition.revision')

                    aws ecs update-service \
                     --service $AWS_ECS_SERVICE_PROD \
                     --task-definition $AWS_ECS_TD_PROD:$LATEST_TD_REVISION \
                     --cluster $AWS_ECS_CLUSTER

                    aws ecs wait services-stable \
                     --services $AWS_ECS_SERVICE_PROD \
                     --cluster $AWS_ECS_CLUSTER

                echo "Deployment to AWS ECS completed successfully."
                '''
                }
            }
       }   
    }
}
````
