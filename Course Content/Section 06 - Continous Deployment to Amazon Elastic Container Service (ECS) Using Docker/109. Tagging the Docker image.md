## Lecture 109 — Tagging the Docker Image

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

# Why Docker Image Tagging Is Important

In the current pipeline setup, each time a Docker image is built, it is tagged only with the application name.

Example:

```bash
docker build -t my-jenkins-app .
```

This means every new build **overwrites the previous image**.

Since Jenkins pipelines generate many builds over time, this approach causes several problems:

* Previous builds are lost
* Images cannot be uniquely identified
* Rollbacks become difficult
* Version tracking becomes impossible

To solve this, Docker images should be **tagged with a version identifier**.

---

# What Is a Docker Tag

A Docker image consists of two parts:

```
image-name:tag
```

Example:

```bash
my-app:1.0
```

| Component | Meaning             |
| --------- | ------------------- |
| `my-app`  | Image name          |
| `1.0`     | Image tag (version) |

If a tag is not provided, Docker uses the default tag:

```
latest
```

Using only `latest` is not recommended in CI/CD pipelines because it hides which build produced the image.

---

# Using Jenkins Build Numbers as Tags

Jenkins automatically provides a variable called:

```text
$BUILD_ID
```

This variable contains the **unique identifier of the current pipeline build**.

Example:

| Jenkins Build | BUILD_ID |
| ------------- | -------- |
| Build #107    | 107      |
| Build #108    | 108      |
| Build #109    | 109      |

By using this value as a Docker tag, each image becomes uniquely identifiable.

---

# Example Docker Tag With Build ID

Instead of:

```bash
docker build -t my-jenkins-app .
```

The pipeline will use:

```bash
docker build -t my-jenkins-app:$BUILD_ID .
```

Example resulting images:

```
my-jenkins-app:107
my-jenkins-app:108
my-jenkins-app:109
```

Each image now corresponds directly to a Jenkins build.

---

# Using an Application Version Variable

The pipeline already contains an environment variable representing the application version.

Example:

```text
REACT_APP_VERSION
```

This value can be combined with the build number to create a structured tag.

Example:

```
1.0.109
```

Where:

| Part | Meaning              |
| ---- | -------------------- |
| 1.0  | Application version  |
| 109  | Jenkins build number |

---

# Example Tagged Image

Final image format:

```bash
learn-jenkins-app:1.0.109
```

This tag provides clear traceability:

* Application version → **1.0**
* Jenkins build → **109**

---

# Avoid Hardcoding Values in Commands

A good pipeline practice is to avoid placing configuration values directly inside commands.

Instead, environment variables should be used.

---

# Defining an Application Name Variable

A new environment variable is created:

```groovy
APP_NAME="learn-jenkins-app"
```

This variable represents the Docker image name.

---

# Updated Docker Build Command

Using environment variables:

```bash
docker build -t $APP_NAME:$REACT_APP_VERSION.$BUILD_ID .
```

This command dynamically generates the image tag during the pipeline execution.

---

# Example Jenkins Pipeline Configuration

Environment variables:

```groovy
environment {
    APP_NAME = "learn-jenkins-app"
}
```

Docker build step:

```groovy
stage('Build Docker Image') {
    steps {
        sh '''
        docker build -t $APP_NAME:$REACT_APP_VERSION.$BUILD_ID .
        '''
    }
}
```

---

# Resulting Image Tag Example

If the following values exist:

| Variable          | Value             |
| ----------------- | ----------------- |
| APP_NAME          | learn-jenkins-app |
| REACT_APP_VERSION | 1.0               |
| BUILD_ID          | 109               |

The resulting Docker image becomes:

```
learn-jenkins-app:1.0.109
```

---

# Verifying the Tag in Pipeline Logs

During the pipeline execution, the log shows the resolved variables.

Example output:

```
docker build -t learn-jenkins-app:1.0.109 .
```

This confirms that:

* variables were successfully replaced
* the image was tagged correctly

---

# Mapping Between Jenkins Builds and Docker Images

Because the build number is used in the tag, there is now a direct relationship between Jenkins builds and Docker images.

Example:

| Jenkins Build | Docker Image              |
| ------------- | ------------------------- |
| Build #107    | learn-jenkins-app:1.0.107 |
| Build #108    | learn-jenkins-app:1.0.108 |
| Build #109    | learn-jenkins-app:1.0.109 |

This allows teams to:

* track deployments
* reproduce builds
* perform rollbacks

---

# Benefits of Proper Docker Tagging

### Version Tracking

Each image corresponds to a specific build.

---

### Easier Rollbacks

If a deployment fails, a previous tag can be redeployed.

Example:

```
learn-jenkins-app:1.0.108
```

---

### Debugging

Issues can be traced back to the exact build that produced the image.

---

### Deployment Traceability

Teams can see exactly which application version is running in production.
