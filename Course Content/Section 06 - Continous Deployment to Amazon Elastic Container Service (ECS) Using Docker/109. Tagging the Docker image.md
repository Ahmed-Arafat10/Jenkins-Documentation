## Lecture 109 — Tagging the Docker Image

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

# Why Docker Image Tagging Is Important

Up to this point, the Docker image built in the pipeline had a fixed name:

```bash
docker build -t my-jenkins-app .
```

This approach has a major problem:

Every new build **overwrites the previous image**.

In a CI/CD pipeline where builds happen frequently, this makes it impossible to:

* identify which build produced which image
* deploy a specific version
* roll back to a previous version

Therefore, Docker images must be **tagged properly**.

---

# What Is a Docker Image Tag

A Docker image consists of two main identifiers:

```id="ubmx7r"
image_name:tag
```

Example:

```id="hfnz4b"
learn-jenkins-app:1.0.109
```

### Components

| Part                | Meaning                        |
| ------------------- | ------------------------------ |
| `learn-jenkins-app` | Image name                     |
| `1.0.109`           | Image tag (version identifier) |

The **tag uniquely identifies a specific version of the image**.

---

# Problems with Untagged Images

If images are not tagged properly, every build will replace the previous one.

Example pipeline behavior without tags:

```id="rbcmh2"
Build #108 → my-jenkins-app
Build #109 → my-jenkins-app (overwrites previous image)
```

Result:

* older versions cannot be deployed
* rollback becomes impossible
* debugging becomes harder

---

# Using Version Tags in CI/CD

A common CI/CD strategy is to tag images using:

* application version
* build number

Example:

```id="kt5u3d"
app_version.build_number
```

This ensures that every build produces a **unique image**.

---

# Using Pipeline Variables

The pipeline already contains a version variable:

```id="hxet91"
REACT_APP_VERSION
```

This variable represents the **application version**.

Example:

```id="z0ntp6"
1.0
```

---

# Adding the Build Number

Jenkins automatically provides a variable:

```id="t7t6ch"
BUILD_NUMBER
```

Example:

```id="g5w9wq"
109
```

Combining these values creates a unique image version.

Example final tag:

```id="uyr2z3"
1.0.109
```

---

# Updating the Docker Build Command

The Docker build command is updated to include the version tag.

Example:

```bash id="6bb88m"
docker build -t learn-jenkins-app:$REACT_APP_VERSION.$BUILD_NUMBER .
```

Now each build generates a uniquely tagged image.

---

# Removing Hardcoded Configuration

Hardcoded values inside commands make pipelines difficult to maintain.

Example of a hardcoded image name:

```bash
docker build -t learn-jenkins-app .
```

To improve maintainability, the image name is stored in a variable.

---

# Defining an Application Name Variable

A new environment variable is introduced.

Example:

```groovy id="ds9vpa"
APP_NAME = "learn-jenkins-app"
```

This variable stores the name of the Docker image.

---

# Using the Variable in the Docker Command

The Docker build command becomes:

```bash id="nshrdg"
docker build -t $APP_NAME:$REACT_APP_VERSION.$BUILD_NUMBER .
```

Now both the **image name** and **version** are dynamic.

---

# Example Image Generated by the Pipeline

Suppose the pipeline is running:

* Application version → `1.0`
* Jenkins build number → `109`

The resulting image will be:

```id="t6h8sl"
learn-jenkins-app:1.0.109
```

This provides a direct link between:

* Jenkins build
* Docker image
* deployed application version

---

# Verifying the Image Tag in Pipeline Logs

The pipeline logs show the Docker command executed.

Example output:

```id="64qf2e"
docker build -t learn-jenkins-app:1.0.109 .
```

This confirms the correct tag was applied.

---

# Correlation Between Jenkins Builds and Docker Images

Jenkins build history:

| Build | Docker Image              |
| ----- | ------------------------- |
| #107  | learn-jenkins-app:1.0.107 |
| #108  | learn-jenkins-app:1.0.108 |
| #109  | learn-jenkins-app:1.0.109 |

This allows precise tracking of deployments.

---

# Benefits of Proper Image Tagging

### Version Traceability

Each Docker image can be linked directly to a specific build.

---

### Safer Deployments

The deployment process can specify exactly which version to run.

Example:

```id="o4bbq8"
learn-jenkins-app:1.0.109
```

---

### Easy Rollbacks

If a deployment fails, ECS can redeploy a previous image version.

Example rollback:

```id="fw1y9f"
learn-jenkins-app:1.0.108
```

---

### Better Debugging

Developers can identify which build introduced a bug.

---

# Final Docker Image Tag Format Used in the Pipeline

The final format used in the course is:

```id="pm8r9k"
$app_name:$app_version.$build_number
```

Example:

```id="1by5h6"
learn-jenkins-app:1.0.109
```
