## Lecture 103 — Passing Data Between AWS CLI Commands

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

## The Remaining Problem in the Automation

In the previous lecture, the Jenkins pipeline performed two main operations:

1. Register a **new ECS task definition revision**
2. Update the **ECS service** to use that revision

However, the update command required **manually specifying the revision number**.

Example:

```bash
--task-definition learn-jenkins-app-task-definition-prod:2
```

This introduces a problem:

* Every time a new task definition revision is created, the revision number changes.
* The Jenkins pipeline would need to be **manually updated** each time.

Example scenario:

```
Revision 1 → nginx:1.26
Revision 2 → nginx:1.27
Revision 3 → nginx:1.28
Revision 4 → nginx:1.29
```

Each deployment would require updating the pipeline script.

This defeats the purpose of **full CI/CD automation**.

The solution is to **capture the revision number automatically from the AWS CLI response**.

---

## Understanding the AWS CLI Response

When running the command:

```bash
aws ecs register-task-definition
```

AWS returns a **JSON response** containing information about the newly created task definition.

Example structure (simplified):

```json
{
  "taskDefinition": {
      "family": "learn-jenkins-app-task-definition-prod",
      "revision": 3
  }
}
```

Inside the JSON response, the important field is:

```
taskDefinition.revision
```

This value represents the **newly created revision number**.

The goal is to extract this value automatically.

---

## Extracting JSON Data with `jq`

To parse JSON output in the shell, the tool **jq** is used.

jq JSON processor

`jq` is a lightweight command-line tool designed for:

* Parsing JSON
* Extracting specific fields
* Transforming JSON output

This tool is commonly used in DevOps pipelines to process API responses.

---

## Installing jq in the Jenkins Container

The Jenkins pipeline uses an **AWS CLI Docker image**.

This image does not include `jq` by default.

Therefore it must be installed during the pipeline execution.

Because the container is based on Amazon Linux, the package manager used is:

```
yum
```

Installation command:

```bash
yum install jq -y
```

Explanation:

| Part      | Meaning                                                      |
| --------- | ------------------------------------------------------------ |
| `yum`     | Package manager used in Amazon Linux / Red Hat based systems |
| `install` | Install a package                                            |
| `jq`      | JSON processor                                               |
| `-y`      | Automatically answer "yes" to prompts                        |

(Added explanation – not mentioned directly in the lecture)
`yum` is widely used in distributions derived from **Red Hat Enterprise Linux**, including Amazon Linux and CentOS.

---

## Parsing the Revision from AWS CLI Output

After registering a task definition, the JSON response is piped into `jq`.

Command structure:

```bash
aws ecs register-task-definition ... | jq '.taskDefinition.revision'
```

Explanation:

| Part                       | Meaning                    |                                                  |
| -------------------------- | -------------------------- | ------------------------------------------------ |
| `                          | `                          | Pipe operator (passes output to another command) |
| `jq`                       | JSON processor             |                                                  |
| `.taskDefinition.revision` | Path to the revision field |                                                  |

This command extracts the revision number from the JSON output.

---

## Storing the Revision in a Variable

The next step is storing the extracted value in a shell variable.

Shell syntax for capturing command output:

```bash
variable_name=$(command)
```

Example used in the pipeline:

```bash
latest_td_revision=$(aws ecs register-task-definition \
--cli-input-json file://aws/task-definition-prod.json \
| jq '.taskDefinition.revision')
```

Explanation:

| Component                          | Purpose                                    |
| ---------------------------------- | ------------------------------------------ |
| `latest_td_revision`               | Variable storing the revision              |
| `$()`                              | Executes the command and stores the result |
| `aws ecs register-task-definition` | Registers the new task definition          |
| `jq`                               | Extracts the revision                      |

---

## Debugging the Variable Value

To verify the value during pipeline execution, the script prints the variable.

Example:

```bash
echo $latest_td_revision
```

This helps confirm that the correct revision number is captured.

---

## Using the Variable in the Next Command

Once the revision number is stored in a variable, it can be used in the **service update command**.

Example:

```bash
aws ecs update-service \
--cluster learn-jenkins-app-cluster-prod \
--service learn-jenkins-app-service-prod \
--task-definition learn-jenkins-app-task-definition-prod:$latest_td_revision
```

Now the pipeline automatically deploys the **latest revision** without manual updates.

---

## Jenkins Pipeline Implementation

The deployment stage now performs three steps:

1. Install `jq`
2. Register the new task definition
3. Update the ECS service using the extracted revision

Example implementation:

```groovy
stage('Deploy to AWS') {
    steps {
        sh '''
        yum install jq -y

        latest_td_revision=$(aws ecs register-task-definition \
        --cli-input-json file://aws/task-definition-prod.json \
        | jq '.taskDefinition.revision')

        echo $latest_td_revision

        aws ecs update-service \
        --cluster learn-jenkins-app-cluster-prod \
        --service learn-jenkins-app-service-prod \
        --task-definition learn-jenkins-app-task-definition-prod:$latest_td_revision
        '''
    }
}
```

---

## Permission Issue When Installing jq

The first pipeline execution fails with the error:

```
Permission denied
You need to be root to perform this command
```

This happens because installing packages requires **root privileges**.

The solution is to run the container as root.

Inside the Jenkins pipeline configuration, the Docker agent is modified.

Example:

```groovy
args '-u root'
```

This allows the container to install system packages like `jq`.

---

## Successful Pipeline Execution

After fixing permissions, the pipeline successfully runs.

Example log output:

```
Installed jq
Latest revision: 4
Updating service using revision 4
```

The ECS service is updated automatically using the correct revision.

---

## Verifying the Deployment

Inside the ECS console:

1. Open the **cluster**
2. Navigate to **services**
3. Observe the running tasks

The service temporarily shows:

```
2 / 1 tasks running
```

This indicates a **rolling deployment**.

One task is the old version, and the new task is starting.

---

## Confirming the Task Definition Revision

Inside **Task Definitions**, the newest revision is visible.

Example:

```
Revision 4
```

The service now runs using that revision.

Even if the container image did not change, ECS still creates a **new task instance**.

---

## What Was Achieved in This Lecture

The pipeline now performs **fully automated deployments**.

Instead of manually specifying revision numbers, the pipeline:

1. Registers a new task definition
2. Extracts the revision from the AWS CLI response
3. Passes that revision to the update command
