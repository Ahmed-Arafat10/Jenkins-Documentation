## Lecture 101 — Updating the Task Definition Using AWS CLI

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

## The Problem: Deploying a New Version of the Application

At this point in the workflow:

* An ECS cluster exists
* A service is running
* A container (NGINX) is already deployed

But applications evolve over time. When a new version of the container is released, the deployment must be updated.

A common question is:

> Do we need to create a new cluster and repeat the entire process?

The answer is **no**.

Instead, the update process follows this approach:

1. Create a **new revision of the task definition**
2. Update the **ECS service** to use that new revision

This allows the infrastructure to remain unchanged while only updating the container version.

---

## ECS Task Definition Versioning

In ECS, task definitions are **versioned automatically**.

Every modification creates a new **revision**.

Example:

```
Revision 1 → nginx:1.26
Revision 2 → nginx:1.27
```

The service can then switch to the latest revision without modifying the cluster.

---

## Manual Update Through the AWS Console (Concept Overview)

If performing the update manually:

1. Navigate to **Task Definitions**
2. Select the existing task definition
3. Create a **new revision**
4. Modify the container image
5. Save the revision

Then update the service:

1. Open the ECS cluster
2. Select the service
3. Click **Update Service**
4. Choose the **new task definition revision**

However, performing these steps manually defeats the purpose of **automation with Jenkins**.

Therefore, the goal is to automate this process using the **AWS CLI inside the Jenkins pipeline**.

Inside the task definition JSON, the container image tag is updated.

Example change:

```
nginx:1.26-alpine
```

Updated version:

```

```

This simulates deploying a **new version of the application container**.

## Automating the Update Using AWS CLI

Instead of using the AWS console, the update is executed using the AWS CLI.

The CLI command used is:

```
aws ecs register-task-definition
```

This command registers a new revision of a task definition.

AWS documentation provides a simplified example for registering a task definition from a JSON file.

The command format is:

```
aws ecs register-task-definition \
  --cli-input-json file://task-definition.json
```

This tells AWS:

* Read configuration from a JSON file
* Register it as a new task definition revision

---

## Implementing the Command in Jenkins

Inside the Jenkins pipeline, the command is executed within the **AWS CLI Docker container**.

Example pipeline snippet:

```groovy
stage('Deploy to AWS') {
    steps {
        sh '''
        aws ecs register-task-definition \
        --cli-input-json file://aws/task-definition-prod.json
        '''
    }
}
```

This step performs the following:

1. Jenkins reads the task definition JSON file
2. AWS CLI registers it
3. ECS creates a **new revision automatically**

## First Pipeline Error: Missing Environment Variables

The first pipeline run fails due to an empty environment block.

Example issue:

```
environment {
}
```

Jenkins does not accept an empty environment block.

Solution:

Remove the block entirely.

---

## Second Pipeline Error: Missing AWS Region

The next error encountered during pipeline execution:

```
You must specify a region
```

The AWS CLI requires a region configuration.

To fix this, an environment variable must be defined:

```
AWS_DEFAULT_REGION
```

Example configuration:

```groovy
environment {
    AWS_DEFAULT_REGION = "us-east-1"
}
```

Region explanation:

```
us-east-1
```

This corresponds to the **US East (N. Virginia)** AWS region.

---

## Third Pipeline Error: Access Denied

After fixing the region issue, the pipeline fails again with:

```
AccessDeniedException
```

Error message summary:

```
User is not authorized to perform ecs:RegisterTaskDefinition
```

This indicates that the Jenkins IAM user does **not have permission** to interact with ECS.

---

## Fixing IAM Permissions

To resolve this:

1. Open AWS console
2. Navigate to **IAM**
3. Select the **Jenkins user**
4. Add permissions

Attach the policy:

```
AmazonECS_FullAccess
```

This policy grants permissions required for ECS operations.

(Added explanation – not mentioned directly in the lecture)
In production environments it is recommended to use **least privilege policies** instead of full access permissions.

---

## Successful Pipeline Execution

After fixing permissions and re-running the pipeline:

The command executes successfully:

```
aws ecs register-task-definition
```

A new task definition revision is created.

## Verifying the New Task Definition

To confirm the update:

1. Open the ECS console
2. Navigate to **Task Definitions**
3. Select the task definition family

The interface now shows:

```
Revision 1
Revision 2
```

Revision 2 was created by the Jenkins pipeline.

Opening the JSON configuration reveals the updated image:

```
nginx:1.27-alpine
```

This confirms that the task definition was updated successfully through **automation using Jenkins and AWS CLI**.

## What Has Been Achieved

The first step of automated deployment has been completed.

The pipeline now performs:

```
Jenkins Pipeline
        │
        ▼
AWS CLI Command
        │
        ▼
Register New ECS Task Definition
        │
        ▼
Create New Revision
```

However, the **service is still running the old revision**.

The next step will be to **update the ECS service to use the new task definition revision**, which will deploy the new container version.

````jenkins
pipeline {
    agent any
    environment {
        AWS_DEFAULT_REGION = 'us-east-1'
    }
    stages {
        stage('Deploy to AWS ECS') {
            agent {
                docker {
                    image 'amazon/aws-cli'
                    reuseNode true
                    args "--entrypoint ''"
                }
            }
            steps {
               withCredentials([usernamePassword(credentialsId: 'aws-ecs', passwordVariable: 'AWS_SECRET_ACCESS_KEY', usernameVariable: 'AWS_ACCESS_KEY_ID')]) {
                    sh '''
                    aws --version
                    aws ecs register-task-definition --cli-input-json file://aws/task-definition-prod.json
                    '''
                }
            }
        }
    }
}
````
