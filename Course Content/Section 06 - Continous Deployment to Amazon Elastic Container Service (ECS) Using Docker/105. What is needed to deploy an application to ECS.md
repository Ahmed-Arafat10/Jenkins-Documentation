## Key Requirement: ECS Works Only with Docker Images

ECS does not accept raw application files.

Unlike services such as:

* static hosting platforms
* object storage services

ECS requires applications to be packaged as **Docker container images**.

This means:

* You cannot upload HTML files directly
* You cannot upload compiled binaries directly
* You must provide a **Docker image**

A Docker image contains everything required to run the application.

---

## Step 1 — Creating a Dockerfile

The first step in deploying a custom application is creating a **Dockerfile**.

Docker

A Dockerfile is a configuration file that describes **how to build a container image**.

It contains instructions such as:

* selecting the base image
* copying application files
* installing dependencies
* defining the startup command

Example simplified structure:

```dockerfile id="ymjz36"
FROM nginx:alpine
COPY ./website /usr/share/nginx/html
```

In this example:

* the container uses NGINX as the base image
* the website files are copied into the server directory

(Added explanation – not mentioned directly in the lecture)
The Dockerfile defines the **reproducible build process** for container images, ensuring the same environment can be recreated across development, testing, and production.

---

## Step 2 — Building the Docker Image

After creating the Dockerfile, the next step is building the container image.

This is done using the Docker build command:

```bash id="m6fsys"
docker build
```

Example command:

```bash id="6g6oxu"
docker build -t my-application .
```

Explanation:

| Part           | Meaning                                         |
| -------------- | ----------------------------------------------- |
| `docker build` | Builds a Docker image                           |
| `-t`           | Assigns a tag to the image                      |
| `.`            | Uses the current directory as the build context |

The result of this command is a **Docker image stored locally**.

---

## Step 3 — Understanding Docker Images

A Docker image is not a typical file like:

* `.zip`
* `.tar`
* `.jar`

Instead, Docker images are stored inside the **Docker image repository of the system**.

Because of this:

* They cannot simply be uploaded as build artifacts
* They cannot be uploaded to S3
* They cannot be directly sent to ECS

Instead, they must be stored in a **container registry**.

---

## Step 4 — Using a Container Registry

A **container registry** is a repository designed specifically for storing Docker images.

Registries allow:

* storing container images
* versioning images with tags
* sharing images across environments
* pulling images when deploying containers

A well-known example of a container registry is:

Docker Hub

Docker Hub hosts a large number of **public container images**, including:

* NGINX
* MySQL
* Redis
* Node.js

The NGINX image used earlier in the course was pulled from Docker Hub.

---

## Public vs Private Container Images

Docker Hub supports both:

* public images
* private images

However, many projects prefer not to store private application images in public registries.

For example:

* internal company applications
* proprietary software
* private APIs

Publishing these images publicly would allow anyone to download them.

Therefore, private container registries are commonly used.

---

## Step 5 — Using Amazon Elastic Container Registry (ECR)

AWS provides its own container registry service called:

Amazon Elastic Container Registry

Amazon ECR is a **private Docker container registry** designed for AWS environments.

It allows developers to:

* store Docker images securely
* manage image versions
* integrate directly with ECS
* control access through AWS IAM policies

Because ECR is an AWS service, it integrates very smoothly with ECS.

This simplifies container deployment compared to using external registries.

(Added explanation – not mentioned directly in the lecture)
ECR supports both **private repositories** and **public repositories**, but in most enterprise environments it is used as a private container registry.

---

## Full Deployment Workflow for a Custom Application

To deploy a real application to ECS, the workflow becomes:

1. Create a **Dockerfile**
2. Build the **Docker image**
3. Push the image to **Amazon ECR**
4. Deploy the image to **Amazon ECS**
