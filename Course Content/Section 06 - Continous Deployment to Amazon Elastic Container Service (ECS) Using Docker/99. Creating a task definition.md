## Lecture 99 — Creating a Task Definition

**Section:** Continuous Deployment to Amazon Elastic Container Service (ECS) Using Docker

---

## Why a Task Definition Is Needed

After creating an ECS cluster, the cluster itself **does nothing** until instructions are provided.

To tell ECS **what container should run and how it should run**, we must create a **task definition**.

In ECS, a task definition acts as a **configuration blueprint** that describes:

* Which Docker container to run
* Resource allocation (CPU and memory)
* Network configuration
* Port mappings
* Environment settings

This configuration allows ECS to launch containers consistently.

---

## Navigating to Task Definitions in ECS

Steps inside the AWS console:

1. Open the ECS dashboard.
2. In the left sidebar, locate **Task Definitions**.
3. Click **Task Definitions**.
4. Click **Create New Task Definition**.

Instead of using the graphical interface, the lecture selects:

**Create New Task Definition with JSON**

Using JSON allows the configuration to be written manually.

---

## Why JSON Is Used

AWS provides a graphical UI for creating task definitions, but the lecture intentionally uses **JSON configuration**.

Reasons for using JSON:

* Provides better understanding of ECS configuration
* Allows easy **automation**
* Enables configuration to be stored in version control
* Makes it easier to integrate with CI/CD pipelines later

JSON-based task definitions are especially useful when automating deployments using tools like **Jenkins pipelines**.

---

## Task Definition Compatibility

At the top of the JSON configuration, ECS indicates that the task definition is compatible with:

AWS Fargate

This means the task will run using the **Fargate launch mode** instead of EC2 infrastructure.

---

## Defining the Task Definition Family

The first important property is the **family name**.

The family represents the **logical name of the task definition**.

Example used in the lecture:

```json
"family": "learn-jenkins-app-task-definition-prod"
```

This naming convention suggests the task definition belongs to a **production environment**.

(Added explanation – not mentioned directly in the lecture)
Task definitions are versioned in ECS. Every modification creates a **new revision**, allowing rollback to previous configurations.

---

## Container Definitions

The most important part of the task definition is the **containerDefinitions** section.

This section specifies which containers should run as part of the task.

In this lecture, only **one container** will run.

The container image used comes from **Docker Hub**.

Docker Hub

The selected image is the **NGINX web server container**.

NGINX

---

## Selecting the NGINX Docker Image

Inside Docker Hub, the available image tags are listed.

The lecture chooses the following image version:

```
nginx:1.26-alpine
```

Reasons for choosing this tag:

* `1.26` specifies the NGINX version
* `alpine` uses Alpine Linux, which is lightweight

(Added explanation – not mentioned directly in the lecture)
Alpine-based Docker images are commonly used because they are significantly smaller than standard Linux images, which reduces container startup time and storage usage.

Each container defined in the task requires a name.

Example used in the lecture:

## Configuring Port Mapping

To access the containerized application, networking configuration must be defined.

This is done using **port mappings**.

Port mapping connects:

* The **container port**
* The **host port**

Example configuration:

```json
"portMappings": [
  {
    "name": "nginx-80-tcp",
    "containerPort": 80,
    "hostPort": 80,
    "protocol": "tcp",
    "appProtocol": "http"
  }
]
```

Explanation of each field:

### name

```
nginx-80-tcp
```

An identifier for the port mapping.

---

### containerPort

```
80
```

The port inside the container where the application listens.

NGINX listens on **port 80** by default.

---

### hostPort

```
80
```

The port exposed on the host infrastructure.

Requests arriving at this port are forwarded to the container.

---

### protocol

```
tcp
```

Defines the communication protocol used by the port.

HTTP traffic runs over TCP.

---

### appProtocol

```
http
```

Specifies the application-level protocol.

This helps ECS configure networking correctly.

---

## Resource Configuration (CPU and Memory)

The task definition must specify the compute resources required by the container.

These values directly influence **the cost of running the container**.

AWS provides a compatibility table defining which CPU and memory combinations are allowed.

The lecture chooses the **lowest possible resources** to minimize cost.

### CPU Allocation

The chosen value is:

```
256
```

This corresponds to:

```
0.25 vCPU
```

Meaning the container receives **one quarter of a virtual CPU**.

---

### Memory Allocation

The chosen value is:

```
512
```

This represents:

```
512 MB (0.5 GB) of memory
```

This amount is sufficient for a lightweight application like NGINX.

The default memory value was **3 GB**, which would unnecessarily increase costs.

## Handling the Memory Validation Error

When attempting to create the task definition, ECS returns an error:

```
memory is invalid for this task
```

This happens because the memory value initially included a **unit (MB or GB)**.

To fix the issue:

* Remove the unit
* Provide only the numeric value

Correct example:

```
512
```

instead of

```
512MB
```

After correcting this, the task definition can be created successfully.

## Current Deployment Status

At this stage:

* The **ECS cluster exists**
* The **task definition exists**
* The **container image is defined**

However, the container **has not yet been launched**.

The next step will be to **run the task inside the cluster**, which will start the containerized application.
