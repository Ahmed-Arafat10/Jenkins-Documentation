# 37. Running E2E Tests with Playwright (End-to-End tests)

### 🔹 What Are End-to-End (E2E) Tests?

* E2E tests validate the **entire application flow** in a running environment.
* For web apps: they interact with the actual UI (browser automation).
* **Playwright** is a popular framework for E2E testing.
* Unlike unit tests, E2E tests require a **running version of the application**.

---

### 🔹 Running the App Locally Before Testing

1. Build the application:

   ```bash
   npm run build
   ```

    * Produces the `build/` folder, ready to be served.

2. Start a local server using `serve`:

   ```bash
   npm install -g serve
   serve -s build
   ```

    * Spins up a lightweight HTTP server for testing.
    * Opens at `http://localhost:3000` (default).
    * **Not production-ready**, but good for testing.

---

### 🔹 Integrating Playwright E2E Tests in Jenkins

1. **Add an E2E Stage in Jenkinsfile**

    * Duplicate an existing `test` stage, rename it to `E2E`.
    * Use **Playwright’s official Docker image**:

      ```groovy
      stage('E2E') {
          agent {
              docker {
                  image 'mcr.microsoft.com/playwright:v1.43.0-focal' 
              }
          }
          steps {
              sh '''
                  npm install serve
                  node_modules/.bin/serve -s build &
                  sleep 10
                  npx playwright test
              '''
          }
      }
      ```

        * Installs `serve` **locally** (avoiding root issues).
        * Starts the server in the background (`&`).
        * Adds a `sleep` delay to ensure the server is running.
        * Runs Playwright tests.

2. **Important Note on Root Issue**

    * Attempting `npm install -g serve` in container → **permission errors**.
    * Tempting fix: run container as `root` (`args '-u root:root'`) ❌ **Not recommended**.
    * Correct fix: install `serve` as a **local dependency** and call it via `node_modules/.bin/serve`.

---

### 🔹 Handling Blocking Processes

* Running `serve` **without `&`** blocks Jenkins job forever.
* Solution: append `&` → runs in background, allows pipeline to continue.
* Add `sleep 10` to give server time to start before Playwright runs.

---

### 🔹 Test Result Publishing Conflict

* Jenkins expects **JUnit XML test results** in `test-results/`.
* Conflict:

    * Unit tests write results.
    * E2E tests overwrite them.
* Fix Options:

    1. Configure **Playwright/Jest** to output to a different directory in `package.json` > `jest-junit.outputDirectory="jest-results"`.
       Example: change `jest` results directory to `results/`.
    2. Move the `post { always { junit ... } }` block inside **individual stages**.

---

### 🔹 Final Working Flow

* `npm run build` → produces `build/`.
* `serve -s build &` → launches app server in background.
* `sleep 10` → ensures server is up.
* `npx playwright test` → runs E2E tests.
* Jenkins collects test results properly (no overwriting).

---

### 🔹 Key Takeaways

* Use **Playwright Docker image** for browser dependencies.
* Always start app server **in background** to unblock pipeline.
* Avoid running containers as **root** unless absolutely necessary.
* Manage **test result directories** to prevent conflicts between unit and E2E tests.
* Successful setup = green pipeline + E2E validation of deployed build.

---

✅ With this setup, your Jenkins pipeline now runs **unit tests** and **E2E Playwright tests** against a locally served build.



- Jenkinsfile
````
pipeline {
    agent any

    stages {
        
        stage('Build') {
            agent {
                docker {
                    image 'node:18-alpine'
                    reuseNode true
                }
            }
            steps {
                sh '''
                    ls -la
                    node --version
                    npm --version
                    npm ci
                    npm run build
                    ls -la
                '''
            }
        }

        stage('Test') {
            agent {
                docker {
                    image 'node:18-alpine'
                    reuseNode true
                }
            }

            steps {
                sh '''
                    #test -f build/index.html
                    npm test
                '''
            }
        }

        stage('E2E') {
            agent {
                docker {
                    image 'mcr.microsoft.com/playwright:v1.39.0-jammy'
                    reuseNode true
                }
            }

            steps {
                sh '''
                    npm install serve
                    node_modules/.bin/serve -s build &
                    sleep 10
                    npx playwright test
                '''
            }
        }
    }

    post {
        always {
            junit 'jest-results/junit.xml'
        }
    }
}
````


### Extra

Good question 👌 Let’s carefully break this down because these commands are related but have **different roles** in the React (or similar Node.js) project lifecycle:

---

## 1. `npm start`

* Runs the **development server**.
* Typically uses `react-scripts start` (if CRA/React) under the hood.
* Features:

    * Starts a hot-reloading dev server (via **webpack-dev-server** or **Vite**).
    * Serves files directly from memory (not from `build/` folder).
    * Optimized for developer experience (fast refresh, source maps).
    * Not optimized for performance/production.

👉 Usage: While **developing**.

---

## 2. `npm run build`

* Builds an **optimized production bundle**.
* Generates static files (HTML, CSS, JS) inside a `/build` folder.
* Features:

    * Minifies code.
    * Removes dev-only helpers.
    * Optimizes assets (tree-shaking, code-splitting, etc.).
    * Produces static files ready to be deployed on a server (e.g., Nginx, Apache, Netlify, Vercel).

👉 Usage: When you want to **deploy** your app.

---

## 3. `serve -s build`

* Runs a local **static file server** to serve the `/build` folder.
* Requires the [`serve`](https://www.npmjs.com/package/serve) package (installed globally or via `npx`).
* `-s` (or `--single`) means **single-page app mode**:

    * All routes fallback to `index.html` (important for React Router apps).
* Features:

    * Mimics how your app will behave in production.
    * Doesn’t provide hot reload.
    * Just serves the **already built files** (from `npm run build`).

👉 Usage: To **test your production build locally** before deploying.

---

## ⚡ Quick Comparison

| Command          | Purpose               | Runs Dev Server?   | Optimized? | Output          |
| ---------------- | --------------------- | ------------------ | ---------- | --------------- |
| `npm start`      | Development           | ✅ Yes (hot reload) | ❌ No       | Memory          |
| `npm run build`  | Production Build      | ❌ No               | ✅ Yes      | `/build` folder |
| `serve -s build` | Test Production Build | ✅ Yes (but static) | ✅ Yes      | `/build` files  |

---

✅ **Rule of thumb:**

* During development → use `npm start`.
* Before deployment → run `npm run build`.
* To locally preview production → run `serve -s build`.
