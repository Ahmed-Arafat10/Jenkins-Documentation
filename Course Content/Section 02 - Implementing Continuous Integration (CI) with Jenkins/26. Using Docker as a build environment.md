# Lecture 26 – Using Docker as a Build Environment

## Objective

Understand how to use **Docker containers** inside Jenkins pipelines to provide consistent build environments with the required tools (e.g., Node.js and npm).

---

## Problem Statement

* Jenkins agents do not come preinstalled with tools like **Node.js** or **npm**.
* Example: running `npm --version` in a pipeline fails with:

  ```
  sh: npm: not found
  ```
* GitHub Codespaces or IDEs may automatically provide the right environment, but Jenkins does not know what tools/projects require.

---

## Approaches to Solve This

### 1. **Install tools manually on Jenkins agents**

* Install Node.js and npm directly on each agent.
* **Issues with this approach:**

    * Different projects may require different versions (e.g., Node 10 vs Node 18).
    * Maintaining multiple versions on the same agent is complex and error-prone.
    * Conflicting dependencies may break builds.

### 2. **Use Docker for build environments (preferred)**

* Docker allows creating **isolated containers per project** with all necessary tools.
* Each pipeline stage can specify its own image (e.g., Node.js v18).
* This avoids version conflicts and keeps agents clean.

---

## Jenkins Pipeline with and without Docker

### Example: Stage without Docker

```groovy
pipeline {
    agent any
    stages {
        stage('Without Docker') {
            steps {
                sh 'echo Without Docker'
            }
        }
    }
}
```

### Example: Stage with Docker

```groovy
pipeline {
    agent any
    stages {
        stage('With Docker') {
            agent {
                docker {
                    image 'node:18-alpine'
                }
            }
            steps {
                sh 'echo With Docker'
                sh 'npm --version'
            }
        }
    }
}
```

* `node:18-alpine` = a lightweight Linux distribution (Alpine) with Node.js v18 and npm.
* First run → pulls Docker image from the registry (longer runtime).
* Subsequent runs → image cached locally (much faster).

---

## Steps to Enable Docker in Jenkins

1. **Install the Docker Pipeline plugin**

    * Go to: **Manage Jenkins → Plugins → Available Plugins**.

    * Search for **Docker Pipeline**.

    * Install and restart Jenkins.

   > ⚠️ Do not confuse **Updates** tab with **Available plugins**.

2. **Configure pipeline with `agent { docker { ... } }`**

    * Example: use `node:18-alpine`.

3. **Run pipeline**

    * First execution:

        * Jenkins uses `docker pull` to fetch the image.
        * Slower due to download.
    * Later executions:

        * Jenkins uses cached image → much faster.

---

## Error Handling & Common Mistakes

* **Error: `npm not found`**

    * Means npm is not installed on agent. Solution: run inside a Node.js Docker container.
* **Error: `invalid agent type docker specified`**

    * Happens if Docker Pipeline plugin is not installed.
* **Syntax errors**

    * Example: writing `image: node:18-alpine` (with colon) instead of `image 'node:18-alpine'`.
    * Always wrap image names in quotes.

---

## Workflow Recap

1. Without Docker:

    * Agents must have tools installed → environment conflicts possible.
2. With Docker:

    * Jenkins pulls specific Docker images (e.g., `node:18-alpine`).
    * Containers provide isolated environments.
    * Run commands (`npm install`, `npm start`, etc.) inside containers.
    * After job finishes → container is destroyed, keeping agents clean.

---

## Benefits of Docker in CI

* Consistent build environments across all agents.
* Easy to switch between different tool versions.
* No pollution of Jenkins agents with unnecessary dependencies.
* Faster build setup after the first run.

---

✅ **Key Takeaway**:
Always prefer **Dockerized build environments** in Jenkins pipelines. This ensures reproducibility, avoids conflicts, and makes managing dependencies much simpler.
