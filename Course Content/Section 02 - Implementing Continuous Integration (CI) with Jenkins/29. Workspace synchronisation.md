# 29. Workspace Synchronisation

## Overview

This lecture explains how **workspaces behave differently** when using Docker vs. non-Docker stages in Jenkins pipelines. It highlights why Jenkins creates separate workspaces by default for Docker-based stages, and how to configure **workspace synchronization** so that all stages share the same workspace if needed.

---

## Execution Flow Recap

### 1. **Without Docker**

* Controller finds an **agent**.
* Agent executes the commands **directly on the machine** (e.g., `ls`, `touch file.txt`).
* Workspace is located under:

  ```
  /var/jenkins_home/workspace/<job_name>
  ```

### 2. **With Docker**

* Controller finds an **agent** capable of running Docker.
* The agent:

    1. Pulls the Docker image (if not available).
    2. Starts a **container** from that image.
    3. Executes all stage commands **inside the container**.
    4. Stops the container after execution.
* By default, Jenkins assigns a **different workspace** for container stages:

  ```
  /var/jenkins_home/workspace/<job_name>@2
  ```

---

## Demonstration Example

### Pipeline Structure

```groovy
pipeline {
    agent any
    stages {
        stage('Without Docker') {
            steps {
                sh '''
                    echo "Running without Docker"
                    ls -la
                    touch container_no.txt
                '''
            }
        }
        stage('With Docker') {
            agent {
                docker {
                    image 'node:14'
                }
            }
            steps {
                sh '''
                    echo "Running inside Docker"
                    ls -la
                    touch container_yes.txt
                '''
            }
        }
    }
}
```

### Observed Behavior

* Stage **Without Docker**:

    * Workspace: `/var/jenkins_home/workspace/test-npm`
    * Creates file: `container_no.txt`
* Stage **With Docker**:

    * Workspace: `/var/jenkins_home/workspace/test-npm@2`
    * Creates file: `container_yes.txt`
    * Does **not see** `container_no.txt` from previous stage.

---

## Why Two Workspaces?

* **Non-Docker stages** and **Docker-based stages** are isolated by default.
* Jenkins ensures containerized stages don’t accidentally interfere with the host agent workspace.
* Result: multiple workspaces (`job_name`, `job_name@2`, `job_name@3`, etc.).

---

## Workspace Synchronisation (Solution)

If interdependencies exist between stages (e.g., one stage produces files that the next stage consumes), you need **a shared workspace**.

### Fix: Add `reuseNode true`

```groovy
pipeline {
    agent any
    stages {
        stage('Without Docker') {
            steps {
                sh '''
                    echo "Running without Docker"
                    ls -la
                    touch container_no.txt
                '''
            }
        }
        stage('With Docker Shared Workspace') {
            agent {
                docker {
                    image 'node:14'
                    reuseNode true   // <---- Ensures same workspace is used
                }
            }
            steps {
                sh '''
                    echo "Running inside Docker with shared workspace"
                    ls -la
                    touch container_yes.txt
                '''
            }
        }
    }
}
```

### Result

* Both stages share the **same workspace**.
* Both files (`container_no.txt` and `container_yes.txt`) are visible in `/var/jenkins_home/workspace/test-npm`.

---

## Key Points

* **Default behavior**: Docker stages get **separate workspaces** (`@2`, `@3` suffixes).
* **Shared workspace**: Use `reuseNode true` in the Docker agent block.
* This is essential when stages depend on each other’s output.

---

## (AI: Notes)

* Best practice: Use **separate workspaces** unless you explicitly need sharing. Isolation avoids accidental overwrites and conflicts between host and container.
* If you frequently need shared workspaces across multiple pipelines, consider standardizing with **custom pipeline libraries** that enforce `reuseNode true` where required.
