### 🔹 Purpose of Comments

* **Documentation**: Help you (or teammates) remember why a block exists.
* **Debugging / Speeding up development**: Temporarily disable parts of the pipeline to save time.

---

### 🔹 Types of Comments in Jenkinsfile

1. **Single-line comments**

   ```groovy
   // This is a single-line comment
   ```

    * Highlight intent or clarify a step.

2. **Block (multi-line) comments**

   ```groovy
   /* 
   This is line one
   This is line two
   */
   ```

    * Used to comment out multiple lines (e.g., a full stage).

---

### 🔹 Practical Uses

* **Disabling a Stage**

  ```groovy
  /* 
  stage('Build') {
      steps {
          sh 'npm run build'
      }
  }
  */
  ```

    * Removes the build stage temporarily from the pipeline.
    * Useful when you only want to test later stages without waiting for the build.
    * Still keeps the config in place for later reuse.

* **Disabling a Command in a Stage**
  Inside a shell script (`sh` block), you can use **`#`** to disable individual commands:

  ```groovy
  stage('Test') {
      steps {
          sh '''
              npm test
              # npm run test:extra   <- this will not run
          '''
      }
  }
  ```

---

### 🔹 Key Notes

* Pipeline must remain **valid Jenkinsfile syntax** after commenting.
* Prefer commenting **entire blocks** rather than partial snippets (to avoid syntax errors).
* Speeds up iteration when testing specific stages.

---

👉 In practice: when developing pipelines, comment out expensive stages (like `build` or `deploy`) while focusing on quicker stages (like `test`).
