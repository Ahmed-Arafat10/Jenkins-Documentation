## 🔹 Key Concepts

### 1. **No fixed rules**

* Pipelines vary depending on **language, technology, and project requirements**.
* Typical flow: **Build → Test → Further stages**, but order may differ.

---

### 2. **Dependencies Between Stages**

* Some steps **depend on build artifacts**, others only on dependencies.

    * **Example**:

        * Unit tests → need project dependencies (not build artifacts).
        * End-to-end tests → need build artifacts.
* **Parallelism**:

    * Independent steps (e.g., Unit Tests vs E2E Tests) → can run in parallel.
    * Dependent steps → must run sequentially.

---

### 3. **Fail Fast Principle**

* Run stages most likely to fail **as early as possible** to save resources.
* Examples:

    * If **unit tests fail often**, run them before build.
    * **Linting** (fast & lightweight) should be the **first step**.
* Saves compute by **stopping early** before long-running tests.

---

### 4. **Parallelism vs. Efficiency**

* Just because you *can* run in parallel doesn’t mean you *should*.
* **Example**:

    * Unit tests (15s) vs E2E tests (5m+).
    * If run in parallel → E2E consumes resources even if unit tests fail.
    * Better: run unit tests first, then E2E.

---

### 5. **Linting**

* Checks code for stylistic & static analysis issues.
* **Very fast** → should be at the beginning.
* Prevents wasted capacity if run later (e.g., failing on a missing comma after long builds).

---

### 6. **Experimentation**

* There’s no universal best structure.
* **Iterate, measure, optimize** for:

    * Speed.
    * Resource usage.
    * Failure patterns of your project.

---

## 🔹 Best Practices Checklist

✔️ Start with **fast checks** (linting, small unit tests).
✔️ Identify and respect **dependencies** between stages.
✔️ Use **parallelism** only for independent, heavy tasks.
✔️ Place **frequent-failure steps earlier** to fail fast.
✔️ Continuously **experiment & refine** pipeline structure.
